## Part 1：软件设计思想（通用，不绑定语言）

### 1.1 插件系统设计模式

**为什么需要：** GitTributary 的核心就是插件系统，这不是"加分项"而是"必须项"。

需要掌握的模式：

| 模式 | 解决什么问题 | 在 GitTributary 的应用 |
|------|------------|---------------------|
| **Strategy** | 运行时切换算法/行为 | 不同插件对同一事件的不同处理 |
| **Observer / Event Emitter** | 解耦事件生产者和消费者 | 文件变更通知、Git push 钩子 |
| **Command** | 将操作封装为对象 | 插件命令注册、撤销/重做 |
| **Builder** | 分步构造复杂对象 | Plugin Host Manager 初始化 |
| **Registry** | 集中管理可发现的组件 | 插件注册中心（前后端各一个） |
| **Adapter** | 统一不兼容的接口 | 内部插件和 WASM 插件对外暴露统一 Capability API |
| **Template Method** | 定义算法骨架，子类填充细节 | 插件生命周期（setup → execute → teardown） |

**学到什么程度：** 理解意图和适用场景，能在代码中识别和应用。不需要背 UML。

**参考入口：**
- 《Head First 设计模式》— 入门最友好
- Refactoring Guru: https://refactoring.guru/design-patterns — 免费、图解清晰、有 Rust/TS 示例

### 1.2 依赖注入与控制反转（IoC/DI）

**为什么需要：** 插件系统的本质就是 IoC — 宿主控制流程，插件只提供实现。Capability API 的设计依赖 DI 思想。

核心概念：
- 什么是"依赖"（插件依赖宿主提供的 Git 数据访问能力）
- 什么是"注入"（宿主把 `GitRead` trait 的实现传给插件，而不是插件自己创建）
- Rust 中如何实现 DI（trait object、泛型、闭包）

**学到什么程度：** 能在 Rust 中用 trait + 泛型实现简单的依赖注入。

**参考入口：**
- Rust Book Ch.17 (OOP) 和 Ch.10 (Generics) — trait 就是 Rust 的接口
- 《Rust Design Patterns》: https://rust-unofficial.github.io/patterns/ — idiomatic Rust 设计模式

### 1.3 事件驱动架构

**为什么需要：** 插件前后端联动的"事件推送"通道基于此。文件监听、Git 钩子、插件间通信都依赖事件模型。

核心概念：
- 事件循环（Event Loop）
- 发布-订阅（Pub/Sub）
- 事件命名空间与过滤
- 同步事件 vs 异步事件

**学到什么程度：** 能设计一个简单的事件总线（Rust 侧 `HashMap<String, Vec<Callback>>`）。

**参考入口：**
- Node.js EventEmitter 源码 — 经典实现，概念通用
- Tauri 事件系统文档 — 你马上要用的

### 1.4 序列化与接口契约

**为什么需要：** 插件 manifest（JSON）、前后端 IPC（JSON）、WASM 宿主通信（JSON）全部依赖序列化。接口设计的好坏直接决定开发体验。

核心概念：
- JSON Schema — 用于校验 manifest 格式
- Serde（Rust）— Rust 生态的标准序列化框架
- 强类型契约 vs 弱类型契约的取舍

**学到什么程度：** 熟练使用 Serde 的 `Serialize`/`Deserialize` derive，理解 `#[serde(rename)]`、`#[serde(default)]` 等常用属性。

**参考入口：**
- Serde 官方文档: https://serde.rs/ — 非常好的教程
- JSON Schema: https://json-schema.org/ — 未来用于 manifest 校验

### 1.5 错误处理策略

**为什么需要：** 插件系统是错误的高发区 — 插件崩溃、WASM trap、IPC 超时、权限拒绝。没有统一的错误策略，调试将是噩梦。

核心概念：
- Rust 的 `Result<T, E>` 和 `?` 操作符
- 自定义错误类型（`thiserror` crate）
- 错误传播链路：WASM 插件 → Plugin Host → Tauri IPC → 前端
- 用户可见的错误 vs 开发者可见的错误

**学到什么程度：** 能定义分层错误类型，让插件错误在每一层都能被正确处理和传递。

**参考入口：**
- Rust Book Ch.9 (Error Handling)
- `thiserror` + `anyhow` 的使用场景区分

## Part 2：Rust 技术（后端核心）

### 2.1 Trait 系统深入

**为什么需要：** Capability API 的基石。`GitRead`、`FileRead` 等能力全部用 trait 定义。

需要掌握：
- trait 定义与实现
- trait object (`dyn Trait`) vs 泛型 (`impl Trait`) 的取舍
- trait bound 约束（`fn process<T: GitRead + FileRead>(data: T)`）
- associated types vs generic types
- blanket implementations

**学到什么程度：** 能设计一组 trait 来抽象"插件可用的能力"，并让不同实现（内部 Rust / WASM host function）都满足同一组 trait。

**参考入口：**
- Rust Book Ch.10.2 (Traits)
- 《Rust for Rustaceans》Ch.3 — 进阶 trait 用法

### 2.2 生命周期与所有权在插件场景的应用

**为什么需要：** 插件加载/卸载时，资源所有权的转移是核心难题。WASM 实例的生命周期管理、Plugin Host 对插件的持有关系都需要清晰的所有权设计。

需要掌握：
- `'lifetime` 标注在结构体和函数签名中的含义
- `Rc` / `Arc` 引用计数 — 共享所有权
- `Mutex` / `RwLock` — 内部可变性
- `Box<dyn Trait>` — 堆分配的 trait object

**学到什么程度：** 能设计一个 `PluginHostManager` 结构体，安全地持有和管理多个插件实例的生命周期。

**参考入口：**
- Rust Book Ch.4 (Ownership) + Ch.15 (Smart Pointers)
- Tokio tutorial — 异步场景下的所有权管理（如果后续用到异步）

### 2.3 宏基础（derive macro）

**为什么需要：** Tauri 的 `#[tauri::command]`、Serde 的 `#[derive(Serialize)]` 都是宏。理解宏能帮你读懂框架代码，未来可能需要自定义宏来简化插件开发。

需要掌握：
- derive macro 的工作原理（不需要写 proc macro，但要理解）
- 常见的 derive：`Clone`, `Debug`, `Serialize`, `Deserialize`
- `#[cfg(...)]` 条件编译

**学到什么程度：** 能读懂 `#[tauri::command]` 展开后的代码大概是什么样的。不需要自己写 proc macro。

**参考入口：**
- Rust Book Ch.19.5 (Macros)
- Tauri 源码中 command macro 的实现

### 2.4 异步编程基础

**为什么需要：** Tauri command 是异步的（`async fn`）。Git 操作、文件 I/O、WASM 调用都可能是阻塞操作，需要用异步或 `spawn_blocking` 处理。

需要掌握：
- `async` / `await` 基础
- `tokio` 运行时（Tauri 内部使用）
- `spawn_blocking` — 把阻塞操作放到线程池
- `Future` trait 的基本概念

**学到什么程度：** 能写异步的 Tauri command，知道什么时候该用 `spawn_blocking`。

**参考入口：**
- Tokio Tutorial: https://tokio.rs/tokio/tutorial
- Tauri 异步 command 文档

### 2.5 错误处理工具链

**为什么需要：** 见 1.5。具体到 Rust 工具：

需要掌握：
- `thiserror` — 定义自定义错误类型
- `anyhow` — 应用层错误传递
- `Result<T, E>` 的惯用写法
- Tauri command 中错误如何传递到前端

**学到什么程度：** 能定义 `PluginError` 枚举，让 WASM 调用失败、权限拒绝、超时等错误都有清晰的类型。

**参考入口：**
- `thiserror` 文档: https://docs.rs/thiserror
- Tauri error handling 指南

### 2.6 Cargo 工作空间（Workspace）

**为什么需要：** GitTributary 的代码结构很可能是多 crate 的：

```
gittributary/
├── Cargo.toml              ← workspace root
├── crates/
│   ├── core/               ← 核心类型定义（PluginManifest, Capability 等）
│   ├── plugin-host/        ← Plugin Host Manager
```

需要掌握：
- workspace `Cargo.toml` 的配置
- crate 之间的依赖声明
- 共享类型放在 `core` crate 中

**学到什么程度：** 能搭建一个 workspace，让多个 crate 互相引用。

**参考入门：**
- Cargo Book - Workspaces: https://doc.rust-lang.org/cargo/reference/workspaces.html

## Part 3：React / TypeScript 技术（前端核心）

### 3.1 TypeScript 类型系统

**为什么需要：** `PluginManifest` 是前后端的契约。前端的类型定义要和 Rust 的 Serde 输出严格对应。

需要掌握：
- `interface` vs `type` 的取舍
- 联合类型（`'form' | 'list' | 'custom' | 'webview'`）
- 泛型（`PluginHost.invoke<T>(...): Promise<T>`）
- `Record<string, unknown>` — 动态 key 的对象类型
- 类型守卫（`if (plugin.ui.sidebarEntry) { ... }` 自动收窄类型）

**学到什么程度：** 能写出与 Rust Serde 输出精确对应的 TypeScript 类型定义。

**参考入口：**
- TypeScript Handbook: https://www.typescriptlang.org/docs/handbook/

### 3.2 React 状态管理

**为什么需要：** 插件激活状态、命令面板开关、当前选中的插件 — 这些都是全局状态。选对状态管理方案，代码会清晰很多。

方案选择：

| 方案 | 复杂度 | 适合场景 |
|------|--------|---------|
| `useState` + prop drilling | 低 | 状态少、层级浅 |
| `useContext` + `useReducer` | 中 | 中等复杂度的全局状态 |
| Zustand | 低-中 | 轻量全局状态，推荐 |
| Redux Toolkit | 高 | 大型应用，当前不需要 |

**对 GitTributary 的建议：** 用 **Zustand**。API 极简，TypeScript 友好，不需要 Provider 包裹。

```typescript
// 示例：插件状态管理
import { create } from 'zustand';

interface PluginStore {
  activePlugin: string | null;
  plugins: PluginManifest[];
  setActive: (name: string | null) => void;
  loadPlugins: () => Promise<void>;
}

export const usePluginStore = create<PluginStore>((set) => ({
  activePlugin: null,
  plugins: [],
  setActive: (name) => set({ activePlugin: name }),
  loadPlugins: async () => {
    const plugins = await pluginHost.listPlugins();
    set({ plugins });
  },
}));
```

**学到什么程度：** 能用 Zustand 管理插件相关的全局状态。

**参考入口：**
- Zustand 文档: https://github.com/pmndrs/zustand — README 就够用

### 3.3 React 组件设计模式

**为什么需要：** 插件 UI 系统的核心是"根据 manifest 动态渲染不同的组件"。这需要掌握几种关键模式。

需要掌握的模式：

| 模式 | 用途 | 在 GitTributary 的应用 |
|------|------|---------------------|
| **组合（Composition）** | 用 children/slot 组合 UI | AppShell 组合 Sidebar + MainView + StatusBar |
| **受控组件** | 状态由父组件管理 | 命令面板的打开/关闭状态 |
| **render prop / children as function** | 让调用方自定义渲染 | 未来插件自定义视图 |
| **条件渲染** | 根据状态/配置渲染不同内容 | 根据 mainView.type 渲染不同视图 |
| **错误边界（Error Boundary）** | 捕获子组件渲染错误 | 插件视图崩溃时不影响主应用 |

**学到什么程度：** 能设计可组合、可配置的组件，知道 Error Boundary 的用法。

**参考入口：**
- React 官方文档 - Thinking in Components
- React 官方文档 - Error Boundaries

### 3.4 自定义 Hook

**为什么需要：** 插件通信逻辑（invoke、事件订阅）会被多个组件复用。封装为 Hook 是 React 的标准做法。

需要掌握的 Hook 模式：

```typescript
// 封装插件调用
function usePluginCommand(pluginName: string, command: string) {
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  async function execute(args?: unknown) {
    setLoading(true);
    setError(null);
    try {
      return await pluginHost.invoke(pluginName, command, args);
    } catch (e) {
      setError(e as Error);
    } finally {
      setLoading(false);
    }
  }

  return { execute, loading, error };
}

// 封装插件事件订阅
function usePluginEvent(pluginName: string, event: string) {
  const [data, setData] = useState<unknown>(null);

  useEffect(() => {
    const handler = (d: unknown) => setData(d);
    pluginHost.on(pluginName, event, handler);
    return () => pluginHost.off(pluginName, event, handler);
  }, [pluginName, event]);

  return data;
}
```

**学到什么程度：** 能封装带 loading/error 状态的异步 Hook 和事件订阅 Hook。

**参考入口：**
- React 官方文档 - Reusing Logic with Custom Hooks

### 3.5 Tauri IPC 在前端的使用

**为什么需要：** 这是前端和 Rust 后端通信的唯一通道。

需要掌握：
- `invoke(command, args)` — 调用 Rust command
- `listen(event, handler)` / `once(event, handler)` — 监听后端事件
- `emit(event, payload)` — 前端主动发事件
- 错误处理：`invoke` 返回 `Promise`，失败时抛异常

**学到什么程度：** 能在 React 中熟练使用 invoke 和 listen/emit。

**参考入口：**
- Tauri JS API 文档: https://v2.tauri.app/develop/calling-rust/
- `@tauri-apps/api` npm 包

### 3.6 路由（React Router）

**为什么需要：** 插件视图的切换本质上是路由。`/plugin/:name` 是动态路由。

需要掌握：
- 基本路由配置
- 动态路由参数（`useParams`）
- 嵌套路由（如果 Sidebar 和 MainView 各有自己的路由）

**学到什么程度：** 能配置基本的动态路由。

**参考入口：**
- React Router 文档: https://reactrouter.com/

## Part 4：专项技术（按需深入）

### 4.1 WASM 与 Extism

**为什么需要：** 第二层插件（社区扩展）基于 WASM。

需要掌握：
- WASM 是什么、能做什么、不能做什么
- Extism 的 Host SDK（Rust 侧加载 .wasm、调用函数、注册 host function）
- Extism 的 PDK（插件开发侧，如何编写 WASM 插件）
- WASM 和宿主之间的数据传递（JSON 字符串）

**学到什么程度：** 能用 Extism 加载一个 .wasm 文件并调用其中的函数。

**参考入口：**
- Extism 官方教程: https://extism.org/docs/
- Extism Rust Host SDK: https://github.com/extism/rust-sdk

### 4.2 Git 操作库（git2 / gix）

**为什么需要：** Git 数据加工是核心功能。

两条路线：

| 库 | 特点 |
|---|------|
| `git2` (libgit2 binding) | 成熟稳定、API 丰富、需要 C 依赖 |
| `gix` (纯 Rust) | 无 C 依赖、跨平台友好、API 在演进中 |

**学到什么程度：** 能用其中一个库完成基本的 Git 操作（status、diff、log、commit、push）。

**参考入口：**
- `git2` crate: https://docs.rs/git2
- `gix` crate: https://docs.rs/gix

### 4.3 文件系统监听（notify）

**为什么需要：** 监听笔记目录的增删改是产品需求之一。

需要掌握：
- `notify` crate 的基本用法
- 防抖/节流（文件保存时会触发多次事件）
- 与事件驱动架构的集成

**学到什么程度：** 能监听目录变化并触发回调。

**参考入口：**
- `notify` crate: https://docs.rs/notify
