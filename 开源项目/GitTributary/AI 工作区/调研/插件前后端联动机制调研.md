# 插件前后端联动机制调研

**日期：** 2026-04-04
**目的：** 解决"一个插件如何同时控制前端页面与后端 API"这一核心设计问题

---

## 1. 问题定义

GitTributary 基于 Tauri (Rust + React)，前后端天然分离：

```
React 前端  ──── IPC ────  Rust 后端 ──── WASM 沙箱
   │                         │
   └─── 插件的 UI 在这里      └─── 插件的逻辑在这里
```

一个插件 = **后端逻辑**（Rust/WASM）+ **前端表现**（React 组件）。两者需要协调工作，但中间隔了一层 IPC，无法像 Obsidian 那样通过共享内存的 `this` 引用直接打通。

需要回答的核心问题：

1. 插件的前后端如何绑定在一起？
2. 前端如何调用后端？（请求-响应）
3. 后端如何通知前端？（事件推送）
4. 不同插件的命名空间如何隔离？

---

## 2. 竞品的解法

### 2.1 VSCode：割裂式

扩展会分别注册 WebviewViewProvider（前端）和 registerCommand（后端），两者通过 `postMessage` 通信：

```typescript
// 后端
const provider = new MyViewProvider(context);
context.subscriptions.push(
  vscode.window.registerWebviewViewProvider('myView', provider)
);

// 前端（Webview 内）
const vscode = acquireVsCodeApi();
vscode.postMessage({ type: 'deploy', data: { branch: 'main' } });

window.addEventListener('message', event => {
  const { type, data } = event.data;
  // 处理后端推送
});
```

**问题：** 前后端通信完全靠开发者自己保证 `type` 字符串的一致性，没有统一契约。Webview 里的代码是 HTML 字符串拼接，开发体验差。

### 2.2 Obsidian：直通式

同进程内，插件的 `onload()` 同时注册命令和视图，共享 `this`：

```typescript
export default class MyPlugin extends Plugin {
  private status = 'idle';

  async onload() {
    // 后端：注册命令
    this.addCommand({
      id: 'deploy',
      name: 'Deploy',
      callback: () => this.doDeploy(),  // 直接调用 this 方法
    });

    // 前端：注册视图
    this.registerView(
      'deploy-view',
      (leaf) => new DeployView(leaf, this)  // 把 this 传给视图
    );
  }

  private async doDeploy() {
    this.status = 'deploying';
    // 视图通过 this.status 直接读取状态
  }
}
```

**问题：** 简单但零隔离。一个插件可以读写另一个插件的所有内部状态，可以搞崩整个应用。

### 2.3 Tauri 原生：invoke 桥接

Rust 端注册 command，前端通过 `invoke()` 调用：

```rust
// Rust 端
#[tauri::command]
fn deploy(branch: String) -> Result<DeployResult, String> {
    // 执行部署
}
```

```typescript
// React 端
const result = await invoke('deploy', { branch: 'main' });
```

**问题：** Tauri 不解决"一个插件的前后端如何绑定"的问题。command 是全局扁平的，没有命名空间隔离。

### 2.4 对比总结

| 维度 | VSCode | Obsidian | Tauri 原生 |
|------|--------|----------|-----------|
| 前后端绑定 | 手动 postMessage | this 引用直通 | 无绑定机制 |
| 通信方式 | 字符串消息 | 函数调用 | invoke IPC |
| 隔离性 | 进程隔离 | 无隔离 | 进程隔离 |
| 契约一致性 | 无保障 | 天然一致 | 无保障 |
| 开发体验 | 差（HTML 拼接） | 好 | 中等 |

**结论：** 没有竞品给出既隔离又开发体验好的方案。GitTributary 需要自己设计。

## 3. GitTributary 方案：Manifest 驱动 + 双通道通信

### 3.1 核心思想

**manifest 是唯一的耦合点。** 一个 manifest 同时声明插件的后端能力（commands）和前端表现（ui），前后端都从 manifest 获取对方的信息。

```
┌──────────────────────────────────────────────────────┐
│                    manifest.json                      │
│                                                       │
│  ┌─────────────────────┐  ┌────────────────────────┐ │
│  │  后端关心的           │  │  前端关心的             │ │
│  │  - commands          │  │  - ui.sidebarEntry     │ │
│  │  - capabilities      │  │  - ui.mainView         │ │
│  │  - hooks             │  │  - commands (命令面板)  │ │
│  └──────────┬──────────┘  └───────────┬────────────┘ │
└─────────────┼─────────────────────────┼──────────────┘
              │                         │
     ┌────────▼────────┐      ┌────────▼────────┐
     │  Plugin Host     │      │  PluginRegistry  │
     │  Manager (Rust)  │      │  (React)         │
     │                  │      │                  │
     │  注册 commands   │      │  注册 UI 入口    │
     │  注册 hooks      │      │  注册命令面板项  │
     └─────────────────┘      └─────────────────┘
```

### 3.2 插件单元结构

每个插件是一个自包含的"单元"：

```
git-deploy/
├── manifest.json       ← 声明（前后端都读）
├── plugin.wasm         ← 后端逻辑
└── ui/                 ← 前端表现（可选）
    └── DeployForm.tsx
```

#### manifest 完整定义

```json
{
  "name": "git-deploy",
  "version": "0.1.0",
  "minHostVersion": "0.1.0",

  "capabilities": [
    "git:read",
    "git:write",
    "file:read"
  ],

  "commands": [
    { "id": "deploy", "description": "Deploy notes to GitHub Pages" },
    { "id": "preview", "description": "Preview deploy result" },
    { "id": "status", "description": "Get deploy status" }
  ],

  "hooks": [
    { "event": "on_push_after", "command": "deploy" }
  ],

  "ui": {
    "sidebarEntry": {
      "label": "Deploy",
      "icon": "rocket",
      "order": 10
    },
    "mainView": {
      "type": "form",
      "component": "DeployForm"
    },
    "statusBar": {
      "text": "${status}",
      "priority": 5
    }
  }
}
```

#### TypeScript 类型定义

```typescript
interface PluginManifest {
  name: string;
  version: string;
  minHostVersion: string;
  capabilities: string[];
  commands: CommandDecl[];
  hooks: HookDecl[];
  ui: PluginUIDecl;
}

interface CommandDecl {
  id: string;              // 命令 ID，如 "deploy"
  description: string;
}

interface HookDecl {
  event: string;           // 触发事件，如 "on_push_after"
  command: string;         // 绑定的命令 ID
}

interface PluginUIDecl {
  sidebarEntry?: {
    label: string;
    icon: string;
    order?: number;
  };
  mainView?: {
    type: 'form' | 'list' | 'custom' | 'webview';
    component?: string;
  };
  statusBar?: {
    text: string;          // 支持模板变量 ${status}
    priority?: number;
  };
}
```

### 3.3 双通道通信

插件的前后端联动需要两种通信模式：

| 模式 | 方向 | 场景 | 实现 |
|------|------|------|------|
| **请求-响应** | 前端 → 后端 | 用户点击部署按钮 | `pluginHost.invoke()` |
| **事件推送** | 后端 → 前端 | 部署进度、文件变更通知 | `app.emit()` + `listen()` |

#### 通道一：请求-响应（前端驱动后端）

```
React 前端                    Tauri IPC              Plugin Host          WASM Plugin
  │                             │                        │                    │
  │  pluginHost.invoke(         │                        │                    │
  │    "git-deploy",            │                        │                    │
  │    "deploy",                │                        │                    │
  │    { branch: "main" }       │                        │                    │
  │  )                          │                        │                    │
  │                             │                        │                    │
  ├──── invoke("plugin:         │                        │                    │
  │     git-deploy|deploy") ───►│                        │                    │
  │                             ├──── dispatch ─────────►│                    │
  │                             │                        ├──── call("deploy") ►│
  │                             │                        │                    │
  │                             │                        │◄─── result ─────────┤
  │◄──── result ────────────────┤◄───────────────────────┤                    │
```

**前端封装：**

```typescript
class PluginHost {
  async invoke(pluginName: string, command: string, args?: Record<string, unknown>) {
    return await invoke(`plugin:${pluginName}|${command}`, args);
  }
}

export const pluginHost = new PluginHost();
```

**后端路由（Tauri command）：**

```rust
#[tauri::command]
async fn plugin_dispatch(
    plugin_name: String,
    command: String,
    args: Option<serde_json::Value>,
    state: tauri::State<'_, PluginHostManager>,
) -> Result<serde_json::Value, String> {
    state.dispatch(&plugin_name, &command, args).await
}
```

**命名空间约定：** `plugin:{name}|{command}` 格式保证不同插件不会冲突。

#### 通道二：事件推送（后端驱动前端）

```
WASM Plugin                   Plugin Host              Tauri Event Bus        React 前端
  │                              │                          │                    │
  │  emit("progress", 30%)       │                          │                    │
  ├─────────────────────────────►│                          │                    │
  │                              │  app.emit(               │                    │
  │                              │    "plugin:git-deploy:   │                    │
  │                              │     progress",           │                    │
  │                              │    { percent: 30 }       │                    │
  │                              │  )                       │                    │
  │                              ├─────────────────────────►│                    │
  │                              │                          │  listen(           │
  │                              │                          │    "plugin:         │
  │                              │                          │    git-deploy:      │
  │                              │                          │    progress")       │
  │                              │                          ├───────────────────►│
  │                              │                          │                    │
  │                              │                          │                    │  更新进度条
```

**事件命名空间约定：** `plugin:{name}:{event}`

**前端订阅：**

```typescript
// PluginHost 增加事件订阅
class PluginHost {
  private listeners = new Map<string, Set<Function>>();

  async init() {
    // 监听所有 plugin: 开头的事件
    await listen('plugin:*', (event) => {
      const { pluginName, eventName, data } = parsePluginEvent(event);
      const key = `${pluginName}:${eventName}`;
      this.listeners.get(key)?.forEach(fn => fn(data));
    });
  }

  on(pluginName: string, event: string, handler: (data: unknown) => void) {
    const key = `${pluginName}:${event}`;
    if (!this.listeners.has(key)) {
      this.listeners.set(key, new Set());
    }
    this.listeners.get(key)!.add(handler);
  }

  off(pluginName: string, event: string, handler: (data: unknown) => void) {
    this.listeners.get(`${pluginName}:${event}`)?.delete(handler);
  }
}
```

**前端组件使用：**

```typescript
function DeployForm({ pluginName }: Props) {
  const [progress, setProgress] = useState(0);
  const [result, setResult] = useState(null);

  useEffect(() => {
    // 订阅后端推送的进度事件
    pluginHost.on(pluginName, 'progress', (data: any) => {
      setProgress(data.percent);
    });

    return () => pluginHost.off(pluginName, 'progress', handler);
  }, [pluginName]);

  async function handleDeploy() {
    // 前端主动调用后端
    const res = await pluginHost.invoke(pluginName, 'deploy', { branch: 'main' });
    setResult(res);
  }

  return (
    <form onSubmit={handleDeploy}>
      <ProgressBar value={progress} />
      <button type="submit">Deploy</button>
      {result && <Preview data={result} />}
    </form>
  );
}
```

### 3.4 完整数据流示例：部署插件

以 `git-deploy` 插件为例，走一遍完整链路：

```
1. 用户在 Sidebar 点击 "Deploy"          ← 前端根据 manifest.ui.sidebarEntry 渲染
2. MainView 切换到 DeployForm            ← 前端根据 manifest.ui.mainView.type 渲染
3. 用户填写分支名，点击 "Deploy"          ← 用户交互
4. DeployForm 调用 pluginHost.invoke()   ← 请求-响应通道
5. Tauri IPC 路由到 plugin_dispatch      ← 命名空间解析
6. Plugin Host Manager 查找 WASM 插件    ← 后端路由
7. WASM 插件执行 deploy 函数             ← 业务逻辑
8. 执行过程中 emit "progress" 事件       ← 事件推送通道
9. 前端 listen 到 progress，更新进度条   ← UI 响应
10. WASM 插件返回结果                    ← 请求-响应通道
11. 前端收到结果，展示预览               ← UI 响应
```

## 4. 插件 UI 扩展的层级

不是所有插件都需要完整的前后端联动。按复杂度分级：

### Level 0：纯前端插件

不需要后端逻辑，只在 UI 层做展示增强。

```
manifest.ui: { sidebarEntry, mainView: { type: "custom" } }
manifest.commands: []       ← 空
```

**例子：** 笔记字数统计面板（前端直接读取后端返回的静态数据）

### Level 1：纯后端插件

不需要自定义 UI，通过命令面板触发，结果以通知形式展示。

```
manifest.ui: {}             ← 无 UI 声明
manifest.commands: ["check"]
```

**例子：** 敏感信息扫描（命令触发 → 后端检查 → 返回结果 → 系统通知）

### Level 2：请求-响应联动

前端有自定义 UI，通过 invoke 调用后端。

```
manifest.ui: { mainView: { type: "form" } }
manifest.commands: ["deploy", "preview"]
```

**例子：** 部署插件（表单输入 → 调用部署 → 展示结果）

### Level 3：双向通信联动

前端 UI 需要实时响应后端状态变化。

```
manifest.ui: { mainView: { type: "list" }, statusBar: { text: "${status}" } }
manifest.commands: ["sync"]
额外：后端 emit 事件
```

**例子：** Git 同步插件（同步进度实时展示 + 文件变更实时刷新列表）

### 分级对前端设计的影响

| 级别 | 前端需要的能力 | 实现复杂度 |
|------|--------------|-----------|
| Level 0 | 静态渲染 manifest.ui | 低 |
| Level 1 | 命令面板 + invoke | 低 |
| Level 2 | 自定义视图组件 + invoke | 中 |
| Level 3 | 自定义视图 + invoke + 事件订阅 | 高 |

**MVP 策略：** 先支持 Level 0-2，Level 3 在扩展阶段实现。

## 5. 命名空间隔离设计

### 5.1 为什么需要命名空间

多个插件共存时，需要防止：
- 命令 ID 冲突（两个插件都注册 "deploy"）
- 事件名冲突（两个插件都 emit "progress"）
- UI 路由冲突（两个插件都想接管 MainView）

### 5.2 命名空间规则

| 范畴 | 格式 | 示例 |
|------|------|------|
| IPC 调用 | `plugin:{name}\|{command}` | `plugin:git-deploy\|deploy` |
| 事件 | `plugin:{name}:{event}` | `plugin:git-deploy:progress` |
| 路由 | `/plugin/{name}` | `/plugin/git-deploy` |
| 命令面板 | `{name}: {label}` | `git-deploy: Deploy Notes` |

### 5.3 内部插件与外部插件的统一

内部插件（编译时集成）和外部插件（WASM 运行时加载）使用相同的命名空间规则：

```
内部插件 "git-sync"
  invoke("plugin:git-sync|sync")    ← 直接调用 Rust 函数
  event("plugin:git-sync:progress") ← app.emit()

外部插件 "git-deploy"
  invoke("plugin:git-deploy|deploy")    ← Extism 调用 WASM
  event("plugin:git-deploy:progress")   ← app.emit()
```

前端完全不区分两者。

## 6. 前端组件架构

### 6.1 插件注册中心（React 侧）

```typescript
// PluginRegistry.ts
interface RegisteredPlugin {
  manifest: PluginManifest;
  status: 'active' | 'inactive' | 'error';
}

class PluginRegistry {
  private plugins = new Map<string, RegisteredPlugin>();

  register(manifest: PluginManifest) {
    this.plugins.set(manifest.name, { manifest, status: 'inactive' });
  }

  get(name: string): RegisteredPlugin | undefined {
    return this.plugins.get(name);
  }

  all(): RegisteredPlugin[] {
    return Array.from(this.plugins.values());
  }

  // 获取所有带侧边栏入口的插件
  sidebarEntries(): { name: string; label: string; icon: string; order: number }[] {
    return this.all()
      .filter(p => p.manifest.ui.sidebarEntry)
      .map(p => ({
        name: p.manifest.name,
        label: p.manifest.ui.sidebarEntry!.label,
        icon: p.manifest.ui.sidebarEntry!.icon,
        order: p.manifest.ui.sidebarEntry!.order ?? 99,
      }))
      .sort((a, b) => a.order - b.order);
  }

  // 获取所有注册的命令（用于命令面板）
  allCommands(): { pluginName: string; command: CommandDecl }[] {
    return this.all().flatMap(p =>
      p.manifest.commands.map(cmd => ({ pluginName: p.manifest.name, command: cmd }))
    );
  }
}

export const pluginRegistry = new PluginRegistry();
```

### 6.3 MainView 插件视图分发

```typescript
function MainView({ pluginName }: { pluginName: string | null }) {
  if (!pluginName) return <WelcomeView />;

  const plugin = pluginRegistry.get(pluginName);
  if (!plugin) return <NotFoundView name={pluginName} />;

  const viewType = plugin.manifest.ui.mainView?.type ?? 'custom';

  switch (viewType) {
    case 'form':   return <PluginFormView pluginName={pluginName} />;
    case 'list':   return <PluginListView pluginName={pluginName} />;
    case 'custom': return <PluginCustomView pluginName={pluginName} />;
    case 'webview': return <PluginWebView pluginName={pluginName} />;
    default:       return <PluginCustomView pluginName={pluginName} />;
  }
}
```

## 7. MVP 阶段落地清单

| 层 | 做什么 | 不做什么 |
|---|--------|---------|
| **manifest** | 定义 `PluginManifest` TypeScript 类型，包含 commands + ui | 不做动态 schema 校验 |
| **PluginHost** | 封装 `invoke(plugin, cmd, args)` 统一调用 | 不做插件间通信 |
| **PluginRegistry** | 注册中心，提供 sidebarEntries / allCommands 查询 | 不做热加载 |
| **Sidebar** | 根据 manifest 动态渲染菜单列表 | 不做文件夹视图 |
| **MainView** | 根据 `mainView.type` 渲染预置视图组件 | 不做动态组件加载 |
| **命令面板** | 支持注册 + 简易 when 条件过滤 | 不做快捷键绑定 |
| **事件推送** | 暂不做 | 扩展阶段实现 |

---

## 8. 需要避免的反模式

1. **不要为每个插件写专用前后端绑定** — 统一走 manifest + PluginHost
2. **不要在前端直接 import 插件代码** — 前端只认识 manifest 和命名空间
3. **不要硬编码路由** — 路由由 `pluginName` 动态决定
4. **不要跳过 TypeScript 类型定义** — `PluginManifest` 是前后端的唯一契约
5. **不要让插件直接操作 React state** — 必须通过 PluginHost 间接通信
6. **不要用全局 event bus 而不加命名空间** — 会导致插件间事件污染
