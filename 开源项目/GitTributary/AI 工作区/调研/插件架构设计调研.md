# 插件架构设计调研

**日期：** 2026-04-02
**目的：** 为 GitTributary 的插件系统设计提供理论依据与方案选型

---

## 1. 插件架构的本质

### 1.1 什么是插件架构

插件架构是一种将**核心系统**与**扩展功能**分离的设计模式。宿主程序（Host）定义标准接口，第三方或后续开发的功能模块（Plugin）通过这些接口接入系统，而无需修改宿主代码。

核心思想可以用一句话概括：**不改主程序，也能加功能。**

### 1.2 核心软件工程思想

| 思想 | 含义 | 在插件系统中的体现 |
|------|------|-------------------|
| **控制反转（IoC）** | 框架控制流程，插件只响应事件 | 宿主决定何时调用插件，插件只注册回调 |
| **依赖注入（DI）** | 依赖由外部传入，而非内部创建 | 宿主向插件暴露 API 上下文，插件不自行构造 |
| **事件驱动** | 通过事件/钩子解耦模块间关系 | 插件监听文件变更、Git 操作、UI 交互等事件 |
| **沙箱隔离** | 限制扩展代码的影响范围 | 进程隔离 / WASM 沙箱 / 权限系统 |
| **契约优先** | 接口即协议 | 插件 API 版本化、声明式 manifest |

### 1.3 插件系统的三个关键维度

1. **扩展点类型** — 插件能做什么？（命令注册、UI 扩展、数据处理、事件监听）
2. **隔离机制** — 插件崩溃/恶意行为如何不影响宿主？（进程隔离、WASM、权限声明）
3. **生命周期管理** — 插件何时加载、初始化、卸载？（懒加载、热插拔）

## 2. 经典插件系统分析

### 2.1 VSCode

#### 架构模型

VSCode 采用**进程隔离**方案。所有扩展运行在一个独立的 **Extension Host** 进程中（Node.js），与主 UI 进程通过 IPC 通信。

```
┌──────────────┐    IPC    ┌──────────────────┐
│  UI Process  │ ◄───────► │ Extension Host   │
│  (Renderer)  │           │ (Node.js Runtime) │
│              │           │  ┌─────┐ ┌─────┐  │
│              │           │  │Ext A│ │Ext B│  │
└──────────────┘           │  └─────┘ └─────┘  │
                           └──────────────────┘
```

#### 扩展点类型

| 类型 | 说明 |
|------|------|
| 命令注册 | 在命令面板中注册可调用的操作 |
| UI 扩展 | 侧边栏视图、状态栏项、Webview 面板 |
| 编辑器增强 | 语法高亮、代码补全、诊断信息 |
| 语言服务 | LSP 协议接入，提供智能感知 |
| 主题/图标 | 颜色主题、文件图标 |
| 调试器 | DAP 协议接入 |

#### 隔离机制

- **进程隔离**：扩展崩溃不会导致 VSCode 主进程崩溃
- **懒加载**：通过 Activation Events（如 `onLanguage:markdown`）按需激活，不消耗启动资源
- **无直接 UI 访问**：扩展不能直接操作 DOM，必须通过 API 桥接

#### 对 GitTributary 的启示

- 进程隔离太重，不适合轻量级桌面工具
- **Activation Events 的懒加载思想值得借鉴** — 插件只在需要时激活
- **API 桥接模式**是好的 — 插件通过宿主暴露的 API 操作，而非直接访问底层

### 2.2 Obsidian

#### 架构模型

Obsidian 采用**同进程 + API 限制**方案。插件是 JavaScript/TypeScript 模块，直接加载到主进程中，通过 Obsidian API 访问系统能力。

```
┌──────────────────────────────────┐
│          Obsidian Process        │
│                                  │
│  ┌──────────┐  ┌──────────────┐  │
│  │ Core App │  │ Plugin A     │  │
│  │          │◄─┤ onload()     │  │
│  │ Obsidian │  │ onunload()   │  │
│  │ API      │  └──────────────┘  │
│  │          │  ┌──────────────┐  │
│  │          │◄─┤ Plugin B     │  │
│  └──────────┘  └──────────────┘  │
└──────────────────────────────────┘
```

#### 插件结构

```typescript
export default class MyPlugin extends Plugin {
  async onload() {
    // 注册命令
    this.addCommand({ id: 'my-cmd', name: 'My Command', callback: () => {} });
    // 注册视图
    this.registerView(VIEW_TYPE, (leaf) => new MyView(leaf));
    // 注册事件
    this.registerEvent(this.app.vault.on('modify', (file) => {}));
  }
  onunload() {
    // 清理资源
  }
}
```

**manifest.json** 声明插件元数据：名称、版本、最低 Obsidian 版本、依赖等。

#### 扩展点类型

| 类型 | 说明 |
|------|------|
| 命令 | 在命令面板注册操作 |
| 视图 | 自定义侧边栏/面板视图 |
| 编辑器装饰 | 高亮、小部件、行内渲染 |
| 文件/元数据操作 | 读写笔记内容、frontmatter |
| 数据查询 | Dataview 类的自定义查询引擎 |
| 事件监听 | 文件打开、保存、应用启动等 |
| 快捷键绑定 | 为自定义功能绑定热键 |

#### 隔离机制

- **同进程无沙箱** — 插件与宿主共享内存空间，一个插件可以搞崩整个应用
- **隔离靠 API 设计** — Obsidian 通过 TypeScript 类型系统和有限的 API 暴露来"软隔离"
- **插件生态靠审核** — 社区插件经过审核才能上架官方目录

#### 对 GitTributary 的启示

- **API 设计模式非常优秀** — `Plugin` 基类 + `onload/onunload` 生命周期简洁明了
- **manifest 声明式配置**值得采用
- **同进程模式是双刃剑**：简单但不安全。GitTributary 需要考虑 Rust 插件的安全性
- **Obsidian 的数据污染问题**正是 GitTributary 的差异化机会 — 插件产生的数据不应污染笔记 Git 仓库

### 2.3 Tauri

#### 架构模型

Tauri 本身不是传统意义的插件系统，但它的**插件机制**提供了极好的参考。Tauri 插件是 Rust crate，通过 Builder 模式注册到应用中。

```
┌────────────────────────────────────┐
│           Tauri App                │
│                                    │
│  ┌────────────┐                    │
│  │  Webview   │  invoke()          │
│  │  (React)   │ ──────────┐        │
│  └────────────┘           │        │
│                    ┌──────▼─────┐  │
│                    │ IPC Bridge │  │
│                    └──────┬─────┘  │
│         ┌─────────────────┼──────┐ │
│    ┌────▼─────┐   ┌──────▼───┐  │ │
│    │ Plugin A │   │ Plugin B │  │ │
│    │ (Rust)   │   │ (Rust)   │  │ │
│    └──────────┘   └──────────┘  │ │
└────────────────────────────────────┘
```

#### 插件生命周期

```rust
Builder::<R, Config>::new("plugin-name")
    .setup(|app, api| {
        // 初始化：注册状态、启动后台任务
        Ok(())
    })
    .on_navigation(|app, url| { true })    // URL 拦截
    .on_webview_ready(|window| { })        // 窗口创建
    .on_event(|app, event| { })            // 事件循环
    .on_drop(|app| { })                    // 清理
    .invoke_handler(tauri::generate_handler![cmd_upload, cmd_download])
    .build()
```

#### 关键特性

| 特性 | 说明 |
|------|------|
| 命令系统 | Rust 函数通过 `#[tauri::command]` 暴露给前端 |
| 权限系统 | 每个命令需要显式声明 `allow`/`deny` 权限 |
| 状态管理 | `app.manage()` 注册全局状态 |
| 配置驱动 | `tauri.conf.json` 中声明插件配置 |
| 跨平台 | 同一插件可编译到桌面（macOS/Windows/Linux）和移动端 |

#### 对 GitTributary 的启示

- **Tauri 的 Builder 模式和生命周期钩子可以直接复用** — GitTributary 本身就基于 Tauri
- **权限声明机制值得借鉴** — 插件只能访问它被授权的能力
- **前端通过 invoke 调用 Rust 命令的模式**天然适合插件架构
- Tauri 插件是编译时集成的，GitTributary 需要的是**运行时可插拔**的插件

## 3. Rust 生态的插件方案

GitTributary 基于 Tauri + Rust，需要选择一种运行时加载插件的方案。以下是 Rust 生态中三种主流方案的对比：

### 3.1 方案一：动态库加载（libloading）

**原理：** 插件编译为 `.so`（macOS/Linux）或 `.dll`（Windows），宿主通过 `libloading` crate 在运行时 `dlopen` 加载。

```rust
// 宿主端
let lib = Library::new("target/release/libmy_plugin.so")?;
let func: Symbol<fn(&AppContext) -> PluginResult> = lib.get(b"plugin_init")?;
func(&app_context);

// 插件端
#[no_mangle]
pub extern "C" fn plugin_init(ctx: &AppContext) -> PluginResult { ... }
```

| 维度 | 评价 |
|------|------|
| 实现复杂度 | 中等 — 需要定义 C ABI 接口，处理内存所有权 |
| 安全性 | **低** — 插件与宿主共享进程内存，插件可以做任何事 |
| 性能 | **最高** — 零开销，直接函数调用 |
| 跨平台 | 需要为每个平台编译 `.so`/`.dll` |
| 热插拔 | 可以（但需小心，unload 后指针悬空风险） |
| 插件语言 | 只能是 Rust / C / C++ |

### 3.2 方案二：WASM 沙箱（Extism / wasmtime）

**原理：** 插件编译为 `.wasm`，宿主通过 WASM 运行时（wasmtime / wasmer）加载执行。Extism 是在此基础上的高层封装，提供跨语言 SDK。

```rust
// 使用 Extism
let mut plugin = Plugin::new(wasm_bytes, manifest, true)?;
let output = plugin.call::<&str, &str>("process_note", note_content)?;
```

| 维度 | 评价 |
|------|------|
| 实现复杂度 | 低 — Extism 封装了底层复杂性，几天内可集成 |
| 安全性 | **最高** — WASM 天然沙箱，插件无法访问宿主资源除非显式授权 |
| 性能 | 良好 — WASM 有 JIT 编译，接近原生速度（有 ~10-30% 开销） |
| 跨平台 | **最佳** — 同一个 `.wasm` 文件可在任何平台运行 |
| 热插拔 | 天然支持 — 加载/卸载 wasm 实例即可 |
| 插件语言 | Rust、Go、C、JavaScript、Python 等 16+ 种语言 |

### 3.3 方案三：进程隔离（子进程 + IPC）

**原理：** 每个插件作为独立子进程运行，通过 stdin/stdout 或 Unix Socket 与宿主通信。

```
┌──────────────┐   JSON-RPC   ┌──────────────┐
│  Host Process │ ◄──────────► │ Plugin Proc  │
│              │   stdin/std  │  (独立进程)   │
└──────────────┘              └──────────────┘
```

| 维度 | 评价 |
|------|------|
| 实现复杂度 | **最高** — 需要设计通信协议、进程管理、错误恢复 |
| 安全性 | **最高** — 进程级隔离，操作系统级别的安全边界 |
| 性能 | **最低** — IPC 序列化/反序列化开销 |
| 跨平台 | 需要为每个平台编译插件二进制 |
| 热插拔 | 天然支持 — 杀掉子进程即可 |
| 插件语言 | 任意语言 |

### 3.4 方案对比总结

```
安全性    进程隔离 ████████████  WASM ██████████  libloading ████
性能      libloading ████████████  WASM █████████  进程隔离 ██████
易用性    WASM ████████████  libloading ████████  进程隔离 █████
跨平台    WASM ████████████  进程隔离 ████████  libloading █████
插件语言  进程隔离 ████████████  WASM ██████████  libloading ████
```

## 4. 推荐方案：分层插件架构

### 4.1 方案选择

**推荐：WASM（Extism）为主，内部 Rust 插件为辅的分层方案。**

理由：

1. **安全性优先** — GitTributary 处理用户的笔记数据，安全是底线。WASM 沙箱天然隔离插件
2. **跨语言生态** — 用 Rust/Go/JS/Python 都能写插件，降低社区贡献门槛
3. **跨平台一致性** — 同一 `.wasm` 文件在 macOS/Windows/Linux 都能运行
4. **轻量级** — 相比进程隔离，WASM 方案实现简单，资源开销小
5. **Extism 成熟度** — 提供 Rust Host SDK，与 Tauri 技术栈天然契合

### 4.2 为什么是"分层"而不是单一方案

插件系统存在一个根本矛盾：

```
安全 ↔ 性能 ↔ 灵活性

三者不可兼得，必须分层取舍
```

- **内部插件**解决的是**性能和核心功能的可靠性** — Git 操作、文件监听这些高频操作，不可能每次调用都穿过 WASM 沙箱
- **外部 WASM 插件**解决的是**安全和生态开放性** — 社区贡献者提交的代码，必须在沙箱里跑

如果只用一层：

| 纯内部插件 | 纯 WASM 插件 |
|-----------|-------------|
| 生态封闭，只能官方开发 | 高频操作有 WASM 开销 |
| 同进程无隔离，社区插件可以搞崩宿主 | 核心功能也要走沙箱，架构过度设计 |
| 本质就是 monolith | 插件开发者受限于 WASM 能力边界 |

### 4.3 分层的关键：统一 Capability 接口

两层插件对外暴露的接口必须是**统一的**。无论是内部插件还是 WASM 插件，都通过同一套 Capability API 访问系统能力：

```
内部插件 ──► Capability API (Rust trait) ◄── WASM 插件
                │
                ▼
           Git / File / AI / ...
```

这意味着：
1. 内部插件和外部插件在**接口层面是平等的**
2. 未来可以把某个内部插件迁移到 WASM（反之亦然），对用户透明
3. 权限系统只需要在 Capability API 层做一次拦截

### 4.4 架构设计

```
┌─────────────────────────────────────────────────┐
│                GitTributary (Tauri)              │
│                                                  │
│  ┌────────────┐                                  │
│  │  Webview   │  invoke("plugin:xxx|command")    │
│  │  (React)   │ ────────────────────┐            │
│  └────────────┘                     │            │
│                              ┌──────▼──────┐     │
│                              │ Plugin Host │     │
│                              │  Manager    │     │
│                              └──────┬──────┘     │
│                    ┌────────────────┼────────┐   │
│              ┌─────▼─────┐   ┌──────▼─────┐  │   │
│              │ Internal  │   │ WASM Plugin │  │   │
│              │ Plugins   │   │ (Extism)   │  │   │
│  ┌───────────┤(编译时集成)│   │            │  │   │
│  │ Capabilities           │   │ .wasm 文件 │  │   │
│  │ ──────────┤            │   └────────────┘  │   │
│  │ Git 数据  ├────────────┘                    │   │
│  │ AI Agent  │                                 │   │
│  │ 文件处理  │◄─── 统一 Capability API ───────►│   │
│  │ 文本处理  │                                 │   │
│  │ 生命周期  │                                 │   │
│  └───────────┘                                 │   │
└─────────────────────────────────────────────────┘
```

### 4.5 两层插件定义

**第一层：内部插件（Internal Plugins）**

- 编译时集成在宿主中，用于核心功能（Git 操作、文件监听、基础 AI 能力）
- 直接 Rust 函数调用，零开销
- 对应 Tauri 的 plugin 机制，无需额外运行时

**第二层：外部插件（WASM Plugins）**

- 运行时从 `.wasm` 文件加载，面向社区扩展
- 通过 Extism 沙箱执行，只能访问宿主显式授权的 Capability
- 支持 Rust / Go / JS / Python 等多种语言编写

### 4.6 Extism vs 直接用 wasmtime

```
wasmtime / wasmer  ← 底层 WASM 运行时（Rust 原生）
       │
    Extism         ← 高层封装（提供 Host SDK + PDK）
       │
   GitTributary    ← 我们的应用
```

**Extism 做了什么：**

1. **统一了 Host/PDK 接口** — 用任何语言写插件，用任何语言做宿主，接口一致
2. **管理内存传递** — WASM 和宿主之间的数据交换（字符串、JSON）不再需要手动处理指针
3. **提供 host function 注册** — 宿主可以向 WASM 插件暴露函数（这就是实现 Capability API 的机制）

**什么时候应该直接用 wasmtime：**

- Extism 的抽象层引入了不可接受的性能开销
- 需要 WASM Component Model（wasmtime 率先支持的新标准，用于 WASM 模块间组合）
- 需要更细粒度的资源控制（内存上限、执行时间限制）

**判断：** MVP 阶段用 Extism，不碰底层。如果后续性能成为瓶颈，Extism 可以平滑迁移到直接使用 wasmtime — 因为底层运行时是同一个。

## 5. 插件接口设计（映射数据范式）

基于 `开发文档/3.数据范式.md` 中定义的公共能力，插件接口可以这样设计：

### 5.1 插件 Manifest

```rust
/// 插件 manifest — 声明式配置
struct PluginManifest {
    name: String,
    version: String,
    min_host_version: String,    // 最低宿主版本
    capabilities: Vec<Capability>,  // 申请的能力集
    commands: Vec<CommandDecl>,     // 注册的命令
    hooks: Vec<HookDecl>,          // 注册的钩子
}

/// 能力枚举 — 对应数据范式
enum Capability {
    GitRead,          // Git 数据读取
    GitWrite,         // Git 写操作（commit/push）
    GitHistory,       // Git 历史查询
    AgentInvoke,      // AI Agent 调用
    TextProcess,      // 文本处理
    FileRead,         // 文件读取
    FileWrite,        // 文件写入
    LifecycleHook,    // 生命周期钩子
    FileSystemWatch,  // 文件系统监听
}
```

### 5.2 插件可用的钩子（Hooks）

| 钩子 | 触发时机 | 典型用途 |
|------|---------|---------|
| `on_push_before` | Git push 之前 | 内容检查、敏感信息扫描 |
| `on_push_after` | Git push 之后 | 部署静态站、通知 |
| `on_file_change` | 文件增删改 | 索引更新、自动 tag |
| `on_review` | 复习触发时 | 间隔重复算法 |
| `on_render` | 内容渲染时 | Markdown 增强、图表渲染 |

### 5.3 插件目录结构

```
.gittributary/
├── plugins/
│   ├── manifest.json          # 插件清单
│   ├── git-deploy/
│   │   ├── plugin.wasm        # WASM 插件二进制
│   │   └── config.json        # 插件配置
│   ├── spaced-review/
│   │   ├── plugin.wasm
│   │   └── config.json
│   └── ...
├── capabilities.json          # 全局能力授权
└── ...
```

## 6. 插件安全模型

插件安全是一个多层问题，WASM 沙箱只是第一层：

### 第一层：执行隔离（WASM 沙箱解决）

- 插件代码无法直接访问宿主内存
- 插件无法直接读写文件系统
- 插件崩溃不影响宿主

### 第二层：权限控制（Capability 声明解决）

即使在沙箱内，也需要控制插件能做什么：

```json
// plugin manifest
{
  "capabilities": ["git:read", "file:read"],
  "denied": ["git:write", "agent:invoke"]
}
```

宿主在每次 Capability 调用时校验权限。插件申请 `git:write` 但 manifest 中未授权 → 调用被拒绝。

### 第三层：数据安全（宿主逻辑解决）

这是 WASM 无法覆盖的层面。例如：

- 插件读取了用户的笔记内容，返回结果中是否包含敏感信息？
- 插件申请了 `file:write` 权限，写入的内容是否合法？
- 插件申请了 `agent:invoke` 权限，发送给 AI 的 prompt 是否泄露隐私？

这些需要宿主在 Capability API 的实现中加入**业务逻辑校验**，不是沙箱本身能解决的。

### 第四层：供应链安全（生态层面）

- 插件来源是否可信？（签名验证）
- 插件更新是否被篡改？（哈希校验）
- 插件是否有已知漏洞？（审计机制）

这是 MVP 之后需要考虑的问题，但架构设计时要留好扩展点。

## 7. "非侵入式"理念对插件架构的约束

产品文档中 GitTributary 的核心差异化是**非侵入式** — 不污染笔记仓库。这对插件架构有具体约束：

### 约束 1：插件数据与笔记数据物理隔离

```
用户笔记仓库/           ← GitTributary 不写入任何东西
    ├── notes/
    └── ...

.gittributary/          ← 所有插件数据存放在独立目录
    ├── plugins/
    ├── state/          ← 插件运行时状态
    └── ...
```

`state/` 目录应该在 `.gitignore` 中（或者根本不放在笔记仓库内），防止插件状态污染 Git 历史。

### 约束 2：插件对笔记文件的写操作必须是可审计的

如果一个插件要修改用户的笔记文件，需要：

1. 声明 `file:write` 权限
2. 写操作经过宿主的 Capability API
3. 宿主记录变更日志（哪些文件被哪个插件在什么时间修改了）
4. 用户可以回滚插件的修改（利用 Git）

### 约束 3：插件的 UI 不侵入编辑器

GitTributary 的定位是增强 Typora/Vim 等编辑器，而非替代。因此：

- 插件的 UI 只在 GitTributary 自身的界面中展示
- 插件不能注入内容到用户的 Markdown 文件中（不像 Obsidian 生成各种 `.obsidian/` 目录）
- 插件对笔记的"增强"是只读分析为主，写操作为辅

## 8. 前端与插件的交互

### 当前 Tauri 的 IPC 模式

```
React 前端                    Rust 后端
  invoke("cmd", args)  ──►  #[tauri::command] fn cmd(args) -> Result
       │                         │
       └───── Tauri IPC ─────────┘
```

### 插件化后的 IPC 模式

```
React 前端                     Plugin Host Manager              插件
  invoke("plugin:git-deploy|deploy", {path})
       │
       ▼
  Tauri IPC 路由
       │
       ▼
  PluginHostManager::dispatch("git-deploy", "deploy", {path})
       │
       ├─► 内部插件?  → 直接调用 Rust 函数
       │
       └─► WASM 插件? → Extism::call("deploy", json_args)
                              │
                              ▼
                         WASM 插件执行
                              │
                              ▼
                         返回 JSON 结果
```

关键设计点：

1. **前端不需要知道插件类型** — 不管是内部还是 WASM，调用方式一致
2. **路由表由 Plugin Host Manager 维护** — 注册时建立 `plugin_name → handler` 的映射
3. **数据序列化统一用 JSON** — WASM 插件和内部插件都接收 JSON、返回 JSON

## 9. 竞品的插件困境 — GitTributary 的机会

### Obsidian 的插件困境

Obsidian 有 2000+ 插件，但存在结构性问题：

1. **数据污染** — 插件在 `.obsidian/` 目录下产生大量数据，导致 Git 仓库膨胀
2. **无隔离** — 一个插件的 bug 可以搞崩整个应用
3. **同质化** — 大量插件做类似的事情（10 个不同的 daily note 插件）
4. **维护性** — 很多插件作者弃坑，插件与核心版本不兼容

### GitTributary 的机会

| Obsidian 的问题 | GitTributary 的解法 |
|----------------|-------------------|
| 数据污染 Git 仓库 | 插件数据存放在独立目录，不跟踪到笔记仓库 |
| 无隔离 | WASM 沙箱天然隔离 |
| 插件同质化 | 通过 Capability API 限制能力边界，减少冗余 |
| 维护性差 | WASM 插件跨版本兼容性更好（ABI 稳定） |

**一句话总结：** GitTributary 的插件系统应该走"少而精"路线 — 不追求插件数量，而是保证每个插件都是安全的、不污染仓库的、稳定兼容的。

## 10. 反模式警示：不要过早抽象 Capability API

MVP 阶段最容易犯的错误是：试图设计一个"完美的通用 Capability 抽象层"。

正确的做法是：

```
阶段 1：直接硬编码
  - PluginHostManager 直接调用 git2 库读取数据
  - 插件通过 Extism host function 获取数据
  - 没有中间抽象层

阶段 2：发现共性后提取
  - 当多个插件都需要读 Git 数据时，提取 GitRead trait
  - 当内部插件和 WASM 插件都需要相同能力时，统一接口

阶段 3：完善抽象
  - 形成稳定的 Capability API
  - 版本化，保证向后兼容
```

原因：**你还不知道插件真正需要什么能力。** 数据范式中列出的 5 类能力（Git 数据、AI Agent、文本、文件、生命周期）是基于假设的。只有在写完 2-3 个真实插件后，才能知道哪些能力是高频的、哪些是不需要的、哪些接口签名需要调整。

## 13. 主流软件系统插件架构横向深化

### 13.1 全景对比

| 维度 | VSCode | JetBrains | Chrome Extension | Obsidian | Tauri |
|------|--------|-----------|-----------------|----------|-------|
| **隔离机制** | 进程隔离 (Extension Host) | ClassLoader 隔离 | 进程隔离 (Service Worker) | 同进程无隔离 | 编译时集成 |
| **扩展点** | 命令/UI/LSP/DAP/主题 | Action/ToolWindow/Editor/Psi | Content Script/UI/Background | Plugin 基类 | Command/Builder |
| **通信模型** | JSON-RPC over IPC | Plugin API (Java 接口) | chrome.runtime API (消息传递) | 直接函数调用 | Rust ↔ Webview IPC |
| **激活机制** | Activation Events (声明式) | 按需加载 + listener | Events (声明式) | onload() 启动即激活 | 编译时注册 |
| **安全模型** | 进程沙箱 | ClassLoader 沙箱 | Manifest V3 权限声明 | 无沙箱 | Rust 类型安全 |
| **插件语言** | JS/TS | Java/Kotlin | JS/TS | JS/TS | Rust + JS/TS |

### 13.2 VSCode 深化：Contribution Point 与 Activation Events

#### Contribution Point（贡献点声明）

VSCode 的 `package.json` 中 `contributes` 字段是声明式扩展点的经典范式：

```json
{
  "contributes": {
    "commands": [{ "command": "git.deploy", "title": "Deploy Notes" }],
    "viewsContainers": {
      "activitybar": [{ "id": "gittributary", "title": "GitTributary", "icon": "icon.svg" }]
    },
    "views": {
      "gittributary": [{ "id": "pluginSidebar", "name": "Plugins" }]
    },
    "menus": {
      "commandPalette": [{ "command": "git.deploy", "when": "resourceExtname == .md" }]
    }
  }
}
```

**对 GitTributary 的启示：**
- `manifest.json` 应包含 `contributes` 字段，让插件声明自己提供了哪些命令、UI 组件、菜单项
- `when` 条件表达式控制命令的可见性和可用性上下文

#### Activation Events（激活事件）

```json
{
  "activationEvents": [
    "onCommand:git.deploy",
    "onView:pluginSidebar",
    "onLanguage:markdown",
    "workspaceContains:**/*.md"
  ]
}
```

插件只在满足声明条件时才加载，大幅降低启动开销。

**对 GitTributary 的启示：**
- WASM 插件的 manifest 应支持声明触发条件（如 `onGitPush`、`onFileChange`、`onCommand:xxx`）
- Plugin Host Manager 维护事件 → 插件的索引表，按需实例化 WASM 模块

#### Extension Host 进程模型

```
┌──────────────┐    JSON-RPC    ┌──────────────────┐
│  UI Process  │ ◄────────────► │ Extension Host   │
│  (Renderer)  │   (IPC)       │ (Node.js)        │
│              │                │  ┌─────┐ ┌─────┐  │
│              │                │  │Ext A│ │Ext B│  │
└──────────────┘                │  └─────┘ └─────┘  │
                                └──────────────────┘
```

- 扩展崩溃不会导致主 UI 崩溃
- 扩展无法直接操作 DOM，必须通过 API 桥接
- 代价：IPC 序列化开销，内存占用高（每个 Extension Host ~100MB+）

**对 GitTributary 的启示：** 进程隔离太重，但"API 桥接"思想应在 WASM 插件中保留 — 插件只能通过 host function 调用宿主能力。

### 13.3 JetBrains 深化：Action System 与 Psi

#### Action System（动作系统）

JetBrains 将所有用户操作统一建模为 Action：

```java
public class DeployAction extends AnAction {
    @Override
    public void actionPerformed(AnActionEvent e) {
        Project project = e.getProject();
        // 执行部署逻辑
    }

    @Override
    public void update(AnActionEvent e) {
        // 根据上下文决定是否启用/可见
        e.getPresentation().setEnabledAndVisible(
            e.getProject() != null && hasMarkdownFiles(e)
        );
    }
}
```

Action 注册在 `plugin.xml` 中：

```xml
<actions>
    <action id="gittributary.deploy" class="DeployAction" text="Deploy Notes">
        <add-to-group group-id="ToolsMenu" anchor="last"/>
        <keyboard-shortcut keymap="$default" first-keystroke="ctrl shift D"/>
    </action>
</actions>
```

**比 VSCode command 更结构化的地方：**
- `update()` 方法提供上下文感知（根据项目状态动态启用/禁用）
- Action 天然绑定快捷键、菜单位置、工具栏图标
- Action 可以组合（ActionGroup）形成层级菜单

**对 GitTributary 的启示：**
- 插件命令应包含 `update(context)` 方法，支持上下文感知
- 命令应声明式地绑定快捷键和菜单位置，而非命令式注册

#### Psi（Program Structure Interface）

Psi 是 JetBrains 最强大的扩展点，提供语言级别的 AST 访问：

```java
// 查找所有 Markdown 文件中的标题
PsiTreeUtil.findChildrenOfType(markdownFile, Heading.class)
    .forEach(heading -> {
        // 对每个标题进行处理
    });
```

**对 GitTributary 的启示 — Note Structure Interface（NSI）：**

如果 GitTributary 要做笔记智能分析（复习、索引、交叉引用），可以设计类似的结构化接口：

```rust
trait NoteStructure {
    fn headings(&self) -> Vec<Heading>;
    fn links(&self) -> Vec<InternalLink>;
    fn tags(&self) -> Vec<Tag>;
    fn frontmatter(&self) -> Option<Frontmatter>;
    fn code_blocks(&self) -> Vec<CodeBlock>;
}
```

这比直接给插件原始 Markdown 文本更安全、更高效。

#### ToolWindow 扩展

JetBrains 的 ToolWindow 是侧边栏面板的标准注册方式：

```xml
<extensions defaultExtensionNs="com.intellij">
    <toolWindow id="GitTributary" anchor="right"
                factoryClass="com.example.PluginPanelFactory"/>
</extensions>
```

**对 GitTributary 的启示：** 与前端插件菜单的"平整列表/文件夹视图"需求直接对应，manifest 中插件应能声明 `view` 扩展点。

### 13.4 Chrome Extension 深化：Manifest V3 权限模型

#### 声明式权限

```json
{
  "manifest_version": 3,
  "permissions": ["storage", "activeTab"],
  "host_permissions": ["https://github.com/*"],
  "content_scripts": [{
    "matches": ["https://github.com/*"],
    "js": ["content.js"]
  }],
  "background": {
    "service_worker": "background.js"
  }
}
```

**核心设计思想：**
- 权限在安装时声明，用户可审阅
- `host_permissions` 限定插件能访问的网络范围
- `content_scripts` 声明在哪些页面注入脚本
- Service Worker 替代了旧的 background page（Manifest V3 的关键变更）

**对 GitTributary 的启示：**
- 插件 manifest 的 `capabilities` 字段应细化为类似 Chrome 的层级权限：
  - `git:read` / `git:write` — 对应 `permissions`
  - `file:read:path=notes/**` — 对应 `host_permissions` 的路径限定
  - `agent:invoke:model=*` — AI 调用权限可进一步限定模型

#### Content Script 隔离模型

Content Script 运行在页面上下文中，但 DOM 世界隔离：

```
┌─────────────────────────────┐
│  Web Page                   │
│  ┌────────────────────────┐ │
│  │  Content Script        │ │ ← 可以访问 DOM，但 JS 变量隔离
│  │  (独立 JS 世界)        │ │
│  └────────────────────────┘ │
│                             │
│  Page Script (页面自有 JS)  │ ← 看不到 Content Script 的变量
└─────────────────────────────┘
```

**对 GitTributary 的启示：** 类似于插件处理笔记数据 — 可以读取笔记内容（DOM），但不能直接修改编辑器内部状态（JS 变量隔离）。

#### Service Worker 生命周期

```
事件触发 → Service Worker 唤醒 → 执行 → 空闲 30s → 休眠
                                          ↑          │
                                          └──────────┘
```

**对 GitTributary 的启示：** WASM 插件可以借鉴此生命周期 — 不需要常驻内存，事件触发时加载，空闲时卸载释放资源。

### 13.5 对 GitTributary 的综合设计映射

| 设计模式 | 来源系统 | GitTributary 应用场景 |
|---------|---------|---------------------|
| **Contribution Point** | VSCode `contributes` | 插件声明能提供哪些命令/UI 组件/菜单项 |
| **Activation Event** | VSCode `activationEvents` | 插件按需加载，减少启动开销 |
| **Action System** | JetBrains Action | 统一命令注册 + 快捷键 + 上下文感知 |
| **Note Structure Interface** | JetBrains Psi | 结构化笔记 AST 访问，而非原始文本 |
| **ToolWindow** | JetBrains | 插件 UI 面板标准化注册 |
| **Capability Manifest** | Chrome `permissions` | 插件声明所需的系统能力（Git/AI/文件） |
| **路径限定权限** | Chrome `host_permissions` | `file:read:path=notes/**` 细粒度控制 |
| **Content Script 隔离** | Chrome Extension | 插件可读笔记但不侵入编辑器内部 |
| **Service Worker 生命周期** | Chrome Manifest V3 | WASM 插件事件触发加载、空闲卸载 |
| **Lifecycle Hook** | Tauri `setup/on_drop` | 插件加载/卸载/资源清理 |
| **Plugin 基类** | Obsidian | `onload/onunload` 生命周期 + manifest 声明 |
