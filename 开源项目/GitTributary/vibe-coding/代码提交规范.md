# GitTributary 代码提交规范

> 对标主流开源项目（Conventional Commits + Angular 规范 + 常见 OSS 协作约定），
> 结合本项目技术栈（Tauri / Rust / React / TypeScript）裁剪而成。
> 目标：让提交历史可读、可检索、可自动生成 changelog，并方便 AI 与人协作。

---

## 一、提交信息格式（Conventional Commits）

```
<type>(<scope>): <subject>

<body>

<footer>
```

- **首行（header）必填**，`type` + 可选 `scope` + `subject`。
- header 与 body、body 与 footer 之间各空一行。
- 全部使用 **现在时祈使句**（"add" 而非 "added"/"adds"）。

### 1. type（必填，小写）

| type | 含义 | 触发 changelog |
| --- | --- | --- |
| `feat` | 新功能 | ✅ Features |
| `fix` | 修复 bug | ✅ Bug Fixes |
| `docs` | 仅文档 | — |
| `style` | 不影响逻辑的格式（空格、分号、Prettier） | — |
| `refactor` | 重构（非新增功能、非修 bug） | — |
| `perf` | 性能优化 | ✅ Performance |
| `test` | 增删测试 | — |
| `build` | 构建系统 / 依赖（Cargo、npm、Vite、Tauri 打包） | — |
| `ci` | CI 配置（GitHub Actions 等） | — |
| `chore` | 杂项（不属于以上，如 .gitignore） | — |
| `revert` | 回滚某次提交 | — |

### 2. scope（可选，小写）

标明改动影响的模块。本项目推荐 scope：

| scope | 范围 |
| --- | --- |
| `ui` | 前端通用界面、设计 token、主题 |
| `shell` | 插件 Shell / 侧边栏 / 布局 |
| `plugin` | 插件框架、注册表、某个插件 |
| `git` | Rust 端 Git 操作封装 |
| `ipc` | Tauri 前后端通信（command / event） |
| `core` | Rust 核心（PHM、能力 trait、错误类型） |
| `wasm` | WASM 插件相关 |
| `fs` | 文件监听 / 文件系统 |
| `deps` | 依赖升级 |
| `release` | 发版相关 |

> 单次提交跨多个 scope 时，要么拆成多个提交，要么省略 scope。

### 3. subject（必填）— 这是 GitHub 上展示的标题

- **≤ 50 个字符**（硬限制），首字母小写，结尾不加句号
- 描述「做了什么」，而非「怎么做」
- 这一行就是 `git log --oneline` 和 GitHub commit 列表里展示的内容
- **所有细节放 body**，不要塞进 subject

❌ 错误示例（太长,在 git log 里被截断）:
```
feat(store): 配置同步模块 — Git 备份到远程仓库 - 新增 sync.rs - Tauri commands
```

✅ 正确示例:
```
feat(store): 配置同步模块
```

### 4. body（可选）— 展开才看到的详细内容

- 与 subject 之间**必须空一行**
- 解释 **为什么**这样改、具体做了哪些事
- 每行 ≤ 72 字符,列要点用 `-`
- GitHub 上在 commit 详情页/PR 合并信息里展示

```
feat(store): 配置同步模块

- SyncEngine: init/commit/push/pull/sync
- .git-tributary/ 作为独立 Git 仓库管理
- Tauri commands: sync_get_config/set/now/state
- 认证走 ssh-agent,后续从 credentials 读取
```

### 5. footer（可选）

- 关联 issue：`Closes #12`、`Refs #34`。
- 破坏性变更：以 `BREAKING CHANGE:` 开头，单独成段，说明影响与迁移方式。

## 二、示例

```text
feat(shell): 支持侧边栏拖拽调宽并设置宽度下限

右边缘新增拖拽把手，可在 180–360px 间调整；拖到 140px 以下自动收起。
双击把手重置为默认宽度。

Closes #8
```

```text
fix(git): 清理孤儿 cargo 进程后再读取仓库状态

修复反复启动 tauri dev 留下的 cargo 进程死锁在 package-cache 锁上，
导致 get_repo_status 阻塞的问题。
```

```text
refactor(plugin): 用 category 区分扩展插件与系统插件

BREAKING CHANGE: PluginDescriptor 新增可选字段 category，
旧插件无需修改（缺省按 extension 处理）。
```

```text
build(deps): 引入 tailwindcss v4 与 shadcn/ui 组件
```

## 三、分支规范

| 分支 | 用途 |
| --- | --- |
| `main` | 稳定主干，始终可构建、可发布 |
| `feat/<简述>` | 新功能，如 `feat/sidebar-resize` |
| `fix/<简述>` | 修复，如 `fix/cargo-lock-deadlock` |
| `docs/<简述>` | 文档 |
| `refactor/<简述>` | 重构 |
| `release/<version>` | 发版准备 |

- 分支名小写、用连字符 `-`，简短达意。
- **不直接向 `main` 推送**，通过 PR 合入。
- 备份类临时分支（产品文档提到的「下班备份」工作流）建议加前缀 `backup/`，并定期清理。

## 四、Pull Request 规范

- **标题**遵循提交格式：`feat(shell): 支持侧边栏拖拽调宽`，≤ 70 字符。
- **描述**包含：
  - 改了什么、为什么改（动机 / 关联 issue）
  - 如何验证（运行的命令、测试结果、截图）
  - 风险与待办（若有）
- 一个 PR 聚焦一件事，避免巨型混合 PR。
- 合并前必须通过 CI（构建 + 测试 + lint）。
- 推荐合并策略：**Squash and merge**，让 `main` 历史线性、每个功能一条干净提交。

---

## 五、提交粒度与原则

- **小而完整**：每个提交是一个可独立理解、可回滚的逻辑单元。
- **可构建**：每个提交都应能通过 `npm run build` 与 `cargo check`（不留半截编译不过的提交）。
- **关注点分离**：格式化（`style`）与逻辑改动分开提交，便于 review。
- **不混入无关改动**：暂存时用 `git add <具体文件>`，避免 `git add .` 带入无关变更。
- **敏感信息**：禁止提交 `.env`、密钥、token、个人隐私数据（呼应产品的脱敏 / 安全原则）。

## 六、工具链建议（可选落地）

| 工具 | 作用 |
| --- | --- |
| **commitlint** + `@commitlint/config-conventional` | 校验提交信息是否符合规范 |
| **husky** | Git hooks 管理（`commit-msg`、`pre-commit`） |
| **lint-staged** | 提交前只对暂存文件跑 Prettier / ESLint |
| **Prettier + ESLint** | 前端格式与静态检查 |
| **rustfmt + clippy** | Rust 格式与 lint（`cargo fmt` / `cargo clippy`） |
| **commitizen**（`cz`） | 交互式生成规范提交信息 |
| **standard-version / release-please** | 依据提交自动生成 changelog 与版本号 |

推荐的 pre-commit 钩子：`cargo fmt --check` + `clippy` + 前端 `lint-staged`；
commit-msg 钩子接入 `commitlint`。

> 注意（本机环境）：`~/.npm` 缓存曾有 root 权限文件导致 `EACCES`。
> 安装上述工具如遇该问题，用 `npm_config_cache=/tmp/<dir>` 前缀绕过，
> 或一次性修复：`sudo chown -R 501:20 ~/.npm`。

## 七、版本号（SemVer）

遵循 [语义化版本](https://semver.org/lang/zh-CN/) `MAJOR.MINOR.PATCH`：

- `feat` → MINOR（向后兼容的新功能）
- `fix` / `perf` → PATCH
- `BREAKING CHANGE` → MAJOR
- 0.x 阶段（当前 `0.1.0`）API 不稳定，破坏性变更可只升 MINOR。

---

## 八、一句话总结

> `type(scope): subject` 写清「做了什么」，body 解释「为什么」，每个提交都能构建、能回滚、不夹带私货。
