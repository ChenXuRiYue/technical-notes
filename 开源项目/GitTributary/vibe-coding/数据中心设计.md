# 数据中心设计（发散讨论稿）

> 本文档讨论 GitTributary 的**统一数据持久化层**——"数据中心"。
> 定位：所有后端状态的**单一来源(SSoT)**，向上层插件提供存取服务,向下利用 Git 实现版本化备份。

---

## 一、设计目标与职责边界

### 定位

**所有配置项的统一管理。**

### 目标

| 目标 | 说明 |
| --- | --- |
| 统一配置入口 | 所有用户可调的配置项(设置/偏好/策略/模板/提示词)通过数据中心存取 |
| KV 简洁模型 | 核心 API: `get(key)` / `set(key, value)`,最低认知负担 |
| Git 原生备份 | 纯文本、git-diff-friendly,定时/手动 commit+push 到远程 |
| 用户可见 | 面板中分域可视化浏览,中文友好展示 |
| 跨端同步 | Git push/pull 多设备同步 |
| 不侵入笔记 | 独立目录,不污染笔记 git history |
| 多档切换 | Profile 机制,一键切换整套配置 |

### 职责边界(重要——不做 all-in-one god module)

| 归数据中心(配置)　　　　　　　　| 不归数据中心(运行时/业务)　　|
| ---------------------------------| ------------------------------|
| 用户偏好(主题/语言)　　　　　　 | 运行时缓存(git log 索引)　　 |
| 功能策略(自动备份间隔/复习参数) | 业务数据(笔记内容/提交历史)　|
| 模板/提示词(AI 工作流 prompt)　 | 对话历史(AI 聊天记录)　　　　|
| 插件开关与参数　　　　　　　　　| 插件运行时状态　　　　　　　 |
| 仓库配置(路径/远程/认证)　　　　| Git 仓库本身(.git)　　　　　 |
| UI 布局偏好(宽度/模式)　　　　　| 临时 UI 交互态(当前选中文件) |

**判断标准**:用户主动设定的、希望跨设备/跨重启保留的 → 配置,归这里。系统运行时产生的、丢了可以重算的 → 不归。

后续 AI prompt 模板再复杂(多条提示词、工作流编排),它仍然是"配置",放这里合理。但 AI 对话的聊天记录不是配置,不归。
| 统一状态入口 | 所有后端状态(设置、UI 态、插件数据、缓存)都通过数据中心存取,不允许各模块自建存储 |
| KV 简洁模型　| 核心 API 就是 `get(key)` / `set(key, value)`,最低认知负担　　　　　　　　　　　　|
| Git 原生备份 | 数据文件是纯文本、git-diff-friendly,定时/手动 commit+push 到远程仓库　　　　　　 |
| 用户可见　　 | 不是黑箱——用户可以在面板中浏览、搜索、分类查看自己的数据　　　　　　　　　　　　 |
| 跨端同步　　 | 利用 Git push/pull 实现多设备间数据同步　　　　　　　　　　　　　　　　　　　　　|
| 不侵入笔记　 | 数据文件放在独立目录(`.gittributary/`),不污染笔记仓库的 git history　　　　　　　|

## 二、数据格式方案

### 方案 A：JSON Lines（JSONL）—— 推荐

每个命名空间一个 `.jsonl` 文件,每行一个 KV 记录:

```json
{"k":"sidebar.width","v":220,"t":1719187200}
{"k":"theme","v":"light","t":1719187210}
{"k":"sidebar.width","v":256,"t":1719188000}
```

字段:
- `k` — key(字符串,点分层级命名,如 `sidebar.width`、`repo.last_opened.path`)
- `v` — value(任意 JSON 值:string / number / boolean / object / array / null)
- `t` — timestamp(Unix 秒,记录写入时间,用于冲突解决和审计)

**读取规则**:同一 key 多次出现时,取最后一条(append-only log 模型,最新值在最后)。

**优势**:
- 纯文本 → `git diff` 能精确看到「哪个 key 在什么时候改了什么值」
- 追加写入 → 写入开销极低,不需要重写整个文件
- 天然版本历史 → 同一 key 的历史值都保留在文件中,可回溯
- 合并友好 → 跨端冲突时,按 `t` 时间戳取最新即可自动解决
- 人可读 → 用户用任何文本编辑器都能查看

**劣势**:
- 文件会随时间增长(同一 key 反复写入产生历史行)→ 需要 compaction 策略
- 读取需扫描全文件构建 HashMap → 启动时有一次性开销(但 JSONL 解析极快,万级行 <10ms)

### 方案 B：单 JSON 文件

```json
{
  "sidebar.width": 220,
  "theme": "light"
}
```

**优势**:极简,读取不需要扫描。
**劣势**:每次写入需重写整个文件(不是 append);git diff 虽可读但改一个值整个文件会重格式化;无内置历史;并发写入不安全。

### 方案 C：SQLite

**优势**:高性能查询,结构灵活。
**劣势**:二进制文件,git diff 完全不可读;合并冲突无法解决;违背「Git 原生备份」目标。

### 结论

**选 A(JSONL)**,配合 compaction 解决增长问题。

## 三、目录结构

```
<笔记仓库根>/
├── .gittributary/           ← GitTributary 专属目录(gitignore 可选)
│   ├── data/                ← 数据中心文件
│   │   ├── settings.jsonl   ← 用户设置
│   │   ├── repos.jsonl      ← 仓库记录(历史打开过的仓库)
│   │   ├── ui-state.jsonl   ← UI 布局状态(宽度、折叠、选中 tab 等)
│   │   ├── review.jsonl     ← 复习曲线调度数据
│   │   ├── tags.jsonl       ← 笔记标签/元数据索引
│   │   ├── secrets.enc      ← 加密存储(token/密钥,AES)
│   │   └── plugins/
│   │       └── <plugin-id>.jsonl  ← 各插件私有数据
│   ├── cache/               ← 可丢弃的缓存(git log 索引等)
│   └── config.json          ← 数据中心自身配置(备份策略等)
```

### 与笔记仓库的关系

两种模式(用户可配置):
- **内嵌模式**:`.gittributary/` 在笔记仓库根目录下,数据随笔记一起 commit/push(最简单)
- **独立模式**:`.gittributary/` 是一个独立的 git 仓库(git submodule 或完全分离),不影响笔记 history

建议默认**内嵌模式**(零配置即可用),高级用户可切换为独立模式。

## 四、配置档(Profile)

### 概念

一个 Profile 是一套完整的配置快照。用户可以有多个 Profile,动态切换:

| Profile | 适用场景 |
| --- | --- |
| `工作` | 连接公司仓库,高频自动备份,关闭加密 |
| `个人` | 连接个人仓库,低频备份,开启日记加密 |
| `演示` | 脱敏模式,只读,隐藏敏感信息 |

### 存储结构

```
.gittributary/
├── profiles/
│   ├── _active.json          ← 指向当前激活的 profile: {"active": "personal"}
│   ├── work.jsonl            ← "工作" 配置档
│   ├── personal.jsonl        ← "个人" 配置档
│   └── demo.jsonl            ← "演示" 配置档
├── data/                     ← 全局数据(不随 profile 切换)
│   ├── ui-state.jsonl
│   ├── repos.jsonl
│   └── ...
```

### 切换逻辑

- Profile 里的 KV 是 `settings` 域的覆盖值
- 切换 Profile = 加载对应 `.jsonl` 覆盖到内存中的 settings 层
- 全局数据(UI 状态、仓库列表、缓存)不随 Profile 切换
- 前端可以一键切换,也可以在数据中心面板对比不同 Profile 的差异

### API

```rust
store.list_profiles() → Vec<String>           // ["work", "personal", "demo"]
store.active_profile() → String               // "personal"
store.switch_profile(name: &str) → Result<()> // 切换并重新加载
store.create_profile(name: &str) → Result<()> // 基于当前 settings 创建快照
store.delete_profile(name: &str) → Result<()>
```

## 五、域(Domain)与配置元信息

### 为什么需要域

裸 KV(`sidebar.width = 220`)对开发者可读,但对普通用户不友好。需要一层**元信息描述**,让每个配置项有:
- 中文展示名
- 所属域(分类)
- 值类型(string / number / boolean / enum / path)
- 默认值
- 描述(简短说明)

### 域(Domain)定义

| 域 ID | 中文名 | 包含的配置 |
| --- | --- | --- |
| `general` | 通用 | 主题、语言、启动行为 |
| `git` | Git | 默认仓库路径、自动备份间隔、commit 消息格式 |
| `ui` | 界面 | 侧边栏宽度、折叠态、视图模式偏好 |
| `security` | 安全 | 脱敏开关、加密模式、密钥来源 |
| `review` | 复习 | 复习算法参数、通知时间 |
| `ai` | AI | 模型选择、API 端点、上下文长度 |
| `plugins` | 插件 | 各插件的独立配置 |

### 配置项元信息(Schema)

定义在代码中(Rust 端),不是存在 JSONL 里:

```rust
pub struct ConfigMeta {
    pub key: &'static str,           // "auto-backup.interval"
    pub domain: &'static str,        // "git"
    pub label: &'static str,         // "自动备份间隔"
    pub description: &'static str,   // "每隔多少秒自动执行一次 commit"
    pub value_type: ValueType,       // Number
    pub default: Value,              // 300
    pub options: Option<Vec<Value>>, // 枚举选项(如主题: ["light","dark","auto"])
}

pub enum ValueType {
    String,
    Number,
    Boolean,
    Enum,    // 从 options 中选
    Path,    // 文件/目录路径
}
```

前端通过 `invoke("store_schema")` 获取全部配置项的元信息列表,用于渲染可视化面板。

## 七、命名空间(Namespace)—— 数据文件分类

命名空间是数据**文件级**的隔离(每个 `.jsonl` 一个),和域(Domain)是不同层次:
- **域**:面向用户的分类标签,用于可视化分组(一个域的 key 可以散落在多个文件里)
- **命名空间**:面向系统的文件隔离,用于 IO 和权限(一个文件就是一个命名空间)

| 命名空间 | 文件 | 内容 |
| --- | --- | --- |
| `settings` | settings.jsonl | 用户设置(所有域的配置值都写这里) |
| `repos` | repos.jsonl | 仓库记录(历史打开过的仓库) |
| `ui-state` | ui-state.jsonl | UI 布局状态 |
| `review` | review.jsonl | 复习曲线调度数据 |
| `tags` | tags.jsonl | 笔记标签索引 |
| `secrets` | secrets.enc | 加密敏感数据 |
| `plugins/<id>` | plugins/<id>.jsonl | 各插件私有数据 |

## 八、核心 API 设计

### Rust 端(gt-store crate)

```rust
pub struct Store { /* ... */ }

impl Store {
    /// 打开(或初始化)数据中心
    pub fn open(base_dir: &Path) -> Result<Self>;

    /// 获取值
    pub fn get(&self, namespace: &str, key: &str) -> Option<Value>;

    /// 获取值并反序列化为具体类型
    pub fn get_typed<T: DeserializeOwned>(&self, namespace: &str, key: &str) -> Option<T>;

    /// 设置值(立即写入内存 + 异步落盘)
    pub fn set(&self, namespace: &str, key: &str, value: Value) -> Result<()>;

    /// 删除 key(写入一条 value=null 的记录)
    pub fn delete(&self, namespace: &str, key: &str) -> Result<()>;

    /// 列出命名空间下所有 key
    pub fn keys(&self, namespace: &str) -> Vec<String>;

    /// 列出所有命名空间
    pub fn namespaces(&self) -> Vec<String>;

    /// 按前缀查询(如 "sidebar." 开头的所有 key)
    pub fn scan(&self, namespace: &str, prefix: &str) -> Vec<(String, Value)>;

    /// 手动触发落盘(正常情况自动,异常退出前调用)
    pub fn flush(&self) -> Result<()>;

    /// Compaction:重写 .jsonl 只保留每个 key 的最新值
    pub fn compact(&self, namespace: &str) -> Result<()>;

    /// 获取某个 key 的历史值(用于审计/回溯)
    pub fn history(&self, namespace: &str, key: &str) -> Vec<(Value, i64)>;
}
```

### 前端(Tauri Commands)

```typescript
invoke("store_get", { namespace, key })       → Value | null
invoke("store_set", { namespace, key, value }) → void
invoke("store_delete", { namespace, key })     → void
invoke("store_keys", { namespace })            → string[]
invoke("store_namespaces")                     → string[]
invoke("store_scan", { namespace, prefix })    → [key, value][]
```

## 六、读写模型

```
写入路径:
  前端 invoke("store_set") 
    → Rust: 写入内存 HashMap 
    → 追加一行到 .jsonl 文件(fsync)
    → 返回成功

读取路径:
  前端 invoke("store_get")
    → Rust: 从内存 HashMap 直接返回(O(1))

启动时:
  扫描 .gittributary/data/*.jsonl
    → 逐行解析,同 key 后值覆盖前值
    → 构建内存 HashMap

落盘策略:
  每次 set 立即 append(防崩溃丢数据)
  不需要定时批量写——append 本身足够轻量
```

## 七、备份策略(Git 同步)

### 本地 commit

- 触发条件:数据有变更 + 距离上次 commit 超过 N 分钟(用户可配置,默认 5 分钟)
- 或:应用退出前自动 commit 一次
- commit message 自动生成:`data: auto-sync 2026-06-24 01:30`

### Push 到远程

- 触发条件:commit 后自动 push(可配置为手动)
- 使用 gt-git 的 push 能力,复用认证配置
- 失败时静默重试,不阻塞用户操作

### Pull / 合并

- 启动时:自动 pull 远程数据
- 冲突解决:同一 key 取 `t` 最大(最新)的值——JSONL 的 append-only 模型天然支持这个策略
- 合并后重新加载内存 HashMap

## 八、Compaction(压缩)

随着时间推移,.jsonl 文件会增长(同一 key 反复写入留下历史行)。

**Compaction 操作**:重写文件,只保留每个 key 的最新行。

触发条件(可配置):
- 文件行数 > 1000(默认阈值)
- 用户手动触发(面板里的「清理」按钮)
- 备份 commit 前自动执行(保持推送的数据精简)

Compaction 后文件大幅缩小,但历史值丢失。如果需要保留历史,可在 compact 前先 commit(git history 里自然保留了压缩前的完整文件)。

---

## 九、加密存储(secrets)

`secrets.enc` 单独处理:
- 内容是 AES-256 加密的 JSONL
- 解密密钥来源(三选一):
  - macOS Keychain / Windows Credential Store(系统钥匙串)
  - 用户输入的主密码
  - 环境变量(`GT_SECRET_KEY`)
- 前端无法直接读取明文——通过 `store_get_secret` 命令,Rust 端解密后返回

│   └── sync.rs      # 备份同步(调用 gt-git)

## 十三、开放问题(待讨论)

1. **数据仓库位置**:内嵌笔记仓库(`.gittributary/`)还是独立 repo?还是让用户选?
2. **实时性**:set 后是否立即通知前端(类似事件推送)?还是前端每次主动 get?
3. **数据迁移**:如果 KV 结构后续变了(比如 key 改名),怎么做迁移?是否需要 schema version?
4. **多仓库场景**:如果用户打开不同笔记仓库,每个仓库有各自的 `.gittributary/data/` 还是有一个全局 store?
5. **大文件值**:如果某个 value 非常大(如 AI 对话历史),是否应该拆成独立文件引用?
6. **备份粒度**:是整个 `data/` 目录一起 commit,还是按命名空间分别管理?
7. **权限边界**:插件只能读写自己的命名空间,但「设置面板」需要跨命名空间读写——如何管理权限?
