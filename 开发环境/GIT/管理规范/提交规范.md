# 📝 提交规范

## 🔖 格式约定

```
动词：描述
```

- 使用中文撰写，与笔记仓库语言一致
- 动词与描述之间用中文冒号 `：` 分隔
- 描述尽量简洁，一句话说清做了什么
- 不写句号

🪏 **经验**

| 场景 | 使用 | 频率 |
| ---- | ---- | ---- |
| 个人笔记项目，提交风格统一，方便 `git log` 回溯 | 1 | 5 |
| 团队协作项目，统一规范减少 code review 沟通成本 | 1 | |

## 🔖 动词清单

| 动词 | 适用场景 |
| ---- | ---- |
| 新增 | 新建文件、添加功能模块 |
| 更新 | 修改已有内容 |
| 完善 | 补充细节、优化表述 |
| 重构 | 调整结构，不改变功能 |
| 纠正 | 修复错误 |
| 补充 | 追加之前缺失的内容 |
| 删除 | 移除文件或内容 |

## 🔖 不同场景的提交风格

**个人 / 笔记项目**

轻量即可，`动词：描述` 足够。重在记录可回溯，不需要严格类型约束。

```
新增：Agent Readme 文件
完善：mac 快捷键 - 语音转文字
```

**公司项目 — Conventional Commits**

业界主流规范，Angular 团队率先推行，Vue、React 等知名项目均采用。格式为 `type(scope): description`，支持自动化 changelog 生成。

```
feat(auth): 新增 OAuth2 登录
fix(api): 修复分页参数越界问题
docs(readme): 更新部署说明
refactor(user): 抽取用户校验逻辑
```

常用 type：`feat` / `fix` / `docs` / `style` / `refactor` / `test` / `chore`

**开源项目 — Linux 风格**

Linux 内核采用 `子系统：描述` 的格式，描述首字母小写，不加句号。适合底层系统类项目。

```
mm: fix page fault handling in COW path
net/tcp: reduce memory usage in timewait
```

## 🔖 通用原则

无论哪种风格，核心原则相通：

- **原子提交**：一次提交做一件事，方便 revert 和 review
- **同类合并**：碎片化的调试 commit（如 fix typo、wip）在 push 前压缩
- **动词准确**：区分新增 / 更新 / 完善 / 重构 / 纠正，不要混用
