## 🔖 配置 SSH 密钥

🪏 **经验**

| 场景           | 使用 | 频率 |
| -------------- | ---- | ---- |
| 入职、更换电脑 | 2    |      |
|                |      |      |

🔗 引用

进入公司后的首要任务是完成 Git 环境的初始化配置。Git 作为分布式版本控制系统，安装后可实现本地仓库管理，并与 GitHub 等代码托管平台协同开发。对于安全性要求较高的代码仓库，需通过身份验证机制确保操作权限 —— 用户需在本地生成 SSH 密钥对，并将公钥添加至平台的密钥管理系统中。通过这种基于密钥的身份验证方式，可实现安全、免密的代码仓库访问与协作。

## 🔖 取消文件跟踪 git rm --cached

```txt
# 取消单个文件的跟踪（保留在工作区，仅从暂存区移除）
git rm --cached <file>

# 取消整个目录的跟踪
git rm --cached -r <directory>

# 取消跟踪后，将文件加入 .gitignore
echo "<file>" >> .gitignore
git add .gitignore
git commit -m "stop tracking <file>"
```

🪏 **经验**

| 场景 | 使用 | 频率 |
| ---- | ---- | ---- |
| 误提交了配置文件（如 application.yml、.env），需要取消跟踪但保留本地文件 | 1 | |
| 初始化项目时忘记添加 .gitignore，后续补救 | | |

🔗 引用

`--cached` 的含义是只操作暂存区（index），不影响工作区的物理文件。如果不用 `--cached`，`git rm` 会同时删除工作区文件。

## 🔖 暂存工作区修改 git stash

```txt
# 暂存当前工作区和暂存区的修改
git stash

# 暂存时添加备注信息
git stash push -m "描述信息"

# 查看所有 stash 记录
git stash list

# 恢复最近一次 stash（保留 stash 记录）
git stash apply

# 恢复最近一次 stash（同时删除 stash 记录）
git stash pop

# 恢复指定 stash
git stash apply stash@{n}

# 删除指定 stash 记录
git stash drop stash@{n}

# 清空所有 stash 记录
git stash clear
```

🪏 **经验**

| 场景 | 使用 | 频率 |
| ---- | ---- | ---- |
| 开发到一半需要切分支处理紧急 bug，先暂存当前修改 | 1 | 2 |
| pull 远程代码前，先暂存本地未完成的修改 | 1 | |
| 尝试性修改不想提交，临时保存 | | |

🔗 引用

`git stash` 默认不会暂存未跟踪（untracked）的文件。如需包含，使用 `git stash -u`。

## 🔖 压缩提交

**① `git rebase -i` — 交互式压缩，精细控制**

```bash
git rebase -i HEAD~N
```

编辑器中将后面的 `pick` 改为 `squash`（或 `s`），第一个保持 `pick`：

```
pick   abc1234 第一个提交
squash def5678 第二个提交
squash ghi9012 第三个提交
```

保存后弹出 commit message 编辑窗口，默认是所有 message 拼接，改成想要的即可。

**② `git reset --soft` — 一步到位，最快速**

```bash
git reset --soft HEAD~N
git commit -m "合并后的提交信息"
```

所有改动回到暂存区，需手动写 message。

**③ `git merge --squash` — 合并时压缩**

```bash
git checkout main
git merge --squash feature-branch
git commit -m "feat: 新功能描述"
```

所有 commit 合入暂存区，不产生 merge commit，历史一条直线。GitHub/GitLab 的 **Squash and merge** 按钮效果相同。

🪏 **经验**

| 场景 | 使用 | 频率 |
| ---- | ---- | ---- |
| 开发过程中 commit 太多，合并前整理成一个干净的提交 | 2 | 2 |
| PR 合入前，将调试性 commit（如 fix typo、wip）压缩 | 1 | |
| 本地开发分支提交杂乱，push 前压缩保持记录整洁 | | |
| feature 分支合入主分支，想保持历史一条直线 | 1 | 2 |

🔗 引用

选择建议：
- 需要精细控制（保留部分 commit、编辑 message）→ `rebase -i`
- 追求最快、不需要精细控制 → `reset --soft`
- 合并分支时一步完成压缩 → `merge --squash`
