# 工作中遇到的问题

## 依赖冲突

> 依赖冲突并不只是无脑排除就好。有一些严重问题

简单概括依赖冲突是：**项目中通过不同的路径，引入了同一个第三方库（Jar包）的多个不同版本**。
Maven 中解决依赖冲突的方式：

- 最短路径
- 优先声明

依赖冲突解决的标准并不是项目启动就行。依赖冲突的影响一般有几种情况

- 项目启动失败，存在一些项依赖缺失

- RunTime 时，会出现运行异常。最常见的是 NoSuch Method

  > ❓这个 maven 编译时，项目不会发现吗？
  >
  > 📝 Java 的**编译期**和**运行时**是两套完全独立的环境，而 Maven 帮你解决的只是“编译期”的问题。
  >
  > - **编译期**：IDE（如 IntelliJ IDEA）或者 Maven 编译器会去你的本地仓库或依赖树中，找到**包含该方法/类的 Jar 包版本**（通常是高版本）。只要它能找到这个方法的定义，语法检查就能通过，顺利生成 `.class` 字节码文件。此时，编译器并不关心将来真正跑起来的时候，会不会有别的低版本 Jar 包混进来捣乱。
  > - **运行期**：启动项目（比如 `java -jar` 或在 Tomcat 中运行）时，JVM 会去加载 classpath 下的所有 Jar 包。由于发生了依赖冲突，Maven 按照“最短路径优先”等规则，可能最终把一个**不包含该方法的低版本 Jar 包**放进了运行环境。JVM 在执行到你的代码时，拿着编译好的指令去找那个方法，结果发现实际加载进来的旧版 Jar 包里根本没有这个方法，于是直接抛出 `NoSuchMethodError`。
