# 📌 limit offset

查出所有符合条件数据后的最后一道**瘦身关卡**
`LIMIT OFFSET` 并不是查询优化器的捷径，而是在执行计划末端对已定位/已排序结果集进行截断的操作。它无法避免对偏移部分的遍历，因此在大数据分页中成为性能瓶颈。”

## 📄 语法介绍

```mysql
SELECT column1, column2, ...
FROM table_name
WHERE condition
ORDER BY column [ASC|DESC]
LIMIT row_count OFFSET offset_count;
```

> ⚠️ **一般配合 `ORDER BY` 使用**，否则查询结果的行顺序是任意的。

## 📄 底层原理

> 🔍 关键点：**OFFSET 不是“直接跳到第 N 行”，而是“遍历并丢弃前 N 行”**。

简要过程：

```txt
1. 执行 WHERE 条件过滤 → 得到中间结果集 R
2. 对 R 按 ORDER BY 排序（若未排序则按物理顺序读取）
3. 从排序后的结果中：
   - 跳过前 OFFSET 行（逐行计数并丢弃）
   - 返回接下来的 LIMIT 行
```

**性能瓶颈关键：**
当 offset 非常大时，有几个关键影响：

1. 即使只需要最后 10 行，数据库仍需**依次读取前 10,010 行**（10,000 被跳过 + 10 返回）。
2. **回表（可能）**：如果索引不覆盖 SELECT 字段，每行还需回主键索引查完整数据（IO 放大）。

> 📌 即使有索引，**OFFSET 越大，扫描的索引条目越多**，CPU 和 I/O 开销线性增长。

3. 无索引时：数据库必须全表扫描 → 将所有匹配 `WHERE` 的行加载到内存。可能造成巨大的内存压力，以及全表扫描的灾难级 $O(Nlog_N)$ 的复杂度。

## 📄 优化手段

### 🔖 游码

利用上一页的**最后一条记录的排序字段值**作为下一页起点。

### 🔖 延迟关联
先通过索引获取主键，再回表：
```mysql
SELECT t.* 
FROM orders t
INNER JOIN (
    SELECT id 
    FROM orders 
    WHERE user_id = 123 
    ORDER BY created_at DESC 
    LIMIT 10 OFFSET 10000
) tmp ON t.id = tmp.id;
```

### 🔖 业务层限制分页深度

1. 分页只能一页一页往后跳
2. 一次跳转限制深度（一次只能跳 10 页以内）
