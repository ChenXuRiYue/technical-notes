# 📌 Mutex 与共享状态并发

`Mutex<T>` 是互斥锁保护的数据容器:任务可以共享这个容器,但同一时刻只有一个访问者能获得锁并访问内部的 `T`。Rust 把解锁与 `MutexGuard` 的生命周绑定,能防止忘记解锁,但不会自动消除死锁、长时持锁和业务竞态。

## 📄 Mutex 解决什么问题

普通的 `&mut T` 保证某一时刻只有一个可变借用,但多个线程或多个并发任务需要在运行时竞争共享状态。`Mutex<T>` 把这个独占权延伸到运行时:

```text
没有 Mutex
编译器不允许多个可变访问同时存在

使用 Mutex
多个访问者可以共享锁容器
真正访问 T 前必须先获得锁
同一时刻只有一个锁守卫
```

基本使用:

```rust
use std::sync::Mutex;

let counter = Mutex::new(0);

{
    let mut guard = counter.lock().unwrap();
    *guard += 1;
} // guard 离开作用域,自动解锁
```

## 📄 `lock()` 返回的不是 T

`std::sync::Mutex::lock` 的简化签名:

```rust
fn lock(&self) -> LockResult<MutexGuard<'_, T>>
```

分层阅读:

```text
&self
  -> 获得锁不需要 &mut Mutex<T>

MutexGuard<'_, T>
  -> 代表已经持有锁
  -> 内部可访问 T
  -> 守卫销毁时解锁

LockResult<...>
  -> 前一个持锁线程若 panic,锁可能处于中毒状态
```

`MutexGuard<T>` 实现了 `Deref` 和 `DerefMut`,所以代码看起来像在操作 `T`:

```rust
let mut guard = counter.lock().unwrap();
*guard += 1;
```

锁保护的不是某段函数,而是守卫存活期间对内部 `T` 的访问权。

## 📄 RAII 与锁的释放

Rust 使用 RAII:资源获取与对象生命周绑定。`MutexGuard` 的 `Drop` 实现负责解锁。

使用局部作用域提前解锁:

```rust
{
    let mut state = shared.lock().unwrap();
    state.update();
}

do_slow_io(); // 此时已不持锁
```

或显式销毁守卫:

```rust
let mut state = shared.lock().unwrap();
state.update();
drop(state);

do_slow_io();
```

优先让锁作用域尽可能小,不要在持锁时做网络请求、长时文件 I/O 或复杂计算。

## 📄 锁中毒与 `unwrap()`

如果一个线程持有 `std::sync::Mutex` 时 panic,内部数据可能只修改了一半。标准库会将锁标记为 poisoned,后续 `lock()` 返回 `Err(PoisonError)`。

```rust
let guard = shared.lock().unwrap();
```

这个 `unwrap()` 表示:如果锁已中毒,当前线程也 panic。这可以是一种策略,但不是获取锁的固定语法。

需要恢复时可以显式处理:

```rust
let guard = shared
    .lock()
    .unwrap_or_else(|poisoned| poisoned.into_inner());
```

是否可恢复取决于内部数据的不变式,不能只为了避免 panic 而忽略数据可能已不一致。

## 📄 `Mutex<T>` 与 `Arc<Mutex<T>>`

`Mutex<T>` 解决互斥访问,`Arc<T>` 解决多线程共享所有权。当多个线程需要共同拥有一把锁时,常组合为:

```rust
use std::sync::{Arc, Mutex};

let shared = Arc::new(Mutex::new(Vec::<String>::new()));
let worker_state = Arc::clone(&shared);

std::thread::spawn(move || {
    worker_state.lock().unwrap().push("done".to_string());
});
```

```text
Arc<...>   -> 多个线程共同拥有容器
Mutex<...> -> 同一时刻只允许一个可变访问
Vec<T>     -> 真正被保护的数据
```

`Mutex<T>` 并不自动包含 `Arc`。某些框架会自己拥有并分发状态引用,此时业务状态未必需要再手动包一层 `Arc`。

## 📄 从外向内读 `Mutex<Option<T>>`

```rust
Mutex<Option<GitRepo>>
```

一层一层读:

```text
GitRepo
  -> 一个具体仓库对象

Option<GitRepo>
  -> 可能有仓库,也可能没有

Mutex<Option<GitRepo>>
  -> 这个可变的可缺失状态会被并发访问
  -> 读取或替换 Option 前必须先拿锁
```

典型读取:

```rust
let guard = state.repo.lock().unwrap();
let repo = guard.as_ref().ok_or("repository is not open")?;
repo.status()
```

类型变化:

```text
&Mutex<Option<GitRepo>>
  -- lock() --> MutexGuard<Option<GitRepo>>
  -- as_ref() --> Option<&GitRepo>
  -- ok_or(...) --> Result<&GitRepo, Error>
  -- ? --> &GitRepo
```

`as_ref()` 很重要:它只借用 `GitRepo`,不会把仓库对象从共享状态中移出。

## 📄 并发安全不只是“加了锁”

### 🔖 死锁

如果两条路径以不同顺序获取两把锁:

```text
任务 A: 先拿 repo,再等 store
任务 B: 先拿 store,再等 repo
```

两者可能永久等待。应尽量避免同时持有多把锁;无法避免时必须统一获取顺序。

### 🔖 业务竞态

每次访问都没有数据竞争,不代表整个业务操作是原子的:

```text
读取余额 -> 释放锁 -> 计算 -> 再拿锁写回
```

两个任务可能读到同一旧值。需要根据业务不变式决定关键区范围,不能只看单次内存访问。

### 🔖 持锁跨异步等待

一般不应持有 `std::sync::MutexGuard` 跨越 `.await`。这会长时阻塞其他访问者,还可能使 Future 无法满足 `Send`。优先在 `.await` 前复制必要数据并释放 guard;需要异步互斥时再评估 runtime 提供的 async mutex。
