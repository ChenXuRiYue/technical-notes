# 📌 包和 Crate

想象你写代码就像在建房子。一开始你一个人搭个小木屋，所有工具、材料都堆在院子里，没问题。但当你开始建摩天大楼，就需要：

- **分房间**（模块化）：厨房、卧室、机房……各司其职，互不干扰。
- **打包工具箱**（复用）：别人建类似房子时，可以直接拿你做好的“智能门锁模块”或“水电布线方案”，不用重造。
- **明确谁负责什么**（命名与边界）：避免两个人都以为对方装了窗户，结果没装。

Rust 的 **Crate** 就是这样一个“独立交付单元”——它可以是一个库（别人拿来用），也可以是一个可执行程序（最终产品）。而 **包（Package）** 是一个更上层的组织单位，它包含一个或多个 Crate，并附带元数据（比如名字、版本、依赖哪些别人的包）。



**引自AI：**

>你知道乐高吗？每个乐高套装（比如太空飞船）是一个完整的‘包’，里面有说明书（`Cargo.toml`）、零件（源代码），还能拼出一个能玩的东西（程序）。而每一个零件类型——比如‘2x4 红色砖块’——就有点像一个 **Crate**，它是乐高世界里最小的、可以被重复使用的标准单元。Rust 就是用这种方式，让程序员既能自己搭小玩具，也能和全世界的人一起拼巨型城市，还不怕零件撞车。

## 📄 Crate

Crate 是 Rust 编译器的最小单位。Crate 可以包含模块，模块可以定义在其他文件，然后和 Crate 一起编译。

### 什么动作定义了一个 Crate

Rust 没有 `crate xxx {}` 这样的声明语法。**当 Cargo 把某个文件识别为独立编译目标的根文件（crate root）时，这个根文件及其引入的所有 module 就组成一个 crate。**

Cargo 默认按文件位置识别 crate root：

| 创建的文件 | Cargo 的判断 |
| --- | --- |
| `src/lib.rs` | 定义一个 library crate |
| `src/main.rs` | 定义一个 binary crate |
| `src/bin/admin.rs` | 再定义一个 binary crate |
| `src/user.rs` | 只是普通源文件，不会自动成为 crate |

例如：

```text
my-package/
└── src/
    ├── lib.rs        # library crate 的 crate root
    ├── user.rs       # library crate 内的 module
    ├── main.rs       # binary crate 的 crate root
    └── bin/
        └── admin.rs  # 另一个 binary crate 的 crate root
```

```rust
// lib.rs
mod user;
```

`mod user;` 表示把 `user.rs` 作为 module 纳入当前 library crate，并没有创建新 crate。Cargo 最终会把每个 crate root 分别交给 `rustc` 编译。

`cargo new` 只是帮我们创建 package 和对应的 crate root：

```shell
cargo new app-demo       # 创建 src/main.rs，因此定义 binary crate
cargo new library-demo --lib  # 创建 src/lib.rs，因此定义 library crate
```

> 判断方法：这个 `.rs` 文件是被 Cargo 独立交给 `rustc` 的 crate root，还是由 crate root 通过 `mod` 纳入的 module？前者定义 crate，后者不定义 crate。

**Demo**: 使用一个随机数的 Crate

```rust
use std::cmp::Ordering;
use std::io;

use rand::Rng;

fn main() {
    println!("猜数字游戏\n");

    // mut 关键字定义了一个变量。省略则代表定义了一个常量
    let secret_number = rand::thread_rng().gen_range(1..=100);
    loop {
        println!("请输入一个数字\n");
        let mut guess = String::new();

        io::stdin()
            .read_line(&mut guess)
            // 该语句使用处理常用错误
            .expect("读取失败\n");
        let guess: u32 = match guess.trim().parse() {
            Ok(num) => num,
            Err(_) => continue,
        };
        match guess.cmp(&secret_number) {
            Ordering::Less => println!("太小了"),
            Ordering::Greater => println!("太大了"),
            Ordering::Equal => {
                println!("正确！");
                break;
            }
        }
    }
}
```

### 🔖 二进制 Crate

二进制 crate（binary crate）的编译结果是可直接启动的可执行程序，例如命令行工具、Web 服务、消息消费者或定时任务。普通二进制 crate 需要 `main` 函数作为程序入口：

```rust
fn main() {
    println!("program started");
}
```

#### 一个文件就是一个二进制 Crate 吗？

**不是。`main.rs` 只是二进制 crate 的根文件（crate root），不等于整个 crate。**

一个 crate 是一次独立编译所包含的整棵模块树。根文件可以通过 `mod` 继续引入其他文件：

```text
src/
├── main.rs      # binary crate root
├── cli.rs       # crate 内的 cli module
└── commands.rs  # crate 内的 commands module
```

```rust
// src/main.rs
mod cli;
mod commands;

fn main() {
    let args = cli::parse();
    commands::run(args);
}
```

上面的 `main.rs`、`cli.rs` 和 `commands.rs` 会一起组成**一个** binary crate，而不是三个 crate。`rustc` 从 `main.rs` 出发，遍历它声明的模块树，最终生成一个可执行文件。

library crate 也一样：`lib.rs` 只是它的 crate root，它同样可以通过 `mod` 拥有很多文件和模块。两者在“能否包含多个模块”上没有区别，主要区别是：

| 类型 | crate root | 是否有程序入口 | 主要产物 |
| --- | --- | --- | --- |
| binary crate | `main.rs` 或 `src/bin/*.rs` | 有 `main` | 可执行程序 |
| library crate | `lib.rs` | 没有 `main` | 供其他 crate 调用的库 |

#### 一个 Package 中的多个二进制 Crate

Cargo 默认将下列文件识别为不同的二进制 crate root：

```text
my-package/
└── src/
    ├── main.rs        # 二进制 crate：my-package
    └── bin/
        ├── admin.rs   # 二进制 crate：admin
        └── worker.rs  # 二进制 crate：worker
```

它们是三个独立的编译单元，各自拥有 `main`，并生成三个可执行文件：

```shell
cargo run --bin my-package
cargo run --bin admin
cargo run --bin worker
```

##### 二进制 Crate 一定要放在 `src/bin` 中吗？

**不一定。`src/main.rs` 和 `src/bin/` 只是 Cargo 无需配置就能自动识别的约定路径。** Cargo 默认识别：

```text
src/main.rs            # 名称默认等于 package 名
src/bin/admin.rs       # 名为 admin
src/bin/worker/main.rs # 名为 worker
```

如果想使用其他路径，可以在 `Cargo.toml` 中显式声明 binary target：

```toml
[[bin]]
name = "server"
path = "cmd/server.rs"

[[bin]]
name = "worker"
path = "programs/worker.rs"
```

```shell
cargo run --bin server
cargo run --bin worker
```

需要注意，`src/bin` 目录下的每个顶层 `.rs` 文件都会被 Cargo 当作独立的 binary crate root：

```text
src/bin/
├── server.rs  # binary crate
└── helper.rs  # 也会被当作 binary crate，而不是 server 的 module
```

如果 `helper.rs` 只是 `server` 的内部 module，可以放到以该程序命名的子目录中：

```text
src/bin/server/
├── main.rs    # server binary crate root
└── helper.rs  # server crate 内的 module
```

如果多个 binary crate 都需要该代码，更适合将其放入 `src/lib.rs` 所在的 library crate 中共享。

#### 二进制 Crate 如何共享代码

不同二进制 crate 不应该相互复制或引用 `main` 中的业务逻辑。如果多个程序需要共享能力，通常将共享代码放入同一 package 的 library crate，让每个 binary crate 只保留启动和组装逻辑：

```text
my-package/
└── src/
    ├── lib.rs         # 公共业务能力
    └── bin/
        ├── server.rs  # 启动 HTTP 服务
        └── worker.rs  # 启动消息消费者
```

```rust
// src/bin/server.rs
fn main() {
    my_package::start_server();
}
```

```rust
// src/bin/worker.rs
fn main() {
    my_package::start_worker();
}
```

`server` 和 `worker` 是两个 binary crate，`my_package` 是它们共同调用的 library crate。一个项目中常见的多二进制角色包括 Web 服务、后台 worker、定时任务、数据库迁移工具和管理命令。

> 其他项目将该 package 写入 `[dependencies]` 时，通常只使用它的 library crate，不会导入或自动安装这些 binary crate。只有当使用者要直接运行命令行工具或服务时，才会关注 binary crate。

### 🔖 库 Crate

**库 crate**（*Library crates*）并没有 `main` 函数，它们也不会编译为可执行程序。相反它们定义了可供多个项目复用的功能模块。比如 [第二章](https://kaisery.github.io/trpl-zh-cn/ch02-00-guessing-game-tutorial.html#生成一个随机数) 的 `rand` crate 就提供了生成随机数的功能。大多数时间 `Rustaceans` 说的 “crate” 指的都是库 crate，这与其他编程语言中 “library” 概念一致。

#### 在代码中使用库 Crate

例如，`math-kit/src/lib.rs` 通过 `pub` 对外公开函数：

```rust
pub fn add(left: i32, right: i32) -> i32 {
    left + right
}
```

Cargo 下载并编译依赖 package 后，使用者的 Rust 代码通过 **library crate 名**调用该函数：

```rust
fn main() {
    println!("{}", math_kit::add(1, 2));
}
```

`math_kit` 是 library crate 名，`add` 是它通过 `pub` 公开的函数。package 名可以使用连字符 `math-kit`，Rust 标识符不允许连字符，所以默认 crate 名会写成 `math_kit`。

## 📄 包

*包*（*package*）是提供一系列功能的一个或者多个 crate 的捆绑。一个包会包含一个 *Cargo.toml* 文件，阐述如何去构建这些 crate。

包中可以包含至多一个库 crate(library crate)。包中可以包含任意多个二进制 crate(binary crate)，但是必须至少包含一个 crate（无论是库的还是二进制的）。

### 包如何被分享和依赖

**Cargo 分享、下载和管理的是 package；`rustc` 编译的是 package 中的 crate。**

```text
作者编写 package
      ↓
通过本地路径、Git 或 crates.io 分享
      ↓
使用者在 Cargo.toml 中声明 package 依赖
      ↓
Cargo 获取 package，rustc 编译其 library crate
      ↓
使用者在 Rust 代码中调用 library crate
```

例如，使用者的 `Cargo.toml` 可以通过以下三种方式之一依赖 `math-kit` package：

```toml
[dependencies]
# 本地目录：适合本机联调
math-kit = { path = "../math-kit" }
```

```toml
[dependencies]
# Git 仓库：适合团队或未发布版本
math-kit = { git = "https://example.com/math-kit.git" }
```

```toml
[dependencies]
# crates.io：适合已正式发布的版本
math-kit = "1.0"
```

作者执行 `cargo publish` 时，上传的是整个 package（源代码、`Cargo.toml` 和元数据），而不是预先编译好的通用二进制文件。Cargo 会在使用者的机器上编译其 library crate。

> crates.io 的日常语言会把发布的 package 也简称为“crate”。严格区分时，package 是 Cargo 管理和发布的单元，crate 是 Rust 的编译单元。

**创建包过程：**

```shell
$ cargo new my-project
     Created binary (application) `my-project` package
$ ls my-project
Cargo.toml
src
$ ls my-project/src
main.rs
```

Cargo 会给我们的包创建一个 *Cargo.toml* 文件。查看 *Cargo.toml* 的内容，会发现并没有提到 *src/main.rs*，因为 Cargo 遵循的一个约定：*src/main.rs* 就是一个与包同名的二进制 crate 的 crate 根。同样的，Cargo 知道如果包目录中包含 *src/lib.rs*，则包带有与其同名的库 crate，且 *src/lib.rs* 是 crate 根。crate 根文件将由 Cargo 传递给 `rustc` 来实际构建库或者二进制项目。

在此，我们有了一个只包含 *src/main.rs* 的包，意味着它只含有一个名为 `my-project` 的二进制 crate。如果一个包同时含有 *src/main.rs* 和 *src/lib.rs*，则它有两个 crate：一个二进制的和一个库的，且名字都与包相同。通过将文件放在 *src/bin* 目录下，一个包可以拥有多个二进制 crate：每个 *src/bin* 下的文件都会被编译成一个独立的二进制 crate。

## 💭 反复绊脚

1. 包中可以包含至多一个库 crate(library crate)。包中可以包含任意多个二进制 crate(binary crate)，但是必须至少包含一个 crate（无论是库的还是二进制的）。

   怎么感觉反了？

   **没有反。核心是：一个包只有一个“公共库主体”，但可以有多个“程序入口”。**

   - 库 crate 通过多个 `module` 拆分内部代码，不需要多个库 crate。
   - 每个可执行程序都需要独立的 `main`，所以一个包可以有多个二进制 crate。
   - 如果真的需要多个独立库 crate，就建立多个包，再用 workspace 管理。

   ```text
   一个 package
   ├── src/lib.rs       # 最多一个库 crate，内部可有很多 module
   ├── src/main.rs      # 二进制 crate
   └── src/bin/tool.rs  # 另一个二进制 crate
   ```

   类比 Maven：一个 Rust package 更像一个 Maven module，通常只对外提供一个库，但可以有多个 `main` 程序。

2. 二进制 crate 和库 crate 有什么区别？

   **库 crate 负责“提供能力”，二进制 crate 负责“启动程序”。**

   | 类型 | 常见根文件 | 是否需要 `main` | 编译结果 |
   | --- | --- | --- | --- |
   | 库 crate | `src/lib.rs` | 不需要 | 供其他代码调用的库，不能直接运行 |
   | 二进制 crate | `src/main.rs` 或 `src/bin/*.rs` | 需要 | 可直接运行的程序 |

   两者都可以包含很多 module。在同一个 package 中，通常由库 crate 实现业务功能，由二进制 crate 的 `main` 调用它。

   类比 Java：库 crate 像供别人调用的 JAR；二进制 crate 像包含 `public static void main` 的应用程序。

3. 我还是感受不到 package 和 crate 的区别。

   看动词：

   - Cargo **下载、发布、管理依赖** package。
   - `rustc` **编译** crate。
   - Rust 代码通过 `crate_name::item` **调用** library crate 的公开内容。

   **`Cargo.toml` 中依赖的是 package，`src/*.rs` 中使用的是 crate。**

4. 到底哪个动作定义了 crate？

   **没有专门的 crate 声明；Cargo 将一个文件识别为 crate root，并作为独立目标交给 `rustc` 编译时，就定义了一个 crate。**

   `src/lib.rs`、`src/main.rs`、`src/bin/*.rs` 默认是 crate root；由 `mod` 引入的普通 `.rs` 文件只是 module。

5. 拥有多个二进制 crate 和一个库 crate 的 package 长什么样？

   ```text
   my-package/
   ├── Cargo.toml
   └── src/
       ├── lib.rs          # 唯一的 library crate root
       ├── service.rs      # library crate 内的 module，不是 crate
       ├── main.rs         # binary crate：my-package
       └── bin/
           ├── admin.rs    # binary crate：admin
           └── worker.rs   # binary crate：worker
   ```

   这个 package 共有四个 crate：`lib.rs`、`main.rs`、`admin.rs` 和 `worker.rs`。三个 binary crate 相互独立，但都可以调用 `lib.rs` 所在 library crate 提供的公共能力。

   ```shell
   cargo run --bin my-package
   cargo run --bin admin
   cargo run --bin worker
   ```

   **记忆：一个 `lib.rs` 集中提供能力，多个 `main` 提供不同的程序入口。**
