# 📌 变量声明

TypeScript / JavaScript 声明变量有三种关键字：`var`、`let`、`const`。三者差别不在「能不能用」，而在**作用域**、**提升行为**、**能否重新赋值**。读项目代码时，文件顶部一组声明就是一张清单：它告诉我们这个文件把哪些值当作固定常量、哪些值会在后续被改写。

在 GitTributary 的 `src/App.tsx` 中，变量声明回答了一个问题：

```text
App.tsx 这个前端外壳，声明了哪些绑定？哪些是固定常量，哪些会被改写？
```

## 📄 三种声明方式总览

```text
const  → 块级作用域，不可重新赋值（首选）
let    → 块级作用域，可重新赋值（确实需要改时才用）
var    → 函数作用域，有提升，基本不用
```

| 关键字 | 作用域 | 提升 | 重新赋值 | 初始值 |
| --- | --- | --- | --- | --- |
| `const` | 块级 | TDZ（声明前访问报错） | ❌ | 必须给 |
| `let` | 块级 | TDZ | ✅ | 可省略 |
| `var` | 函数 | 提升为 `undefined` | ✅ | 可省略 |

判定选 `const` 还是 `let` 一句话：**这个绑定后面还会被重新赋值吗？** 会 → `let`，不会 → `const`。`var` 基本不碰，只在维护老代码时遇到。

## 📄 从 App.tsx 看到的声明现场

`App.tsx` 顶部（组件函数之外）的常量大致如下：

```ts
/** 侧边栏宽度边界（px） */
const COLLAPSED_WIDTH = 56;
const MIN_WIDTH = 180;
const MAX_WIDTH = 360;
const DEFAULT_WIDTH = 208;
const COLLAPSE_THRESHOLD = 140;
const PROJECT_REPO_URL = "https://github.com/ChenXuRiYue/GitTributary";
const NAV_MORE_STATE_NS = "ui-state";
const NAV_MORE_STATE_KEY = "app.nav.more.open";
const NAV_MORE_STATE_TTL_MS = 3 * 24 * 60 * 60 * 1000;
```

App.tsx 这一层几乎全是 `const`——配置值从声明起就不变。组件函数体内才出现 `let`（取消标志）和更多 `const`（state 解构）。先看三件事：声明位置、命名风格、值的写法。

### 🔖 声明位置：模块级 vs 函数内

```text
文件顶层（模块级）  → 整个文件共享，组件每次渲染不会再算一次
组件函数体内        → 每次渲染都会重新求值，通常配 useState/useMemo
```

App.tsx 这些常量都放在文件顶层，因为它们的值与组件实例无关：`COLLAPSED_WIDTH` 是侧边栏收起宽度，全局唯一，不需要每个组件实例各存一份。放顶层 = 省一次 re-render 开销。

### 🔖 命名风格：全大写 + 下划线

```ts
const COLLAPSED_WIDTH = 56;
const NAV_MORE_STATE_TTL_MS = 3 * 24 * 60 * 60 * 1000;
```

全大写 + 下划线是 TS/JS 社区约定，表「编译期常量、不要动它」。这不是语言强制的（TS 没有真正的 `const enum` 语义，`const` 只保证不可重新赋值，不保证值不可变），纯粹是可读性约定。

对比：

```ts
const navItems = modules.map((module) => ({ ... }));   // 普通绑定，小驼峰
```

`navItems` 也是 `const`，但值是运行时算出来的数组，命名用小驼峰。约定大致是：固定字面量 / 配置值用全大写，运行时计算的绑定用小驼峰。

### 🔖 值的写法：表达式 vs 字面量

```ts
const NAV_MORE_STATE_TTL_MS = 3 * 24 * 60 * 60 * 1000;
```

写 `3 * 24 * 60 * 60 * 1000` 而不是 `259200000`。意图自解释：3 天的毫秒数。编译器/运行时会算，可读性远比省几个字重要。这是项目里反复出现的习惯——优先让值的含义可读，而不是写死最终数字。

## 📄 const：锁绑定不锁值

一个常见误区：`const` 不可重新赋值 ≠ 值不可变。

```ts
const arr = [1, 2, 3];
arr.push(4);        // 合法，数组内容变了
arr = [1, 2, 3, 4]; // 报错，重新赋值才被禁止
```

`const` 锁的是「绑定」本身，不是绑定指向的对象。要锁对象内容，得用 `Object.freeze` 或只读类型。

App.tsx 顶层常量全用 `const`，因为它们的值（宽度边界、URL、key 名）从声明起就不变。`const` 还要求**声明时必须给初始值**，不能只声明不赋值：

```ts
const x;       // 报错：必须在声明时初始化
let y;         // 合法，y 是 undefined
```

## 📄 let：需要重新赋值时才用

项目里 `let` 出现在几类典型场景：

**1. 取消标志（async 里防止 setState 到已卸载组件）**

```ts
// App.tsx
let cancelled = false;

useEffect(() => {
  // 异步读取后判断
  return () => { cancelled = true; };
}, []);
```

`cancelled` 后面会被改成 `true`，所以用 `let`。这是 Tauri/React 异步副作用里极常见的模式。

**2. 循环计数**

```ts
// plugins/site/state.ts — FNV hash
let hash = 2166136261;
for (let i = 0; i < value.length; i++) {
  hash ^= value.charCodeAt(i);
  hash = Math.imul(hash, 16777619);
}
```

`hash` 每轮累加、`i` 每轮自增，必须可重新赋值 → `let`。

**3. 游标 / 累加器（遍历时移动指针）**

```ts
// plugins/site/capture.tsx
let cursor = root;
let child = cursor.children.find((node) => node.name === part);
while (child) {
  cursor = child;            // 游标下移
  child = cursor.children.find(...);
}
```

`cursor` 在循环里不断被替换成下一层节点 → `let`。

**4. 中间结果（多步算同一个值）**

```ts
// plugins/git/GitPanel.tsx
let nextRecentRepos: string[] = [];
let pathToOpen = repoPath?.trim() || "";
// 后续根据条件改写 nextRecentRepos / pathToOpen
```

一次函数里逐步算出最终值，中间需要覆盖 → `let`。如果用 `const` 就得写成一连串三元或 `if/else` 直接 return，看哪种更清晰。

## 📄 var：作用域与提升（为什么基本不用）

`var` 是 ES6 之前唯一的声明方式，有两类坑，所以现代代码基本不用。

**1. 函数作用域，不是块作用域**

```ts
function f() {
  if (true) {
    var x = 1;
  }
  console.log(x); // 1，var 泄漏出 if 块
}
```

`let` / `const` 同样写法会在 `if` 外访问报错——块级作用域把 `x` 关在 `{}` 里。`var` 没有块级作用域，只有函数作用域，变量会泄漏到整个函数。

**2. 提升（hoisting）+ 默认 undefined**

```ts
console.log(x); // undefined，不报错
var x = 1;
```

`var` 声明被提升到函数/全局顶部，但赋值不提升，访问到的是 `undefined`。这掩盖 bug——本该报「未定义」却拿到 `undefined` 继续跑。

`let` / `const` 也会提升，但进入**暂时性死区（TDZ）**：声明前访问直接抛 `ReferenceError`，不让你用 `undefined` 糊弄过去。

```ts
console.log(y); // ReferenceError: Cannot access 'y' before initialization
let y = 1;
```

所以默认 `const` / `let`，`var` 只在读老代码时认识即可。新代码里出现 `var` 几乎都是该改的信号。

## 📄 选型决策

| 情况 | 用 |
| --- | --- |
| 配置常量、字面量、不会改的绑定 | `const` |
| 解构出来的 state/setter | `const`（解构绑定本身不改） |
| 后面要 `=` 重新赋值 | `let` |
| 循环计数、累加器、游标 | `let` |
| 取消标志、中间结果 | `let` |
| 维护老代码 | `var`（尽量改成 let/const） |

一条经验：写的时候先用 `const`，编译器报「无法分配到 const」时再改成 `let`。这样默认最严，该松才松。

## 📄 与解构、泛型推断

声明常和解构、类型推断一起出现：

```ts
const [collapsed, setCollapsed] = useState(true);      // 解构 + 推断 boolean
const asideRef = useRef<HTMLElement>(null);            // 显式泛型
const active = modules.find((m) => m.id === activeId) ?? modules[0];
```

- 泛型省略时，TS 从初始值推断类型，`useState(true)` 推断出 `boolean`。
- `??` 是 nullish 合并，只对 `null`/`undefined` 触发回退（不影响 `0`/`""`）。

这部分详见 [[use-state-syntax]]。
