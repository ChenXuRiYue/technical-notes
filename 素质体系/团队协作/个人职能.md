# 📌 个人职能

## 🎯 业务变更影响面盘点

### 📋 场景

一次兼容性修改需求，需要盘点所有受影响的模块。盘点过程中遗漏了一条关键链路，上级在评审时追问，给出了模糊甚至错误的回答。

### 🔍 问题分析

| 层面 | 问题 |
|------|------|
| 盘点方式 | 靠脑子记，没有结构化 checklist |
| 汇报心态 | "差不多"就去汇报，而非"确认了"才汇报 |
| 应对方式 | 被追问时硬答，而不是说"我再确认一下" |

### 💡 核心教训

**模糊回答比说不知道更损害信任。**

- 说"不确定，我回去确认" → 专业、负责
- 说"大概是这样"但说错了 → 不专业、不可靠

### 📐 向上沟通原则

1. **先盘再报** — 任何变更/需求，先自己过一遍影响面，确认无遗漏再汇报
2. **不确定就说不确定** — "我需要再确认一下"永远比瞎猜好
3. **盘点要有结构** — 兼容性/重构类变更，用表格或 checklist 逐项核对，不凭感觉
4. **留 buffer** — 盘点时多问一句"还有没有遗漏的入口/调用方"

### 🧭 盘点思维框架

1. **入口** — 这个东西从哪些地方进来？（调用方、触发条件、用户入口）
2. **出口** — 这个东西影响到哪些地方？（下游、展示层、存储）
3. **关联** — 和它有依赖关系的还有什么？（配置、上下游服务、数据源）
4. **边界** — 什么时候会走到，什么时候不会？（正常路径、异常路径、特殊场景）
5. **回溯** — 谁引入的？当时为什么这么设计？有没有隐藏意图？

### 💬 向上沟通通用准则

| 准则 | 说明 |
|------|------|
| 信息准确 | 汇报的内容必须是经过确认的事实，不是猜测 |
| 主动暴露风险 | 发现问题先说，不要等被问到 |
| 结论先行 | 先说结论/方案，再说过程/细节 |
| 带着选项去 | 提问时带上 2-3 个方案及各自的利弊 |
| 事后闭环 | 说了"回去确认"就一定要给回复 |
