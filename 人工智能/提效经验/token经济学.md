# Token 经济学

## **❓ WHY**

**对个人的意义**

- 虽然公司每月配给 500r 额度。理论上只使用 Mimo 可以大力蹬。但是难免总要拿昂贵模型攻坚，节俭可以囤出更多攻坚机会，降低工作压力
- 节俭不完全等于牺牲模型性能。减少不必要的噪音，有时可以带来模型工作质量上升
- 自费使用模型时，省一点是一点
- 👍 节俭是中华民族的传统美德

**对团队的意义**

- 一些需要大模型项目，非常耗费 token (AI 运维举例，经常出现一次排查两瓶可乐的成本)

**对全人类的意义**

- Token ⬇️ -> 无效计算 ⬇️ -> 耗电 ⬇️ -> 碳排放 ⬇️ -> 温室效应 ⬇️ -> 生物多样性 ⬆️ -> 命运共同体 🤝

## 输入

**特性，现象：**

- 大模型是金鱼记忆，每次输入都把所有上下文全部带上
- 大模型侧具有前缀缓存优化机制
- 每次工具调用都会成为输入，产生中间数据（比如文件查找、代码读取）

**建议**

- 避免会话历史滚雪球，及时使用 /compact 等命令压缩上下文
- 缓存优化的意识
  - 保持静态上下文（System prompt 、Tool / MCP、skill ）等稳定。把静态内容放前边
- 压缩非必要输入
  - 针对系统终端指令做压缩（git、ls、cat、grep）等命令。
    [rtk](https://github.com/rtk-ai/rtk) 项目对常用的终端命令做了定制
  - 针对 AI 的代码检索场景，建立代码图谱

## 调研 AI generated

> 深水区的"主战场"只有两种：① **输入累积型**——多轮把上下文反复重发，随轮数平方膨胀，最不可控；② **输出体量型**——一次性生成一大坨，烧最贵的输出 token。看清一个任务的深水落在哪边，就知道该用哪个杠杆。下面四个高频场景是不同配比的代表。

### 速查表

| 场景                | 主深水                                 | Token 画像              | 可控性 | 核心抓手                                    |
| ------------------- | -------------------------------------- | ----------------------- | ------ | ------------------------------------------- |
| 实现一个需求        | 输入累积 + 输出（双高）                | 长会话、多轮、读+写都重 | 🟡 中   | 精准定位相关代码 + diff 输出 + 分段 compact |
| 解决一个 bug        | 输入累积（平方膨胀）                   | 多轮试探、输出小        | 🔴 低   | 开局喂准线索砍轮数；难 bug 直接上强模型     |
| Spring 后台基架     | 输出体量                               | 输入小、输出巨大        | 🟢 中高 | 脚手架工具 / 代码生成器，别拿贵模型当打字机 |
| 了解某技术（Dubbo） | 输出体量（问概念）/ 输入累积（啃源码） | 取决于怎么问            | 🟢 高   | 便宜模型 + 分层提问，先要大纲再按需深入     |

### 地基 · 一次 LLM 调用的本质

> 整篇笔记的所有现象，都是这**一个原子事实**的不同侧影。先拿住它，后面全串得起来。

**一次 LLM 调用 = 一次无状态的序列补全**：喂进一串 token，吐出一串 token，是个纯函数 `f(token 序列) → token 序列`。三个内核：

1. **无状态**——模型不记得上一次调用、也不记得"上一句"。每次都从白纸开始读完整个输入序列。你感受到的"对话 / 记忆"，是外层程序**每轮把完整历史重新塞进输入**伪造出来的连续感。
2. **自回归**——一次调用内部 = 1 次并行 `prefill`（把输入一次读完）+ N 次串行 `decode`（一次吐一个 token、接到序列尾再跑一遍），直到吐出停止符或撞 `max_tokens`。
3. **每 token 算力定额**——模型不能"在一个 token 上想得更深"，想更深只能多吐 token（思考链就是把推理外化成 token）。

**由这一行，推出整套经济学：**

| 本质                 | 推论                                                    |
| -------------------- | ------------------------------------------------------- |
| 无状态、不记事       | 历史必须每轮重发 → 输入累积、平方膨胀（**深水区根源**） |
| 1 prefill + N decode | 输入便宜（并行）、输出贵 3~6 倍（串行一个个挤）         |
| 每 token 算力定额    | token = 算力 = 钱；想更深 = 烧更多 token                |
| 概率补全、无完成裁判 | 靠预测"停止符"收尾 → 会啰嗦 / 截断                      |
| 纯函数、前缀确定     | 相同前缀可复用 KV → prompt caching 打 1/10              |

> 一句话：它不是"持续思考的智能体"，而是"**被反复唤醒、每次读完全部上下文补一段、然后彻底遗忘**"的函数。



- 只要思路 → "给证明脉络、跳过逐步代数"，直接砍掉最贵的输出体量。
- 要完整证明 → 这种有标准答案、不依赖你代码库推理的纯知识题，**别用 opus**，丢给便宜模型（Mimo / DeepSeek）即可，输出价省一个数量级。



### 例 1 · 实现一个需求 —— 双引擎烧钱

**表现**：会话天然分两段叠加，是总量最大的一类。

- **理解段**：读现有代码、约定、相关模块——多轮，输入累积。
- **编写段**：生成 / 修改多个文件——输出大。
- **修正循环**：写完 → 编译报错 → 改 → 再编译，每一轮把前面全部历史重发一遍。

**深水**：理解段的 context 重发 ＋ 编写段若"全文重写文件"则输出爆炸 ＋ 修正循环把前面所有内容再算一遍。两种深水同时存在。

**抓手**：① 用符号检索精准定位要改的那几处，别整目录读；② 强制 diff / patch 输出，禁止全文重写；③ 理解段结束、进入编写前做一次 compact，把"读懂了什么"压成几句结论，扔掉原始大段代码。

### 例 2 · 解决一个 bug —— 平方级输入累积

**表现**：典型工具调用循环。一句"帮我修这个 bug"→ Agent 自主走 N 轮（读文件 → 跑测试 → 再读 → 再跑）。每轮把"目前全部上下文"重发，早期资料被反复计费。

**算账**（假设 8 轮、每份资料约 2000 token、系统前缀 3200）：

| 轮次     | 累积上下文 | 本轮计费输入 |
| -------- | ---------- | ------------ |
| R1       | 3,200      | 3,200        |
| R2       | +2,100     | 5,300        |
| R3       | +2,100     | 7,400        |
| …        | …          | …            |
| R8       | +2,100     | 17,900       |
| **合计** |            | **≈ 84,400** |

独一无二的信息量 ≈ 8 × 2000 ＝ **16k**，实际计费 ≈ **84k**——**同样的资料平均付了 5 遍**。轮数翻倍 → 账单翻**四倍**（平方）。

**不可控点**：轮数由 bug 难度 ＋ Agent 策略决定；这些字你没打、重发你看不见。

**抓手**：① 开局就把精确报错栈、复现步骤、可疑文件喂准，砍掉盲目试探的轮数；② 难 bug 用强模型一次定位，往往比便宜模型多轮试探的总成本更低（**少走轮数 > 单价便宜**）。

### 例 3 · Spring 后台管理基架 —— 输出怪物（样板代码）

**表现**：输入极小（"搭个带 RBAC 的 Spring Boot 后台基架"几句话），输出极大——`pom.xml`、配置、实体、Controller / Service / Mapper、SecurityConfig、统一返回体、全局异常处理……几十个文件，**清一色样板代码**。是"输出怪物"，但烧在代码体量而非思考链；多轮生成时已生成文件还会累积重发。

**关键认知**：样板代码**价值密度极低**（全网千篇一律），却占着最贵的输出 token。

**抓手**：① 用 Spring Initializr / Maven archetype / 代码生成器出骨架，让 AI 只补"非模板的定制逻辑"；② 真要 AI 生成就用便宜模型——样板不需要推理；③ **别用 opus 当打字机逐字吐样板**，这是 token 经济学里最亏的用法之一。

### 例 4 · 了解某技术（如 Dubbo）—— 看你怎么问

**表现**：两种问法天差地别。

- **问概念**（"Dubbo 的 SPI 机制是什么"）：纯知识题，输出型，像素数定理——量大但可控、可预测。
- **啃源码**（"读 Dubbo 源码给我讲清楚服务注册流程"）：触发大量检索 ＋ 源码 / 文档灌入上下文 ＋ 多轮，瞬间掉进输入累积深水。

**关键认知**：纯知识题**不依赖你的代码库**，是最适合便宜模型的场景；一旦让它"去读真实源码"，性质就从可控的输出型变成不可控的输入累积型。

**抓手**：① 纯概念用便宜模型；② **分层提问**——先要一页大纲 / 脉络，再就感兴趣的点按需深入，别一次让它把整个 Dubbo 讲完（一次讲完 ＝ 一大坨最贵输出 ＋ 大量你用不上的内容）；③ 真要啃源码就限定范围（指定模块 / 类），别放它满仓库爬。

### 一句话规律

> **输出体量型**深水（脚手架、问概念）→ 控"让谁生成、生成多少"：便宜模型 ＋ 砍输出 ＋ 分层提问。
> **输入累积型**深水（bug、读源码、需求理解段）→ 控"放多少进上下文、走多少轮"：精准喂 ＋ compact ＋ 砍轮数。

### 一个反直觉真相 · "我问得短，所以便宜"是错的

> 最普遍的误解：把"输入 token"等同于"我打的字"。实际上你那句话从来不是单独发出去的。

**输入 token 的精确定义**

```
单次调用输入 = tokenize(系统提示 + 工具定义 + 全部历史 + 本轮内容 + 注入上下文)
任务输入总量 = Σ（每一次调用的完整输入）
```

注意是"每次调用的完整输入之和"，**不是"你给过的不同东西之和"**——每次调用都重新驮一遍历史，同一块内容被反复计入：

```
输入 token 总量 = 独一无二信息 × 它被重发的次数
```

**一句话提问里的隐形背包**（你没打、也看不见，却每轮重发）：

| 装了什么                       | 量级（agent / CLI 环境） |
| ------------------------------ | ------------------------ |
| 系统提示词 + 规则              | 3k – 8k token            |
| 全部工具定义（哪怕一个没调用） | 5k – 15k token           |
| 整段会话历史（每轮重发）       | 累积，可达几万           |
| 你打的那句话                   | ~15 token                |

> 同一句"Spring Cloud 是什么"，在干净对话框里输入十几 token，在装了一堆工具、聊了二十轮的会话里能到 3 万+——**差三个数量级，全在环境，不在问题。** 缓存能把固定前缀打到 1/10，但减不掉"之和"本身；真要瘦身得靠 compact / 砍轮数。

### 关键分界 · 答案在不在模型权重里

通识知识在训练时已"烤进"权重，模型从参数记忆直接生成，**不检索、不读文件、零补充资料** → 输入小、便宜。答案不在权重里，就必须把资料灌进上下文 → 输入大。这条线划开"便宜可控"与"深水烧钱"：

| 问题性质                                       | 在权重里 | 要喂资料   | 输入画像      |
| ---------------------------------------------- | -------- | ---------- | ------------- |
| 通识 / 公开知识（Spring 思想、Dubbo SPI 概念） | ✅        | 否         | 小、便宜      |
| 私有代码                                       | ❌        | 必须读文件 | 大            |
| 训练截止后的新事                               | ❌        | 必须检索   | 大            |
| 特定项目上下文（这个 bug / 这个需求）          | ❌        | 必须喂     | 大 + 多轮累积 |

**代价权衡**：凭记忆答便宜，但可能记错 / 过时 / 幻觉。所以——**稳定通识凭记忆（省），精确易变的事实喂资料查证（该花）**。这与评测框架里"减噪 vs 减信息只有一线之隔"是同一判断。

### 机制补充 · 模型怎么知道"答完了"

它不知道，也没有完成度裁判。**停止 = 预测出一个停止符**：训练里"完整答案后面跟结束符"见得多了，当生成内容"长得像写完了"，停止符就成了概率最高的下一个 token，生成即止。硬兜底是 `max_tokens`。这也解释了它为何会啰嗦（该停不停）或截断（没答完就停）——全程没有"要求是否达成"的判断，只有"这里像不像该收尾"的模式匹配。

### 调用次数怎么预判 · 1 次 vs N 次

精确次数是黑盒，事先算不出；但**量级可靠预判**，判据只有一句：**回答这道题需不需要"碰外部世界"，碰几次？**

| 问题类型                                        | 调用次数 | 为什么                                      |
| ----------------------------------------------- | -------- | ------------------------------------------- |
| 知识 / 纯推理（1+1=2、Spring 是什么、写个正则） | **1 次** | 全在权重内部，不碰外部                      |
| 要读代码 / 跑测试 / 查最新资料                  | **N 次** | 每"看一眼 / 动一下"算一轮，N 由难度临场决定 |

> "调用次数"数的是模型**停下来做一个动作、再回来**的次数（≈ 工具轮数 + 1），不是它思考了几步——思考链是单次调用内部的输出 token，不算额外调用。

### 面对黑盒，你能控制的六件事

> 心态转变：你控制不了它每一步怎么走子，但你**设计整张棋盘**。模型决定走法，边界条件由你定。按杠杆从大到小：

1. **要不要进多轮**（最大杠杆）：纯知识 / 纯推理题放**干净、少工具**的环境问，锁死在 1 次；别在重型 agent 会话里顺口问通识题（等于给一堆用不上的工具定义和历史付费）。
2. **多轮时走几轮**：①前置喂料——开局把报错栈 / 复现步骤 / 相关文件给全，省掉"摸索发现"的轮次；②限定范围和目标，减少它选错子目标空转；③设最大轮数 / token 上限；④难题直接上强模型（少空转 > 单价便宜）。
3. **每一轮多大（输入）**：compact 砍历史、精准读（outline → range）、稳定前缀吃满缓存。
4. **输出多大（最贵）**：要思路 / 大纲 / diff，而非完整长篇 / 全文重写；简单任务别开重思考。
5. **把黑盒变灰盒（观测）**：盯 harness 给的 token 用量 / 轮数计数器，对成本有体感。
6. **止损**：发现它在"打转"（反复读同一批、来回推翻）→ 立刻打断、重新限定范围。这是你最直接的权力。

**一句话**：你无法预测黑盒内部，但"**这题碰不碰外部世界**"决定 1 次还是 N 次；而"**喂什么进上下文、怎么限定范围、什么环境、设什么上限、要什么输出、何时打断**"这六件事，能把不可控的多轮往可控里拽。

### 多次调用是谁判定的 · 模型提议，harness 裁决

> 多轮不是模型"自己决定连着跑"——模型无状态，跑完一次就遗忘、就停，没法自我续命。多轮是**外层程序（harness / agent runtime）**用循环造出来的。两方分工。

**控制流（ReAct 循环）：**

```
harness 发起第 1 次调用
  └ 模型输出 → harness 解析这是什么：
       ├ 是【工具调用】？ → harness 执行工具(读文件/跑命令/搜网)
       │                  → 结果追加进上下文 → 发起下一次调用 ↺
       └ 是【最终答案】(停止符、且没要工具)？ → 循环结束
  直到：模型不再要工具，或撞上 harness 上界(max_iterations / token 预算 / 超时)
```

- **模型那侧（只负责"举手"）**：一次调用里输出"工具调用"而非最终答案 = 提议"我要做个动作再继续"。但这"举手"**也是预测出来的**——训练 + 系统提示里的工具描述让"调某工具"成为当前最可能输出，不是真实意图。
- **harness 那侧（真正的循环控制者）**：模型无法让自己再跑一次；是 harness 解析输出 → 执行工具 → 把结果喂回 → 决定要不要拿新上下文再唤醒模型一次，并握有硬上界（撞了就强停）。

**反直觉点**：模型无法自主连续运行。你感受到的"它在连着干活"的连续感，完全是 harness 用「循环 + 每轮重发历史」伪造的——这正是输入累积的来源。

**控制点落在两边**：

- 让模型**少举手**：喂准上下文、限定范围 → 它没必要调那么多工具 → 轮数自然少。
- 让 harness **早收敛**：设 max_iterations / token 预算、发现空转就打断 → 强制封顶。

### 思维链（CoT）怎么理解

**本质**：让模型在给最终答案前，先把中间推理步骤当成普通文字一个个写出来。**它"想"的过程就是它"写"的过程**——没有藏在背后的独立思考器，思考即生成。

**为什么管用（接"地基"）**：每个 token 前向算力定额，模型在一个 token 内能做的串行推理深度有限，难题单次前向算不出来。CoT 把推理**外化成 token 流、拿生成的文字当草稿纸**，把难题拆成几十步"预测下一个 token"，每步基于前面已写出的步骤继续推——**用 token 数量换计算深度**。

> 理论：单次前向是有限深度电路；CoT 通过生成中间 token，让有效计算深度随 token 数增长，从而解出单次前向解不了的题。

**它真在用自己写的草稿**：自回归——生成第 k 步时，前 k-1 步已在上下文里，模型看着自己刚写的推理往下接。中间步骤真实参与后续计算，不是表演。这就是"让它一步步想"能实打实提准的原因。

**两种形态（都按输出 token 计费）**：

- 显式 CoT：prompt 引导"一步步想"，推理写在正文、看得见。
- Reasoning 模式：推理模型在专门"思考区"生成大量推理 token，通常对用户隐藏，但照样按输出收费——燃烧图里看不见的红色深水。

**Token 经济学含义**：

- 思维链 = 输出 token，而输出最贵；难题的思考链可能比答案还长，是账单隐形大头。
- 想更深 = 烧更多 token，没有免费的深思（同"钻牛角尖深度 = token"）。
- 双刃：难题上提准确率值这钱；简单题上开重思考就是空烧，还可能"一本正经推一长串错的"（推理链顺 ≠ 结论对）。
- 可控点：简单任务关掉 / 调低 reasoning effort；要思路时让它"简要推理"而非长篇演算；只在真难题上放它深想。

> CoT 不是模型"开窍"，它仍只是在预测下一个 token，被训练 / 引导成"先写推理再写结论"。威力来自"把串行计算摊进 token 流"；代价也在这——思考越深、输出 token 越多、钱越多。

### LLM 调用常见参数 · 意义与 token 关系

> 这些参数本质都在调两样东西：**采样过程**（从词表概率分布里怎么选下一个 token）和**循环边界**。标 💰 的是直接拧得动的 token 经济学旋钮。

**一、控制"多随机 / 多发散"**（动"怎么挑 token"）

- **temperature** 温度：把概率分布压尖还是压平。低=确定保守（代码/事实题 0~0.3），高=随机有创意（头脑风暴 0.7~1+）。temperature=0 ≈ 确定性输出。
- **top_p** 核采样：只从"累积概率前 p"的头部 token 里采样。和 temperature 二选一调。
- **top_k**：只从概率最高的 k 个候选里选（部分 API）。

**二、控制"多长 / 何时停"** 💰

- **max_tokens**：输出硬上限，撞了强制截断 → 封顶输出花费、防失控空烧。
- **stop / stop_sequences**：自定义停止符，遇到指定字符串就停。
- **n**：一次生成几个候选 → n 份输出 = n 倍输出费用。

**三、控制"重复 / 啰嗦"**

- **frequency_penalty**：惩罚已高频出现的 token，压车轱辘话。
- **presence_penalty**：惩罚已出现过的 token（不论次数），鼓励换新话题。

**四、控制"想多深"（推理模型）** 💰

- **reasoning_effort / thinking budget**（low/med/high 或 token 预算）：控思维链长度/深度。高=更准但更贵（思考链是最贵的隐形输出），简单题调低。

**五、控制"可复现"**

- **seed**：固定随机种子，配合 temperature=0 让输出尽量可复现（调试/测评用）。

**六、工具与结构** 💰

- **tools / functions**：提供工具定义——本身占输入 token、**每轮重发**，工具越多固定输入越重。
- **tool_choice**：强制/禁止/自动选工具。禁掉无关工具 = 减少模型乱举手的轮数。
- **response_format**（JSON mode）：强制结构化输出，减少废话。

**七、缓存** 💰

- **cache control**（如 Anthropic）：标记哪段前缀做缓存、TTL（5min/1h）。命中按 1/10 价——省钱第一杠杆的开关。

> 串起来：一~五调"怎么挑 token、挑多少、停在哪"；六、七调"带多少行李、怎么少付重复的钱"。token 经济学旋钮 = `max_tokens` / `reasoning_effort` / `n` / `tools` / `cache`。

### 不同厂家模型差在哪 · 为什么有的好用 · 中美差距

> 模型是"训练出来的纯函数"，差距来自怎么训练出来的；选型别只看跑分，按场景 + 成本选（即"模型分层"）。

**一、差在哪（决定能力与体感的维度）**

1. 预训练数据：质量、规模、配比（代码/多语言/数学占比）、清洗——**决定能力上限**。
2. 规模与架构：参数量、MoE vs dense、上下文窗口、注意力优化。
3. 后训练（**最拉开"好不好用"体感**）：SFT 质量、RLHF 对齐（听话/格式/简洁）、推理训练（数学/代码/CoT）。
4. agentic 训练：有没有专练工具调用、长程任务、指令遵循——**coding agent 体感差距的大头**。
5. 工程：推理速度、稳定性、缓存/工具支持、API 生态。

> 关键：原始"智力"（跑分）只是一部分，"好不好用"主要由后训练 + agentic 能力决定。所以有的模型分高却难用。

**二、为什么有的好用、有的难用**（体感 ≠ 跑分）

- 指令遵循 + 格式稳定（agent 命脉）；工具调用可靠（不乱编、知道何时停 = 少空转）；长上下文真实可用（非标称长却中间遗忘）；简洁不啰嗦（省最贵输出）；一致性稳定。
- 难用 = 后训练/对齐没打磨好：过度对齐（乱拒绝、车轱辘话）或指令遵循差（让 diff 却全文重写）。

**三、为什么美国领先半年到一年（结构性，且在收窄）**

1. 算力：顶级 GPU 获取受出口管制，中国拿货难还要囤；前沿预训练极吃算力。
2. 资本 + 人才密度：头部实验室融资大、顶尖研究员聚集、敢长期烧钱探索。
3. **先发探路时间差**：美国更早探出有效路线（scaling law / RLHF / reasoning RL）。前沿是"探路"、跟随是"修路"，知道"能做到"后复现追赶就快——lag 是探索时间差，非不可逾越的代差。
4. 生态：英文高质量语料、成熟开源社区与工具链。

**但在收窄，且中国有强项**：开源权重（Llama/DeepSeek/Qwen/GLM）加速追赶；DeepSeek 等证明算法+工程创新能部分抵消算力劣势；**成本效率是真优势**（Mimo 输入 3 vs opus 35.89，差一个数量级），很多场景"够用且便宜得多"本身就是核心竞争力。

> 定调：差距主要在"绝对前沿的最强推理"（约半年~一年探路时间差）；在"性价比/实用够用"维度，中国模型非常能打甚至更优。

### 实测案例 · 一次"简单问题"为什么花了 2.88 credit

> 现场抓到的活案例：一句"为什么有的模型好用"感觉很简单，却花了 2.88。它完美印证了「我问得短所以便宜是错的」。（以下为按机制的估算，非精确账单）

**钱花在哪（你的问题本身约 30 token，贡献趋近 0）：**

1. **整段历史每轮重发**：这场已聊二十多轮，前面长答案全在历史里，累计几万 token；每次调用整个重发（模型无状态，不重发就失忆）。
2. **这轮用了工具 → 两次 LLM 调用**：一次产出答案 + 写入指令，一次拿写入结果后产出收尾语；**两次都各自重发整段历史**，近乎翻倍。
3. **harness 系统提示 + 工具定义巨大**：几十个工具的 JSON schema 每次调用都在输入里（~1.5 万 token），一个没用也付费。
4. **输出最贵**：长答案 + 往文档追加几十行 ≈ 几千输出 token，opus 输出价 179.45/M（输入的 5 倍）。
5. **模型是 opus**：换 Mimo（输入 3 / 输出 6）成本差一个数量级。

**粗算**：单轮输入 ≈ 历史几万 + 系统/工具 ~1.5万 ≈ 4~5万 token，× 2 次调用 + 输出 ~2~3千（最贵）+ 全程 opus ≈ 2~3 元量级，与 2.88 对得上。

**最大讽刺**：我们正在用最贵的 opus 进行一场"如何省 token"的纯知识讨论——恰恰违反杠杆三（知识问答该用便宜模型）。

**对应抓手（活生生应验）**：

- 杠杆三：知识型讨论换便宜模型 → 省一个数量级。
- 杠杆二：聊太长 → compact / 开新会话砍历史（代价：丢缓存，权衡）。
- 隐形大头：重型 agent 环境问通识题 = 每轮给用不上的工具定义付费 → 纯聊天挪到轻环境。

### 选型方法论 · 高置信研究报告该用什么模型

> 修正一个常见误区：选型不是单看"知识在不在权重里"，而是 **模型档次 = f(难度 × 后果)**。

- 随手问、答错无所谓 → 便宜模型。
- **团队共享 / SKILL 化 / 要求高置信的研究报告** → 一处讲错会扩散，**错误成本 >> token 成本** → 前沿模型值这个溢价。"发散全面 + 深刻"也正是前沿模型强项（held 住更多角度、综合更深、微妙处少翻车）。

**置信度分层（低幻觉 + 深推理 ≈ 前沿档）：**

| 档位                   | 代表                             | 适合                 |
| ---------------------- | -------------------------------- | -------------------- |
| 前沿（最高置信、最深） | opus-4.8 / gpt-5.5 / 顶级 Gemini | 研究报告发散阶段     |
| 强力高性价比           | DeepSeek 顶配推理 / GLM 高配     | 要质量又想省钱的次选 |
| 便宜                   | Mimo / 轻量款                    | 归纳收口、排版       |

**⚠️ 高置信不能只靠选对模型——再强的模型也会在冷门事实上幻觉。置信度来自验证：**

1. 开 reasoning / thinking 模式（更深、更少低级失误）。
2. 要求它标注不确定性，而非一律自信输出。
3. 关键事实（数字/版本/定价）交叉验证：web 检索 + 第二模型复核。
4. 要来源——能附依据的结论才算高置信。

> 分工：**模型负责"深度+广度"，验证负责"置信度"，缺一不可。** 配方：发散用前沿+reasoning，硬事实联网核对，归纳收口切便宜模型。

### 缓存机制 · 核心思想与主流范式

> 省钱第一杠杆。多数人只知"命中打一折"，不知它省的是什么、为何一处改动就击穿、为何写入反而更贵。

**核心思想：复用确定性计算**。transformer 生成每个 token 要对前面所有 token 做注意力，每 token 每层算出一对 **Key/Value 向量**；对固定前缀这些 K/V 完全确定（纯函数）。既然确定，重算就是浪费——**把已算的 K/V 存起来复用 = KV Cache**。

**两层缓存（分清）：**

- **第1层 KV Cache（单次请求内）**：decode 时每步只算新增 token 的 K/V，不重算前面所有（否则 O(N²)）。是 decode 的前提，永远开着、不单独计费。
- **第2层 Prompt/上下文缓存（跨请求）⭐ 计费表"缓存命中 1/10"说的是它**：多次请求共享同一前缀（系统提示+工具定义+早期对话）时，服务端存下该前缀 KV，下次跳过对它的 prefill → 按 ~1/10 收费（省的就是 prefill）。

**两条关键规则：**

1. **前缀从第0个 token 连续匹配**（最长公共前缀）：早期一个 token 不同，后面全部击穿。→ 稳定内容放最前、逐字不变；易变内容（当前问题/动态数据）放最后。
2. **写贵读便宜**：写入缓存比普通输入更贵（Anthropic 5min 1.25x / 1h 2x），命中读 ~0.1x。写了没被复用几次 = 白花钱。只缓存稳定、大块、反复用的前缀。

**两种主流范式：**

|        | 隐式/自动（OpenAI、DeepSeek、Gemini） | 显式/手动（Anthropic）                   |
| ------ | ------------------------------------- | ---------------------------------------- |
| 用法   | 服务端自动检测重复前缀，零改动        | `cache_control` 手动标 + 选 TTL(5min/1h) |
| 优缺点 | 省心 / 控制弱、TTL 固定               | 精确可控 / 要自管、写入有溢价            |

底层基建：vLLM / SGLang 的 **Prefix Caching / RadixAttention** 在服务层自动共享请求间前缀 KV。

**TTL（连接深水区）**：缓存会过期（5min 默认 / 1h 延长）。停顿超 TTL → 缓存蒸发，下次重新全价 prefill。所以连续对话比断续省。

**浓缩**：①复用确定性计算 ②省的是 prefill ③前缀从头连续匹配，一处改动击穿全部 ④写贵读便宜，复用够多才回本。实践抓手（杠杆一）：稳定内容前置且逐字不变、易变后置、别频繁改系统提示、别动不动开新会话、连续用比断续用省。

### 给后端工程师 · 这套数据模型 & 缓存到底复用了什么

> 困惑:"我只是一个接一个地问问题,有什么可复用的?" 卡点在一个没意识到的事实上。

**关键事实:你以为发了"一个问题",其实发了"整本聊天记录"。** 因为模型无状态(不记得任何前轮),客户端每问一个新问题,实际拼给模型的请求体是:

```json
[
  {"role":"system","content":"系统提示+几十个工具定义"}, // 固定不变
  {"role":"user","content":"问题1"}, {"role":"assistant","content":"回答1"}, // 历史原样重发
  ... ,
  {"role":"user","content":"问题5"}  // 只有这行是新的
]
```

你只敲了几个字,发出去的是前面整本历史+系统提示+工具定义,而那一大坨前缀这次和上次**逐字一样**。**可复用的就是这段"每轮原样重发、逐字不变的前缀"。**

**复用的具体是什么数据?** 模型读输入时把每个 token 过几十层网络,算出中间数据(K/V 向量 = "给每个词预计算的内部索引数据")。这步很贵(=prefill),且只取决于文字本身(纯函数)。同样前缀→同样中间数据→第二次没必要重算。

**= 以"输入前缀"为 key 的计算结果缓存**(和 Redis 缓存昂贵函数返回值同一思想):

```
key=输入前缀(逐字节)  value=该前缀算出的KV  被缓存的昂贵函数=prefill
第1轮: 算"系统+问题1"的KV,存缓存
第2轮: 前缀逐字相同→命中,取出旧KV跳过重算,只算新尾巴(问题2)
```

| 后端概念            | 对应                                                |
| ------------------- | --------------------------------------------------- |
| Redis / memoization | 同入参(前缀)→返缓存结果(KV),不重跑昂贵函数(prefill) |
| HTTP ETag / 304     | "这段上次要过、没变,不重新生成"                     |
| 增量编译 / 构建缓存 | 没改的源文件复用编译产物,只编译新文件               |

**为何改一字缓存全失效**:每个词的 KV 依赖它前面所有词(注意力),前缀第10词改了→第11词起全变→只能用到第9词。像**流式 hash / 视频关键帧**:从头连续匹配,一处变后面全废,不是逐行 diff。

**整套数据模型对齐后端心智:**

| LLM 机制              | 后端类比                                     |
| --------------------- | -------------------------------------------- |
| 一次调用无状态        | 无状态 HTTP 接口,不存会话                    |
| messages 每轮全量重发 | 客户端每次带全量上下文("对话"是拼出来的假象) |
| token                 | 字节/字符切片                                |
| KV 向量               | 预计算的中间索引数据(贵)                     |
| prefill               | 昂贵的批处理计算(文字→KV)                    |
| prompt caching        | 前缀为 key 的计算缓存,前缀不变就复用         |
| 前缀连续匹配          | 流式 hash / 增量编译,一处改动后面全失效      |

> 收束:可复用的不是"答案",而是"把每轮重发的那段不变前缀转成模型内部中间数据(KV)"这步昂贵计算。

### 打地基 · token 是什么、"复用"到底复用什么(给零基础后端)

> 纠正核心误解:**被复用的不是 token(文字)本身,是 token 被加工后产生的"重计算结果"(K/V)。** 文字是缓存的 key,内部数据才是 value。

**① token 是什么**:一小段文字(约一个词或词片),模型先把它查表换成编号——`"我在银行工作"→[我][在][银行][工作]→[101][205][8888][3021]`。文字→编号很便宜(查表),不是贵处。

**② 真正的贵活**:每个 token 送进几十层网络,**不是孤立处理——它回头看前面所有 token,吸收上下文**,产出一组带语境的中间数据(K/V 向量)。如处理`[银行]`时回看`[我][在]`,于是知道是【机构】非【河岸】。这步=prefill,输入成本大头。

**③ 关键:这份数据只看它前面,不看后面**(因果注意力,从左往右只能回头看):

```
第1次: [我][在][银行][工作] → [银行]看[我][在][银行] → 数据 X
第2次: [我][在][银行][上班] → [银行]看的还是[我][在][银行] → 还是 X!
```

前缀逐字相同→K/V 逐字相同(纯函数无随机),后面换词不影响前面。

**④ 所以复用复用什么**:同前缀→同中间数据,第二次何必重跑几十层网络?存上次的 K/V,直接取用,只对新增/变化的 token 重算。**复用的是"把不变文字过网络加工成 K/V"这步昂贵计算的结果,不是文字本身。**

**后端锁死:**

```
token(文字/编号)    = 函数入参(传进来很便宜)
prefill(过几十层网络) = 一个超贵的函数(真正烧钱)
K/V 向量            = 该贵函数的返回值(值得缓存)
prompt caching      = memoize(贵函数, key = token 前缀)
```

每轮重发整本历史 = 每轮拿同样入参调那个超贵函数;缓存 = 入参(前缀)没变就返回上次结果不重跑;前缀改一个 token = 入参变了 = 从那往后 miss 重算。

### 确定性 vs 随机 · temperature/思考强度的意义在哪(解表面矛盾)

> 矛盾:既然"相同前缀→相同 K/V(纯函数)",那 temperature 的随机从哪来?解法:把"算分布"和"选 token"两步分开。

**一次"预测下一个 token"= 两步:**

```
①前向传播 → 算出【概率分布】      ← 完全确定(纯函数,可缓存的就是这部分)
②从分布采样 → 选出【具体 token】   ← 这里才有随机
```

例:输入`"中国的首都是"`→①算出分布`北京92%|上海3%|...`(每次都一样)→②temp=0 永远选"北京"(确定);temp 高 大概率北京但偶尔蹦别的(随机)。

所以:**模型算出的"分布"给定相同序列完全一样 ✅(这也是能缓存的部分);最终吐出的 token 不一定一样(采样)。**

**参数各作用在哪:**

| 参数                      | 作用步骤           | 干什么                                                       |
| ------------------------- | ------------------ | ------------------------------------------------------------ |
| temperature/top_p/top_k   | 第②步(怎么选)      | 把确定的分布压尖/压平,控采样随机度。temp=0=永选最高(退化为确定);temp 高=放平分布、容冷门 token→输出多样有创意 |
| 思考强度/reasoning_effort | 另一维度(生成几步) | 不改单步确定性,改"吐多少思考 token、思考链多长"。每步仍是①算分布→②采样 |

> 重点:temperature 不改变"算出什么分布",只改"从分布怎么挑";思考强度不改单步确定性,只改"走几步"。

**为何与缓存不冲突**(两者不同阶段):

```
缓存复用 = 输入前缀过网络的 K/V   ← 输入侧,文字已固定,完全确定 ✅
采样随机 = 输出每步从分布挑 token ← 输出侧
```

缓存只复用已固定输入的加工结果(prefill),不碰"下个 token 选谁"。输入侧确定可缓存,输出侧采样有随机,各管各的。

**诚实脚注**:即便 temp=0,生产环境也未必逐字可复现——浮点误差、GPU 并行批处理、MoE 路由受同批次其他请求影响,带来微小非确定性。是工程噪声,非设计本意;数学上"相同输入→相同分布"成立。

### 思考强度高低的差别是什么

**本质**:控制"模型给最终答案前生成多少思考 token"(思考链多长多深)。**同一个模型,不换模型,只是告诉它多琢磨还是少琢磨。**

**高 vs 低 具体干什么**(同题,内部思考过程不同):

```
题:数组找两数之和=target
低强度: 读题 → 直接双重循环暴力 → 答
高强度: 读题 → 拆解 → 暴力O(n²) → "能更快?" → 哈希O(n)
       → 查重复元素? → 查空数组/无解? → 验算例子 → 最优解+边界
```

高强度多吐一大段"拆解/试错/回退/自检/验算"中间 token;低强度跳过、快速给答案。

**机制(接地基)**:推理深度 ≈ 花掉的 token 数,模型没法在一个 token 里想更深,只能多吐 token。思考强度 = 给它设思考 token 的预算:高=预算大(几千~几万),容许充分展开搜索+自检;低=点到为止。每个思考 token 仍走①算分布②采样,只是步数多了。

**差别体现在三处:**

| 维度       | 高强度                              | 低强度           |
| ---------- | ----------------------------------- | ---------------- |
| 难题准确率 | ↑↑ 显著更高(能试错/回退/自检)       | 易一步错、漏边界 |
| 成本+延迟  | ↑↑ 思考 token 多=输出更贵更慢       | 便宜、快         |
| 简单题     | ⚠️ 边际收益递减,甚至过度思考反而出错 | 刚好             |

> 类比:10 秒 vs 10 分钟作答;张口就答 vs 打草稿+验算;搜索算法浅搜 vs 深搜。

**实践**:难推理(复杂 bug/算法设计/数学证明/棘手架构)→开高,准确率换 token 划算;简单/查询/机械任务→关低,开高是交"过度思考税"(多花钱、更慢、还可能想拧)。本质=拿 token(钱)+延迟换准确率,花在准确率重要处。

**与 temperature 区分**:temperature 管单步"从分布怎么挑"(随机度);思考强度管"总共走多少步"(链长)。一个是每步多随机,一个是总共想多久,两根轴。

### 输入可控、输出不可控 · 为什么同一道题价格差几十倍

> 案例:同样输入"数组两数之和",低强度 vs 高强度,输入 token 完全一样,价格却差几十倍。

**输出 token 确定吗?不确定,事先无法精确知道。**

- 输入:你打的字,token 数完全确定、你说了算。
- 输出:涌现的——模型一路生成到吐停止符为止,实际吐多少思考+答案 token 取决于这次走的路径;高强度只给"思考预算",一次很快想通、另一次反复试错回退,长度差很多;加采样随机,同输入每次长度都可能不同。只能用 `max_tokens` 封上限,封不住"具体吐多少"。

> 输入是下单时确定的;输出是开盲盒,只能设盒子上限,开出多少事先不知。

**为什么价差这么多?价差几乎全在输出(双重放大):**

1. 数量:低强度 ~80 token 就给答案;高强度要拆解+试错+自检+验算,2000~3000 token,**几十倍差距**。
2. 单价:输出本就最贵(输入的 3~6 倍),多出的思考 token 全是最贵那种。

用 opus 价(输入 35.89 / 输出 179.45 每 M)算账:

```
输入"数组两数之和 高强度"≈20t → 20×35.89/M≈0.0007元 (两路一样,可忽略)
低强度输出≈80t  → 80×179.45/M ≈ 0.014 元
高强度输出≈2800t → 2800×179.45/M ≈ 0.50 元
→ 高 vs 低 差约 35 倍,几乎 100% 来自输出 token 数量差,输入纹丝没动
```

**收束**:输出体量型任务,成本几乎完全由输出 token 数量决定,而该数量事先不可精确预估,只能 max_tokens 封顶。高强度 = 主动多烧几十倍最贵的输出 token 换难题准确率。两数之和这种简单题开高 = 典型过度思考税:多花 35 倍钱,准确率却不涨。

### 形式化 · 生成循环 → 完整计费模型

> 一个干净的数学骨架,把前面所有概念焊在一起。

**单次调用(自回归骨架):**

```
s1 → s2=s1+1tok → s3=s2+1tok → ... → s_final = s1 + ans
输入 = s1 ; 输出 ans = [思考 token] + [答案 token]   (思考链也全算输出)
单次费用 = 输入价 × len(s1) + 输出价 × len(ans)
```

- `s1→第1个token` = prefill(并行读完 s1,输入大头在此);其余每步 = decode(逐 token 吐,输出最贵)。
- 每步靠 **KV 缓存只算新增 token**,不重跑整条序列;循环终止 = 采样到【停止符】。

**这只描述单次。完整计费再加两层:**

**① 缓存分层(把 s1 拆开):**

```
s1 = [命中前缀] + [未命中部分]
输入费 = 命中价(~0.1x) × len(命中前缀) + 全价 × len(未命中部分)  (+ 首次写入溢价)
```

**② 多轮累积(任务 = N 次调用求和):**

```
任务 = 调用1 + 调用2 + ... + 调用N        (N = 工具轮数 + 1)
第 i+1 次输入 ≈ 第 i 次输入 + 上轮 ans + 工具返回   ← 滚雪球
任务总费用 = Σ_{i=1..N} [ 输入费(s1_i, 含缓存分层) + 输出价 × len(ans_i) ]
```

→ 这就是**输入平方膨胀**的来源:历史每轮被重新计入。

> 结论:`s1 → ans` 是完整计费的**单次原子单元**;套上 ①缓存分层 + ②多轮求和 = **完整计费模型**。一句话覆盖全篇:**费用 = Σ每次调用( 输入按缓存分层计价 + 输出含思考链按最贵计价 )。**

### 参数怎么作用 · 选 token / 控长度 / 控思维链的机制

> 分清两类控制:**选哪个 token** vs **生成多长**;思维链属后者特例。

**流水线(参数在哪一步动手):**

```
前向传播 → logits(每个词表 token 一个原始分数)
  ├─[temperature 缩放][top_k/top_p 截断][penalty/bias 调整]  ← ①选择类在这动手
  → softmax → 概率分布 → 采样 → 选出 1 个 token
  ├─ 是 EOS / "结束思考"标记? → 停 / 切阶段
  └─ 否则 append → 回前向传播 (受 max_tokens 硬上限约束)   ← ②长度类在这动手
```

**① 选择类(不改原始 logits,只在采样前给分布变形):**

| 参数                       | 机制                                                         |
| -------------------------- | ------------------------------------------------------------ |
| temperature T              | logit/T 再 softmax。T<1 变尖(确定)、T>1 变平(随机)、T→0 退化永选最高 |
| top_k                      | 只留最高 k 个,其余砍掉再采样(截断尾部)                       |
| top_p                      | 按概率累加到 p,在这"核"里采样(动态截断)                      |
| frequency/presence penalty | 对已出现 token 的 logits 扣分,压低重复                       |
| logit_bias                 | 直接给指定 token 加/减分,强行抬高/压低                       |

**② 长度类:**

| 机制           | 谁来停                                                       |
| -------------- | ------------------------------------------------------------ |
| EOS 结束符     | 模型自己——采样到 EOS 自然终止;何时出现是预测的(训练学过"答案后跟 EOS") |
| max_tokens     | 外部硬截断,到量强砍                                          |
| stop_sequences | 外部,命中字符串即停                                          |

实际长度 = min(模型吐 EOS 时机, max_tokens, stop 命中)。

**③ ⭐ 思维链怎么控制(reasoning effort,非简单采样参数):**

- 机制一·训练塑造:推理模型经 RL 学会"先在思考区(如 `<think>…</think>`)推理,自认想清才吐'结束思考'标记,再切答案区"。"想多久"是被训练出来的。
- 机制二·effort 旋钮接入:① 思考 token 预算——high=大预算(容许一直想)、low=小预算(近预算时催收尾,提前吐结束标记);② 训练条件信号——模型见过 high→长细/low→简短 的样本,effort 作条件让它自发想长/短。

> 本质:思维链长度 = "结束思考"标记多晚被采样出来。high 推迟(继续想)、low 提前(快收尾);思考区每个 token 仍走前向→算分布→采样,只是步数多了。

**串起来**:选哪个 token ← temperature/top_k/top_p/penalty/bias(对分布变形);何时停/多长 ← EOS(自然)+max_tokens/stop(硬停);思维链多长 ← reasoning effort 通过【思考预算】+【训练条件】调"结束思考标记"出现的早晚。

### 打地基 · 前向传播是什么 & 它和思维链的关系

**前向传播 = 把当前 token 序列喂进神经网络,从头到尾算一遍,输出"下一个 token 的概率分布"的过程。**

```
输入:当前 token 序列(的向量)
 → 第1层(注意力+前馈,矩阵运算) → 第2层 → ... → 第几十层   (数据单向流,故"前向")
 → 输出 logits → softmax → 下一个 token 的概率分布
```

关键:**一次前向传播 = 产出一个 token 的分布。** 生成 N 个 token = 跑 N 次前向传播 = 生成循环里每个 `+1token` 那步。("前向"相对训练时的"反向传播";推理时只有前向。)

**控制思维链影响的是前向传播吗?——不影响单次前向传播本身,影响的是它跑几次。**

|                                           | 思维链控制碰它吗                                        |
| ----------------------------------------- | ------------------------------------------------------- |
| 单次前向传播内部(权重/层数/矩阵运算/算法) | ❌ 完全不碰,算一个 token 分布的过程一模一样              |
| 前向传播的次数/轨迹(跑多少步)             | ✅ 改的就是这个。high→多吐思考 token→多跑很多次;low→少跑 |

> 类比:前向传播是"算一步"的固定动作;思维链控制是"决定走多少步"。不改你怎么迈一步,改你总共迈几步。

机制:思维链控制通过"让'结束思考'标记多晚被采样出来"决定 decode 循环跑多少轮 = 多少次前向传播。high 推迟、low 提前。

**诚实的微妙点**:effort 的条件信号(预算/档位)会作为输入的一部分进入序列,所以会轻微影响每次前向传播算出的分布(往"继续想"或"该收尾"推一点),从而决定结束标记何时出现。但这是通过"改输入"间接影响输出,不是改前向传播的内部机制——权重、层数、算法始终没变。

> 一句话:前向传播 = 单步计算(算一个 token 分布);思维链控制不动这一步内部,只决定这一步重复跑多少次(并通过输入信号轻推每步分布,让"结束思考"标记早出现还是晚出现)。

### 思维链怎么终止 · "结束思考"是个被采样的普通 token

> 关键:没有独立"裁判"决定停不停——"结束思考"标记(如 `</think>`)本身是个普通 token,和别的词在同一分布里竞争采样,被采样中=思考结束。终止/多吐/选择是同一机制的三个面。

**1. 怎么影响终止**:词表里有 `</think>` token,每次前向传播的分布都含它、给它一个概率:

```
推理到一半某步: "因此"18% | "检查"12% | ... | </think> 0.3%  ← 停止概率低→继续吐
想清楚后某步:   </think> 71% | "所以答案"9% | ...           ← 停止概率飙升→被采样→结束
```

终止条件 = `</think>` 被采样中那一刻。其概率由模型按"想到哪了"动态给出:没想清压低,想清抬高。

**2. 怎么"识别"要多吐**:不是刻意识别,是 RL 训练塑造的本能——"多想且答对"的轨迹被强化,模型学会"问题没解决前持续压低 `</think>` 概率",直到内部状态显示解决才抬高。无显式计数器、无元认知,全编码在每步给 `</think>` 的概率里。

**3. 选择机制是什么**:就是普通的 softmax 采样,无特殊机制。`</think>` 和普通词挤在同一分布里一起被采样——采样到普通词→继续,采样到 `</think>`→结束。"停止思考"本质="停止思考这个 token 碰巧被选中"。

**effort 旋钮怎么偏置时机**:

| 方式     | 机制                                                         |
| -------- | ------------------------------------------------------------ |
| 思考预算 | 给思考 token 设上限,接近时硬注入 `</think>` 或抬高其 logit。high=大预算(催得晚)/low=小预算(催得早) |
| 条件信号 | effort 档位作输入进序列,模型训练见过 high→长/low→短,条件在 high 上时学到的策略自发把 `</think>` 压低更久 |

**思考 token 怎么分隔**:边界标记 `<think>…推理…</think>…答案…`,harness 据此隐藏思考区只显答案,但两段都按输出计费;思考 token 数 = `</think>` 之前那段长度。

> 收束:终止/多吐/选择是同一件事——"结束思考"是个普通 token,每步和别的词在同一分布竞争采样;模型被训练成"没想清就压低它";effort 通过预算或条件信号调它多早赢得采样。没有裁判,只有概率。

### 打地基 · RL(强化学习)训练是什么

**RL = Reinforcement Learning,强化学习。一句话:通过"试错 + 奖励"学习,没有标准答案逐条教,只有"结果好不好"的反馈信号。**

**放进 LLM 三段式训练看最清楚(RL 是最后一关):**

| 阶段             | 怎么学                               | 类比                 | 学到什么                         |
| ---------------- | ------------------------------------ | -------------------- | -------------------------------- |
| ① 预训练(自监督) | 读海量文本,预测下一个词              | 博览群书             | 语言/通识/世界知识(权重里的记忆) |
| ② 监督微调 SFT   | 看"问题→好答案"范例,模仿             | 照标准答案抄         | 按指令好好回答                   |
| ③ 强化学习 RL    | 自己生成→拿奖励评判→强化好的抑制坏的 | 自己做题→打分→调策略 | 怎么做更容易拿高分               |

关键区别:SFT 是"抄标准答案",RL 是"没有标准答案,自己试,只按结果好坏调整"。

**RL 在 LLM 上两大用途:**

- **(a) RLHF(人类反馈)**:生成多个回答,人类/奖励模型排序哪个更好,用偏好强化。模型变"有用、懂事、不乱说"靠这步。

- **(b) 推理强化学习**(前面聊的):对有标准答案的任务(数学/代码),生成推理链+答案,**答案对→奖励,错→惩罚**。奖励只看最终答案对不对,不教它怎么想;但模型千万次试错中自己摸出"更长更仔细的推理→答对率更高",于是**自发学会延长思考链、不过早停止**。

  > 这就是"没想清就把 `</think>` 概率压低"的来源——没人规定"这题想 500 token",是它自己试出"想够再停答对率最高",固化进权重。

**后端类比:**

- 监督学习 = 给你"输入→正确输出"的单元测试,照着实现。
- 强化学习 = 没单元测试,只有"跑完系统看 KPI 涨没涨"的反馈,你不断改、涨了留跌了回退,自己摸出策略,没人逐行教。
- 或:打游戏不看攻略——只知"通关+分/死了-分",玩多了摸出最优打法。RL 就是让模型这样"玩"任务。

> 收束:RL = 模型自己生成→奖励(答案对错/人类偏好)评判→强化好的抑制坏的,反复试错自发学出策略。"该想多久才停""怎么答更讨喜"都是 RL 练出来的,非逐条规定。

### 打地基 · context(上下文)是什么

**context = 模型在一次前向传播时能看到的全部 token 序列。就是前面一直叫的 `s1`,换了个名字。**

**里面装什么(=s1 结构):**

```
context = [系统提示] + [工具定义] + [历史对话:之前所有轮 user+assistant]
          + [当前你的问题] + [生成中:已吐出的 token]
```

模型对这次请求的全部认知只来自 context 里的 token,context 之外一无所知。

**关键性质:context 是模型唯一的工作记忆**。回扣无状态——"对话有记忆"是假象,记忆=每轮把历史重新塞回 context 重发;删掉历史某轮模型就真忘了。模型在 context 内所有 token 间做注意力:context 之内=全世界,之外=不存在。

**context window:容量上限**

| 词                              | 含义                              |
| ------------------------------- | --------------------------------- |
| context                         | 这次实际塞进去的 token 序列(内容) |
| context window(上下文窗口/长度) | 能塞多少的容量上限,如 200k token  |

装满了:超过 window 装不下→harness 截断或压缩历史(丢最早轮次/做摘要)→超长对话后面会"忘记"开头(开头 token 被挤出 context)。

**后端类比:**

```
context        = 一次无状态 HTTP 请求的完整 request body(模型不存 session,想让它知道的全打包进 body)
context window = request body 的大小上限
"对话记忆"      = 客户端每轮把聊天历史塞进 body 重发
```

**和成本的关系**:context 越长→输入 token 越多→prefill 越重,越贵越慢;历史每轮累积进 context = "输入平方膨胀"的容器。省钱核心动作之一=管好 context(精简历史、该开新会话开新会话、用缓存接住稳定前缀)。

> 一句话:context = 模型这次"眼睛能看到的全部 token"(=s1);context window = 视野上限;模型无状态,全部记忆就是你塞进 context 的东西,塞不下的被挤出就"忘"了。

### 为什么中文字少却可能更耗 token · 分词器的账

> 反直觉点:把"字符数"和"token 数"混了。字符数 ≠ token 数。

**三个不同的量:**

| 量           | 说明                  | "我爱你" vs "I love you"                |
| ------------ | --------------------- | --------------------------------------- |
| 字符数       | 写出来占几个字        | 中文 3 / 英文 10 → 中文信息密度高、字少 |
| token 数     | 模型实际处理&计费单位 | 花钱的单位,和字符数不是一回事           |
| 谁决定怎么切 | 分词器(tokenizer)     | 对中英文不公平                          |

"中文字少"是字符数的事;"更耗 token"是 token 数的事。脱钩原因全在分词器。

**为什么中文吃亏(机制)**:分词器用 BPE,词表从训练语料统计高频片段,经典分词器语料以英文为主:

- 英文:常见词/词根("the""ing""tion")整体进词表 → 1 个常见英文词 = 1 token,极高效。
- 中文:覆盖不足,很多汉字没整体进词表 → 退回 UTF-8 字节编码,一个汉字 = 2~3 字节 = 2~3 token。

```
"the" → 1 token
"我"  → 1~3 token(看分词器,老的/生僻字拆成多字节 token)
```

虽中文字少,但每字摊到的 token 多 → 同样意思中文 token 反而可能更多。**是分词器不公平,不是中文语言的锅。**

**⭐ 诚实补充:主要是"老分词器"问题,新模型已大幅改善**:

- GPT-3 时代:中文极吃亏(一字常 2~3 token)。
- 现代分词器(GPT-4o o200k 等):常见汉字基本 1 token,有些词还合并 → 中文效率追上甚至反超。
- 国产模型(DeepSeek/GLM/Qwen):分词器专门针对中文优化,中文 token 效率通常优于英文中心分词器。

**实践抓手**:中文任务优先用中文优化的模型(常更划算,同义中文 token 更少);同一英文中心老模型处理中文可能比同义英文更贵;想确认就拿中文丢进该模型 token 计数器数一下。

> 一句话:中文字少(信息密度高)≠ 中文 token 少。token 怎么切由分词器说了算,英文中心老分词器把中文拆碎(一字多 token)才显"更耗";现代/中文优化分词器已抹平甚至反超——这是分词器的账,不是语言的账。

### 什么时候用中文 / 什么时候用英文(附论文)

**学界核心发现(均有论文):**

1. 模型内部其实"用英文思考"——*Do Llamas Work in English?*(Wendler 等, ACL 2024, arXiv:2402.10588):多语言模型把输入映射到一个"概念空间",而该空间更靠近英文,英文是事实上的内部中转语。佐证:*Do Multilingual LLMs Think In English?*(arXiv:2502.15603)。
2. 强迫模型用中文做思维链会掉准确率——*Controlling Thinking Language Comes at the Cost of Accuracy*(arXiv:2505.22888, XReasoning):即便最强推理模型,被要求用非英文思考时要么退回英文、要么推理变碎,准确率显著下降。
3. 复杂推理仍英文中心——*Long Chain-of-Thought Reasoning Across Languages*(arXiv:2508.14828);英文 >> 低资源语言(arXiv:2511.23231)。
4. 中文/多语言 prompt 在多样性/创意上反而占优——*Multilingual Prompting for Improving LLM Generation Diversity*(arXiv:2505.15229):多语言 prompting 比高温采样、persona prompting 更能提升输出多样性。
5. prompt 技巧可拉平多语言能力——*Cross-Lingual-Thought Prompting (XLT)*(arXiv:2305.07004)。

**决策表:**

| 场景                                          | 推荐               | 依据                                   |
| --------------------------------------------- | ------------------ | -------------------------------------- |
| 高难推理/数学/算法/复杂代码逻辑               | 英文               | 内部表征对齐英文+语料偏英文(论文1/2/3) |
| 追求最高准确率、压榨前沿能力                  | 英文               | 同上                                   |
| 大量英文术语/API/库文档                       | 英文               | 避免翻译损耗、术语错配                 |
| 要在多模型上稳定复现                          | 英文               | 英文语料最足,跨模型最稳                |
| 中文领域知识/文化语境/文案营销/本地化         | 中文               | 英文会丢语境语感                       |
| 产出给中文读者/团队(解释、思维链需可读)       | 中文(至少答案中文) | 可读性、可监督性                       |
| 用国产中文优化模型(DeepSeek/GLM/Qwen)处理中文 | 中文               | 对中文对齐好,可能反超英文,token 更省   |
| 需要创意发散/多样性                           | 中文或多语言       | 论文4                                  |
| 简单任务                                      | 怎么顺手怎么来     | 中英差异不显著                         |

**实用偏方:思考用英文,回答用中文**。把思维链语言与答案语言拆开——让模型用英文思考(本就擅长、最准),用中文给最终答案。提示:`Think step by step in English, then give your final answer in Chinese.` ⚠️ 但别强行规定"用中文思考"(论文2:反而掉准确率),让它自然用英文想、中文答最优。

**诚实置信度标注**:

- "英文更准"主要在难推理任务明显,一般/简单任务差异很小,别教条。
- 差距在快速收窄(国产中文模型+最新前沿模型多语言对齐持续进步),结论是趋势性的非铁律。
- 最稳做法:对你的具体任务和模型,拿中英两版各跑几次实测对比,用数据说话。

### 全景收口 · 从训练(含 RL)到推理选 token 的完整过程

> 关键澄清:RL 不在推理时发生,它在训练期把本事烙进权重;推理期只是反复调用烙好的权重选 token。两条时间线。

```
【训练期 · 造模型,离线一次性】
  预训练 ─→ SFT ─→ RL ─→ 固定的【权重 W】
  (学语言/知识)(学按指令答)(学怎么选更准、想多久才停)
                              │ 烙进 W,推理时不再变
                              ▼
【推理期 · 每生成 1 个 token 走一遍】
  context → 前向传播(用 W) → logits → 变形 → 采样 → 1 token → 循环
```

桥梁:训练期决定"前向传播这个函数长什么样(W)";推理期反复调用它挑 token。**logits 形状由 W 决定,从 logits 怎么挑由采样参数决定。**

**Phase A 训练期(决定 W):**

| 步      | 做什么                     | 烙进 W 的                             |
| ------- | -------------------------- | ------------------------------------- |
| ①预训练 | 海量文本预测下一词         | 语言/知识/算分布的基础能力            |
| ②SFT    | "问题→好答案"模仿          | 按指令回答的倾向                      |
| ③RL     | 自己生成→奖励评判→强化好的 | "怎么选更易答对""`</think>` 多晚出现" |

**Phase B 推理期(一个 token 的生命周期),给定序列 `s`:**

```
① tokenize    文字→token id(已是序列则接续)
② embedding   token id → 向量
③ 前向传播    向量过几十层(注意力+前馈,用 W)→ logits(每 token 一个分数)  ← W 决定形状
── 选择阶段(对 logits 动手) ──
④a temperature 缩放  ④b top_k/top_p 截断  ④c penalty/logit_bias 调整
④d softmax→概率分布  ④e 采样→选 1 个 token                              ← 采样参数决定怎么挑
── 
⑤ 是 EOS/</think> 吗? 是→停/切阶段; 否→append 回③(KV 缓存只算新 token)
⑥ 循环至停止 → 整段 ans(思考 token+答案 token)完成
```

**零件归位:**

| 概念              | 在哪一步                                                     |
| ----------------- | ------------------------------------------------------------ |
| token/KV 向量     | ①②③ 文字→id→向量→过网络产 K/V                                |
| prefill vs decode | 首次过完整 `s`=prefill;之后每轮③算 1 新 token=decode         |
| KV 缓存           | ⑤→③ 回圈时复用前面 token 的 K/V                              |
| 概率分布(确定)    | ③④d 算出的分布,给定 `s` 完全确定(纯函数)                     |
| 采样随机          | ④e,温度>0 时引入                                             |
| 思考强度/思维链   | ⑤ 里 `</think>` 多晚被采样,由 RL 烙进 W + effort 信号调      |
| 计费              | 输入按 len(s) 一次(prefill);输出按整段 ans(每轮 decode,最贵) |

> 收束:训练期(预训练→SFT→RL)把"怎么选 token、想多久"烙进 W;推理期每个 token 都走"前向传播(用 W)→ logits → 采样参数变形 → 采样选中 → 判停或回圈"。RL 决定分布形状(倾向),采样参数决定从分布怎么挑,两者合起来才是你看到的 token。

### 开源工具调研 · Headroom(上下文压缩层 / 省 token 抓手)

> 定位纠正:Headroom 不是模型,是 AI agent 的「上下文压缩层」中间件——但它是 token 经济学"省钱抓手"的落地版。

**是什么**:夹在应用和 LLM provider 之间,把 agent 要读的一切(工具输出/日志/RAG chunk/文件/对话历史)在进 context 前压掉,号称 60–95% 更少 token、答案不变。作者 Netflix 工程师 Tejas Chopra;Apache-2.0;27.1k stars;v0.25.0(2026-06);Python+Rust。被报道累计省 ~$70 万、释放 2000 亿 token。

**机制(和本笔记概念对上)**:哲学="发给 LLM 的 80% 是噪音",用本地小压缩器在进 context 前砍噪音。

| 组件                   | 干什么                                                | 对应概念               |
| ---------------------- | ----------------------------------------------------- | ---------------------- |
| SmartCrusher(JSON)     | 剥 null/样板/冗余 key,GitHub API 响应 3000→几百 token | 压垃圾工具输出         |
| CodeCompressor(AST)    | 按语法结构压 Python/JS/Go/Rust/Java/C++ 代码          | 压代码上下文           |
| Kompress-base(HF 模型) | 小模型压散文/RAG,语义打分丢无关句                     | 压 RAG/prose           |
| CacheAligner           | 稳定前缀让 provider KV 缓存真正命中                   | ⭐ 对应 prompt caching  |
| 对话摘要器             | 老对话压成 bullet,最近几轮保留原文                    | ⭐ 对应"发整本聊天记录" |
| CCR 可逆压缩           | 原文存本地,模型需要时调 headroom_retrieve 取回        | 有损但可恢复           |

全本地运行(数据不出本机),框架无关。四种接入:库 compress(messages) / 代理 headroom proxy(零改码) / 包裹 agent headroom wrap claude|cursor|codex / MCP server。

**关键数据(官方 benchmark)**:

- 省 token:代码搜索 17765→1408(92%)、SRE 排障 65694→5118(92%)、issue 分诊(73%)、代码库探索(47%)。
- 准确率:GSM8K ±0、TruthfulQA +0.03、SQuAD v2 压 19% 仍 97%、BFCL 工具调用压 32% 仍 97%。
- 经济账例:客服 agent 1000 次/天×3万 token×$5/M=$150/天≈$4500/月;压到 6k→$30/天,<20 行代码省 ~$3500/月。

**诚实边界**:benchmark 是它自己跑的,建议拿自己真实 payload 复现;压缩比调太狠会误删边缘上下文→影响细致推理,从保守起步慢慢调;不适用单一 provider 原生压缩+不需跨 agent 记忆,或沙箱跑不了本地进程的场景。

**在本笔记的位置**:把前面推导的省钱原理工程化——攻击 context 膨胀(输入平方膨胀的容器)、重发整本聊天记录、主动对齐前缀让 prompt cache 命中。是"方法论/工作场景盘点·省钱抓手"那格的现成答案。

链接:github.com/chopratejas/headroom · 模型 chopratejas/kompress-v2-base(HF)

### 概念消歧 · 模型压缩 vs 上下文压缩(Headroom 是哪种)

> Headroom 不是"模型压缩"。两个完全不同的概念:

|                             | 改的是什么                       | 手段                                                 |
| --------------------------- | -------------------------------- | ---------------------------------------------------- |
| 模型压缩(model compression) | 神经网络本身,让模型变小变便宜    | 量化 quantization / 剪枝 pruning / 蒸馏 distillation |
| 上下文压缩(Headroom 干的)   | 喂给模型的输入数据,让 token 变少 | 剥噪音、丢无关句、摘要历史                           |

Headroom 属后者:根本不碰你调用的大模型(Claude/GPT 原样不动),只在数据进 context 前把文字/JSON/日志压小。

**易混点:"用模型压缩" ≠ "模型压缩"**。Headroom 压缩时用到小模型(Kompress-base、打分 cross-encoder),是"用小模型压你的输入",不是"把大模型压小"。且其手段一半不用模型:

- 确定性规则(不用模型):JSON minifier 剥 null、AST 压代码。
- 小模型/embedding(用模型):语义打分丢无关句、Kompress-base 压散文。

无论哪种,被压对象始终是"你的输入数据",不是大模型。

**后端类比**:

```
模型压缩   = 把数据库引擎换精简版(改引擎本身)
上下文压缩 = 查询前 gzip 一下 request body(引擎没动,只压传过去的数据)
Headroom   = 后者;gzip 时可能用了智能小模型,但压的永远是 payload
```

> 一句话:Headroom = 输入侧上下文压缩(可借助小模型),不动大模型;model compression(量化/剪枝/蒸馏)改模型本身,是另一条赛道。

### Headroom 压缩策略 · 分而治之(路由→分治→可逆兜底)

**总策略三步:**

```
① ContentRouter 认出内容类型(JSON/代码/散文/RAG/对话历史/图片)
② 派给对应专用压缩器(不同类型不同手法)
③ CCR 把原文存本地→压过头时模型可 headroom_retrieve 取回(可逆兜底)
```

**三大压缩术家族:**

| 家族            | 针对内容     | 怎么压                                                       | 用模型  | 有损               | 例子                                                    |
| --------------- | ------------ | ------------------------------------------------------------ | ------- | ------------------ | ------------------------------------------------------- |
| ①确定性结构压缩 | JSON/代码    | SmartCrusher 递归剥 null/空数组/样板/冗余 key;CodeCompressor 按 AST 压(Py/JS/Go/Rust/Java/C++) | ❌纯规则 | 几乎无损(只删噪音) | GitHub API 响应 3000→几百;REST 平均省 ~65% 不丢有效信息 |
| ②语义相关性裁剪 | RAG/检索结果 | SemanticCompressor 用 cross-encoder 给每句按"与当前问题相关性"打分,丢低分句(可换 BM25 更快) | ✅小模型 | 有损但针对性强     | 问"401k 罚金"→丢无关"医保"chunk,只留"10%罚金"句,80→15   |
| ③摘要压缩       | 对话历史     | ConversationCompressor 盯 token 预算,逼近时把老对话压成 bullet(留意图/偏好/事实),最近 3 轮保留原文 | ✅小模型 | 老的有损新的无损   | 不像 LangChain 粗暴砍最老,而是"压旧留新"                |

> 关键区别:②按相关性丢(query-aware),③按新旧压(recency-aware)。

**横切策略(全局加成,不针对某类内容):**

- CacheAligner:稳定前缀让 provider KV/prompt 缓存真正命中——不是删 token,是让发出去的 token 变便宜(命中价更低)。咬合 prompt caching。
- IntelligentContext:按学到的重要性打分,在 token 预算内择优塞。
- CCR 可逆压缩:原文留本地,模型缺信息时按需取回——让"敢压"成立。

**设计心法:** ①发过去 80% 是噪音先砍;②不同内容不同压缩器(无 one-size-fits-all);③query-aware 丢无关;④recency-aware 留最近原文;⑤reversible 不永久丢信息;⑥cache-aware 顺手薅 prompt cache。

> 收束:压缩策略 = 先路由(认类型)→ 再分治(JSON 规则剥/RAG 按相关性裁/历史按新旧摘要)→ 可逆兜底(CCR)+ 横切薅缓存(CacheAligner)。本质=把"哪些 token 是噪音"按内容类型拆给最合适的小工具判断。

### 压缩可靠吗 & 压缩模型该选多大(反直觉:不是越大越放心)

**一、可靠吗——分路径看,取决于走哪条路径 + 压多狠:**

| 压缩路径                               | 可靠度     | 风险点                                 | 原因                                        |
| -------------------------------------- | ---------- | -------------------------------------- | ------------------------------------------- |
| 确定性结构(JSON 剥 null/AST 压代码)    | ⭐⭐⭐ 很可靠 | 几乎没有                               | 只删可证明冗余(null/样板),不用模型,近无损   |
| 语义相关性裁剪(cross-encoder 打分丢句) | ⭐⭐ 中等    | 丢掉"看似不相关、实际多跳推理要用"的句 | 相关性是对当前 query 判断的,多跳/长尾易误伤 |
| 对话摘要(老对话压 bullet)              | ⭐ 风险最高 | 摘要丢细节、误差累积                   | 生成式压缩;但保留最近几轮原文限住爆炸半径   |

关键保险:CCR 可逆——原文存本地,模型缺信息时 headroom_retrieve 取回,比模型规模更重要的兜底(前提:模型知道去取、没过 TTL)。

> 诚实结论:确定性部分很可靠;模型部分在默认/中等压缩比下可靠(benchmark GSM8K ±0),压狠了有损且风险随任务变。"答案不变"是它自己 benchmark+默认比例的结论,非对你工作负载的保证,必须自测。

**二、选多大规模才放心——反直觉,不是越大越好(回扣 模型档次=f(难度×后果)):**

1. 压缩是"判别/抽取"轻任务:相关性打分用几十~百 M 的小 reranker(MiniLM 级)就久经验证且快,甚至 BM25 零模型够用;用大模型判断一句话相不相关是杀鸡用牛刀。
2. 轴选错了:语义压缩该问"是否针对压缩任务专训"而非"多大"——专精小模型(如 Kompress-base 拿 agentic trace 专训)常胜通用大模型。
3. 唯一对规模敏感+后果高的是摘要路径:要花钱买放心就花这;同时把它压得最保守。
4. 真正给"放心"的五条:①任务专精>规模 ②能用规则就用规则(JSON/AST/BM25 零模型不确定性) ③CCR 可逆(比规模重要) ④保守比例+自测 ⑤守延迟/成本预算(压缩每次调用都跑,塞大模型拖慢变贵)。
5. 经济悖论:用大模型压缩=拿大模型 token 去压缩,反噬省钱;故压缩模型就该小。

**实操阶梯(从最放心往上加):**

```
最放心 ── 确定性(JSON minify/AST/BM25)      ← 零模型风险,先用满
   ↑      语义裁剪用专训小 reranker(MiniLM 级) ← 验证充分、快、够用
   ↑      摘要路径才考虑强一点模型,但压得保守
最花钱 ── 全程大模型压缩                       ← 通常没必要且反噬省钱
```

配套:压缩比从保守起步→在真实任务测准确率→再调高;CCR 的 TTL 别太短。

> 收束:确定性路径很可靠、模型路径中等且靠 CCR 兜底;选模型别迷信规模——压缩是判别/抽取轻任务,专训小模型+能用规则用规则+CCR 可逆+保守比例+自测,比"换大模型"更放心也更省。

### Headroom 评估方法拆解 · 省钱数字怎么来的(读源码)

> 读 benchmarks/ 源码后:它有两套完全不同的测试,运维 92% 来自"故事"那套(合成+估算),真准确率在另一套。

**一、省钱 92% 怎么测的——agent_cost_benchmark.py(注释自称 "The Compelling Story"):**

1. 合成数据非真实流量:generate_log_entries/generate_search_results 程序化生成(50 轮会话、一堆日志)。
2. token 是估算:CHARS_PER_TOKEN=4,字符数÷4 当 token,不用真分词器。
3. 根本不调 LLM:纯算术,数压缩前后的"字符/4"。
4. 成本乘出来:token×2025 公开价(Claude 3.5 Sonnet $3/M),非真实账单。
5. "质量"=需要项留存率:数据里埋"针"(is_needle)+error 条目,SmartCrusher 把 100 条列表截到 20 条(max_items_after_crush=20),数埋的关键项有没有活下来。

> README "10144→1260 same FATAL found" 就这么来的:日志=大量 routine+几条关键的长列表,截断到 20→token 暴降 92%,埋的 FATAL 保住。诚实结论:92% 高是因日志冗余度极高;密集代码/散文没这么多(它自己表里"代码库探索"仅 47%)。数字=合成+估算+算术成本的方向性投影,非你真实账单。

**二、真准确率证据——comprehensive_eval.py("Real Data, Real Accuracy"):**

| 维度       | 做法                                                         |
| ---------- | ------------------------------------------------------------ |
| 真实数据集 | BFCL v3(函数调用真 schema+ground truth)、HotpotQA(多跳 QA 真维基+核对答案)、真实 GitHub 数据(k8s/pytorch)、真实日志(Spring/k8s/Python traceback) |
| 真调 LLM   | 需 ANTHROPIC_API_KEY,压缩 vs 不压缩都发真模型,比对 ground truth |
| 双指标     | 同时测压缩率+准确率保持                                      |
| 样本量     | 每集 N≈20(README 另处 GSM8K/TruthfulQA N=100)→ 总之 N 偏小   |

> 亮点:用 HotpotQA 多跳,正好测"语义裁剪会不会丢掉多跳推理需要的句子"的风险。

**三、整个项目评估方法体系(四类):**

### 打地基补丁 · prefill/decode 怎么进前向传播 & 为什么能缓存

**先纠术语**:后向传播(backpropagation,算梯度更新权重)只在训练时发生;推理只有前向传播,没有后向。所以问题应是"整段 token 一次进一次前向,还是一个一个进"。

**答案:两种都有,分两阶段:**

| 阶段                    | 处理什么           | 怎么进前向传播                   |
| ----------------------- | ------------------ | -------------------------------- |
| Prefill(吃输入/context) | 整段 prompt+历史   | 一整段一次性、并行进一次前向传播 |
| Decode(吐输出)          | 生成的每个新 token | 一个一个来,每个走一次前向传播    |

为什么一个并行一个串行:输入整段已知→可同时塞进网络一次算完(因果 mask 保证每位置只看前面,但计算并行);输出自回归→第 N+1 个依赖第 N 个(得先采样才知道)→只能串行。

```
Prefill: 整段 context ─一次并行前向─→ 全部 token 的 K/V(存缓存)+ 预测第1个新 token
Decode:  新token1 ─前向(只算这1个,复用缓存)→ 新token2 → ... 直到 EOS
```

(注:"全景收口"图 step③ 描述的是 decode 单 token 生命周期;完整应是 先 prefill 整段并行 → 再 decode 逐 token。两阶段都只是前向传播,区别在一次喂多少 token。)

**为什么能缓存——根本原因:每个 token 的 K/V 一旦算出就永不改变。** 两条性质:

1. 因果性(causal mask):第 i 个 token 的 K/V 只依赖自己+前面 token,完全不看后面。
2. 追加而非修改:只在末尾追加 token,从不改前面。

合起来:token i 的 K/V = f(token 1..i),而 1..i 永不变 → K/V 被冻住,后面加什么都不回头改它。这个不可变性就是能缓存的全部理由。

**破解"整段 prefill,子前缀哪有余地"**:并行计算≠结果纠缠。整段并行只是计算方式,产出的 K/V 仍按位置独立——第 30 个的 K/V 只由 1..30 决定,与第 31 个之后无关。所以子前缀的 K/V 可单独抠出来复用:

```
请求1: [系统提示50][问题A10] → 前50个 K/V 只依赖系统提示,与问题A无关
请求2: [系统提示50(相同)][问题B8] → 前50个 K/V 与请求1完全相同 → 从缓存加载,只 prefill 问题B
```

前缀的 K/V 不受后缀影响 → 共享前缀永远可复用。

**两个缓存场景同一原理:**

| 场景                          | 复用什么                                    | 靠         |
| ----------------------------- | ------------------------------------------- | ---------- |
| 单次请求内(decode 循环)       | 每生成新 token 复用前面所有 K/V 不重算      | K/V 不可变 |
| 跨请求(prompt/prefix caching) | 两请求共享前缀→前缀 K/V 相同→跳过重 prefill | K/V 不可变 |

"全景收口"图看到的是第一种;Headroom 的 CacheAligner 攻第二种(稳定前缀让跨请求缓存命中)。

> 一句话:推理只有前向无后向;输入整段并行进(prefill)、输出逐 token 串行进(decode)。能缓存是因为 因果性+只追加 ⇒ 每个 token 的 K/V 一旦算出永不改变;整段 prefill 只是并行算法,产出仍是按位置独立互不纠缠的 K/V,所以任何前缀的 K/V 都不受后缀影响,共享前缀(请求内历史或跨请求相同开头)即可直接复用跳过重算。

### 多轮对话缓存实例 · [p1,a1,p2,a2] 第二轮用到第一轮了吗

**答:有,这正是多轮省钱的核心。** 第二轮前缀 [p1,a1] 复用第一轮算好的 K/V——前提:开了 prompt/prefix caching、TTL 内、前缀逐字节相同。

**时间线:**

```
第1轮: 发[p1] → prefill p1 算 K/V → decode a1(同时算 a1 的 K/V)
        ⇒ [p1,a1] 的 K/V 都已算好(开缓存则存下)
第2轮: 发[p1,a1,p2]  ← 历史得整段重发
        前缀[p1,a1]与第1轮逐字节相同 ⇒ 命中缓存,加载其 K/V 跳过重算
        ⇒ 只需 prefill p2 → decode a2
```

为什么能复用:每个 token 的 K/V 只依赖前缀、一旦算出永不变;第2轮后面接 p2 丝毫不改 [p1,a1] 的 K/V,照搬即可。

**好处分两种,别误解成免费:**

| 好处      | 是什么                                                      | 注意      |
| --------- | ----------------------------------------------------------- | --------- |
| 计算/延迟 | 跳过重 prefill [p1,a1]→首 token 更快(TTFT 低)、省算力       | 实打实省  |
| 成本      | 命中缓存的 input token 按骨折价(Anthropic ~1折/OpenAI ~5折) | ⚠️不是免费 |

⚠️ 你第二轮仍要把 [p1,a1] 整段重发(="你发的是整本聊天记录")。缓存不能让你不发,只让重发更便宜更快——付缓存折扣价,非 0。

**三个前提(缺一就没好处):**

1. 得开缓存:OpenAI/DeepSeek 自动;Anthropic 要打 cache_control 断点。没开→第二轮整段从头重算,第一轮白算(但照样得重发并按全价付)。
2. TTL 内:缓存有寿命(如 Anthropic 默认 5 分钟);真人来回通常够快命中,隔久过期按全价重算。
3. 前缀逐字节相同:[p1,a1] 任一处变了(或前面插动态时间戳)→ 从变化点往后缓存全失效——正是 Headroom CacheAligner 要解决的(稳住前缀别漂)。

> 一句话:第二轮确实复用第一轮——前缀 [p1,a1] 命中缓存、跳过重算、按折扣计费(因 K/V 只依赖前缀永不变);但历史照样整段重发,缓存省的是"重发的算力和价钱"非"不用发",且要开缓存+TTL 内+前缀逐字节不变才吃得到。

### 打地基 · prefill 究竟完成了什么

**一句话:prefill 把整段输入"读懂、消化"一遍,留下两样东西。它不生成回答,只为生成做准备。**

```
整段输入 [t1..tn] ─一次并行前向传播→
  产物①:每个输入 token 的 K/V 缓存(每层都存)  ← decode 反复读的"记忆"
  产物②:最后一个位置算出的"下一个 token 分布"   ← 即第一个输出 token 的预测
```

- 产物①KV 缓存(主要成果):对每个输入 token、每一层算并存 K/V,是模型对输入的"理解";decode 每生成一个新 token 都回头读它。这正是"能被缓存复用的东西"(请求内 decode 复用 + 跨请求 prefix caching 复用,复用的都是这批 K/V)。
- 产物②第一个输出 token 分布:整段读完后,最后位置输出→logits→softmax→下一 token 分布,采样得第一个输出 token。

**prefill 不做什么**:不产出整个 a1(a1 由 decode 逐个生成,prefill 最多到"第一个输出 token 分布");中间位置的预测被丢掉(推理只取最后位置分布,中间位置 K/V 留着用、logits 扔掉)。

**prefill vs decode:**

|        | prefill                                 | decode                                     |
| ------ | --------------------------------------- | ------------------------------------------ |
| 干什么 | 整段并行消化,建 K/V 缓存+出第一个 token | 逐个生成后续 token,每步读缓存+追加自己 K/V |
| 瓶颈   | 算力(并行吃满 GPU)                      | 显存带宽(一次一个,反复读大缓存)            |
| 决定   | TTFT(多久见第一个字)                    | 后续吐字速度                               |

后端类比:prefill=把整份输入加载进工作内存+建好索引(K/V)+给出第一条预测;decode=反复查索引,一次追加一条。KV 缓存=索引。

**配套澄清:有缓存时"只 prefill 新段"与"一整段 prefill"不矛盾**。"一整段 prefill"是冷启动(无缓存);命中缓存时前缀 K/V 直接加载不重推,只 prefill 新后缀。能跳过是因为:算 token i 的 K/V 只需它自己 embedding + 读取(非重算)前面 K/V——前缀 K/V 缓存现成就只推新 token,新 token 算时照样 attends 到加载进来的前缀 K/V。故准确说法:prefill=把"还没有 K/V 的那些 token"并行推一次前向;冷启动=全部,命中缓存=只新后缀。(超长输入实现上可能 chunked prefill 分块推,概念不变。)

> 收束:prefill=整段输入并行跑一次前向,产出①每输入 token 的 K/V 缓存(对输入的理解,也是被缓存复用的)+②最后位置下一 token 分布(第一个输出 token);不生成整段回答,把逐字生成交给 decode。

### prefill/decode/缓存 · 数学模型 + 形象化模型

**一、数学模型**　记号:输入 $x=(x_1,\dots,x_n)$,权重 $\theta$(推理固定),$L$ 层,头维度 $d$,温度 $T$;已缓存前缀长 $p$,输出长 $m$。

**① 每 token 的 K/V + 因果引理(能缓存的根本):**
$$
(k_i^{\ell},\, v_i^{\ell}) = f_{\text{KV}}^{\ell}(x_{1:i};\,\theta),
\qquad
\frac{\partial (k_i, v_i)}{\partial x_j} = 0 \quad \forall\, j > i
$$

**② 注意力(算当前 token 要读前缀所有 K/V):**
$$
\alpha_{tj} = \operatorname{softmax}_{j \le t}\!\left(\frac{q_t \cdot k_j}{\sqrt{d}}\right),
\qquad
\text{out}_t = \sum_{j \le t} \alpha_{tj}\, v_j
$$

**③ prefill(对输入并行一次,两个产物):**
$$
x_{1:n} \;\xrightarrow{\ \text{parallel}\ }\; \{(k_i,v_i)\}_{i=1}^{n}\ \text{(KV cache)},
\qquad
P(x_{n+1}\mid x_{1:n}) = \operatorname{softmax}\!\left(\tfrac{\text{LMHead}(h_n)}{T}\right)
$$

**④ decode(自回归循环 + 链式分解):**
$$
x_{t+1} \sim P(\cdot \mid x_{1:t}) = \operatorname{softmax}\!\left(\tfrac{\text{LMHead}(h_t)}{T}\right),
\qquad
P(x_{n+1:n+m}) = \prod_{t=n}^{n+m-1} P(x_{t+1}\mid x_{1:t})
$$

**⑤ 成本模型(decode 复杂度 + 计费):**
$$
\text{decode cost} \;\propto\; \sum_{t=0}^{m-1}(n+t) = mn + \frac{m(m-1)}{2}
$$

$$
\text{Cost} = \underbrace{p\,c_{\text{cached}} + (n-p)\,c_{\text{in}}}_{\text{输入侧(缓存分层)}} + \underbrace{m\,c_{\text{out}}}_{\text{输出侧(含思考链)}},
\qquad c_{\text{out}} \gg c_{\text{in}} \gg c_{\text{cached}}
$$

**二、形象化模型:读书+写批注+续写**(接手一本只写了前半的书往下续写):

| 动作                                                       | 对应                                    |
| ---------------------------------------------------------- | --------------------------------------- |
| 桌上已有内容                                               | 输入 context x₁…ₙ                       |
| 通读全文,逐句旁写"批注"(只概括到这句为止,写完不改)         | prefill 算每 token 的 K/V               |
| 读完最后一句冒出"下句写啥"的念头                           | prefill 产物②第一个输出 token 分布      |
| 一句句续写:每写一句先回头扫所有批注定写什么,写完补一条批注 | decode 逐 token,attends 读缓存+追加 K/V |
| 写到"全文完"                                               | 采样到 EOS                              |
| 前 N 句批注还在手边没擦、且一字没动→直接接着写不重读       | prefix caching 命中                     |
| 每次续写都把整本书+之前续写重新摆上桌                      | 多轮对话重发历史                        |

批注写完不改=能缓存:一句的批注只总结它前面的内容,后面写啥都不让它失效=∂(kᵢ,vᵢ)/∂xⱼ=0(j>i)的大白话。

**三、对照表:**

| 数学               | 形象化               | 概念              |
| ------------------ | -------------------- | ----------------- |
| (kᵢ,vᵢ)            | 第 i 句的批注        | token 的 K/V      |
| f_KV 并行算全部    | 通读逐句批注         | prefill           |
| ∂(kᵢ,vᵢ)/∂xⱼ=0,j>i | 批注只看前文写完不改 | 因果不可变→可缓存 |
| outₜ=Σ_{j≤t}αₜⱼvⱼ  | 写前回头扫所有批注   | 注意力读缓存      |
| x_{t+1}~P(·        | x₁…ₜ) 循环           | 一句句续写        |
| 缓存命中→只算(n−p) | 复用旧批注只读新增   | prefix caching    |
| m·price_out 主导   | 续写越长越贵         | 输出最贵          |

> 一句话:数学上 prefill 把输入并行映射成"每 token K/V 缓存+末位下一 token 分布",decode 按 P(输出)=ΠP(x_{t+1}|x₁…ₜ) 逐步采样、每步读缓存并追加 K/V,因 (kᵢ,vᵢ) 只依赖前缀(∂/∂xⱼ=0,j>i)故可缓存;形象上通读全书逐句写下"写完不改"的批注(prefill),再回头扫批注一句句续写(decode),老批注还在就直接复用(缓存)。


### 数学模型变量词典 · 零基础贯通(配具体例子)

> 配套上一节。核心:这套公式描述的就是"模型拿你输入的对话历史去生成回复"的全过程。

**贯穿例子**:这一轮你实际发出的整段历史 = `[系统]你是助手 [p1]1+1等于几? [a1]2 [p2]那2+2呢?`,被分词成约 20 个 token = $x$;模型要续写出 a2(="4")。

**第一类·输入(全来自你发的对话历史):**

| 符号                | 是什么          | 哪来的/例子                                                  |
| ------------------- | --------------- | ------------------------------------------------------------ |
| $x=(x_1,\dots,x_n)$ | 输入 token 序列 | 就是你这段历史分词后的结果;系统提示+p1+a1+p2 首尾相接成一条长序列 |
| $x_i$               | 第 i 个 token   | 序列里第 i 个词片(如某个是"2"、某个是"+")                    |
| $n$                 | 输入 token 总数 | 你这段历史长度(例≈20);= 你被计费的输入 token 数              |

核心关系:你输入的对话历史 = $x_{1:n}$,是公式唯一输入。模型不区分"哪句是历史哪句是新问题",对它就是一条长 token 序列。

**第二类·模型固定参数(训练完不变,与聊什么无关):**

| 符号     | 是什么                      | 哪来的                       |
| -------- | --------------------------- | ---------------------------- |
| $\theta$ | 模型权重(几千亿数字,"大脑") | 训练烧进去,推理固定不动      |
| $L$      | 层数                        | 架构常数(如 80),每层加工一遍 |
| $d$      | 每个 K/V 向量维度           | 架构常数(如 128)             |

**第三类·你能调的旋钮:** $T$(温度)=控随机性,你调用时设;大→发散,小→确定。

**第四类·模型算出的中间产物(数学模型主体):**

| 符号                     | 是什么                                         | 怎么来                                  |
| ------------------------ | ---------------------------------------------- | --------------------------------------- |
| $k_i^{\ell},v_i^{\ell}$  | 第 i token 在第 ℓ 层的 key/value 向量(="批注") | 用 θ 把 $x_i$ 和其前缀一起算出          |
| $q_t$                    | 第 t 位的查询向量                              | 由当前 token 算出,"问"前面谁相关        |
| $\alpha_{tj}$            | 注意力权重:位置 t 对 j 的关注度                | $q_t\cdot k_j$ 过 softmax,0~1           |
| $\text{out}_t$           | 位置 t 的注意力输出                            | 用 α 给前面所有 $v_j$ 加权求和          |
| $h_t$                    | 位置 t 的隐藏状态(流过所有层的向量)            | 模型在该位的内部理解,用来预测下一 token |
| $\text{LMHead}$          | 末层投影:$h_t$→词表每词分数                    | θ 的一部分(权重矩阵)                    |
| $P(x_{t+1}\mid x_{1:t})$ | 给定 $x_{1:t}$ 下,下一 token 的概率分布        | $\text{softmax}(\text{LMHead}(h_t)/T)$  |

**第五类·成本:** $m$=输出 token 数(回复长度);$p$=命中缓存的前缀长;$c_{\text{in}}/c_{\text{out}}/c_{\text{cached}}$=输入/输出/缓存命中单价。

**五个式子用例子重走:**

- ① 给历史里每个 token、每层算 K/V("批注");"2"的批注只依赖它前面,你后补"那2+2呢"不改它→能缓存。
- ② 算"4"前,拿它的提问 $q$ 去和前面每个批注 $k$ 点积定关注度 $\alpha$,再加权汇总 $v$;会高度关注"2+2"和"1+1=2"。
- ③ prefill:整段 $x_{1:20}$ 并行喂进→算全部 20 个 K/V(存缓存)+预测第 21 个 token(指向"4")。
- ④ decode:从第 21 个起逐个采样生成回复,每个追加 K/V;整段概率=每步条件概率连乘(自回归)。
- ⑤ 账:前缀 p 个按缓存价 + 新输入 (n−p) 个按输入价 + 生成 m 个按(最贵)输出价;decode cost∝mn+m(m−1)/2 → 历史越长、回复越长越慢越贵。

> 一句话:你输入的整段对话历史分词后就是 $x=(x_1,\dots,x_n)$,是公式唯一输入;prefill 把它读成 K/V 缓存并预测第一个回复 token,decode 逐个续写;$n$=历史长度(定输入成本),$\theta/L/d$=模型不变量,$T$=你的旋钮,其余 $k,v,q,\alpha,h,P$ 全是模型加工你这段历史时的中间产物。

### 打地基 · 为什么 prefill 可以并行(decode 不行)

**核心:能并行,因为 prefill 的输入全部已知。串行只在"下一步的输入=上一步的输出"时被迫发生——prefill 没这问题,decode 有。**

**关键区分:依赖"已知的值"≠ 串行。** 常见反例:"token i 要看前面所有 token(注意力),不就得排队?"——依赖前面 token 的"值"没问题,值早摆在那(是你的输入),注意力只是"读";真正逼出串行的是依赖一个"还没被产出"的 token。

- prefill:整段历史 $x_{1:n}$ 每个 token 都现成→谁都不等谁→一起算。
- decode:第 n+2 个要等第 n+1 个先被采样出来才知道→没法提前算→一个一个来。

**一个 transformer 层怎么"一次算完所有位置"**(不是循环 n 次,是几个大矩阵乘一次覆盖所有位置):

```
① Q/K/V: X(所有 token embedding 摞成矩阵) @ W_q/W_k/W_v
          → 每位置 Q/K/V 一次全算出(各只依赖自己 embedding,互不等待)
② 注意力: Scores=Q@Kᵀ(所有位置两两关注度,一个大矩阵)
          加因果 mask:j>i 的格子置 -∞ → softmax 后权重 0
          Out=softmax(Scores)@V
③ FFN:    所有位置并行各过一遍
```

重点是因果 mask:用"右上角全 -∞"的掩码,在一次并行矩阵运算里就保证"位置 i 只看 1..i"。所以"只看前缀"是靠 mask 实现、不是靠串行执行——结果和顺序算完全一样,但计算并行。

**形象类比(接读书+批注):**

- prefill=整本书已印好:第 1 页、第 50 页可分给不同人同时批注;哪怕批注第 10 页要回看第 1~9 页,那些页已印在那随时可读,所以全部批注能一起做。
- decode=你在写新页:第 11 页没法在第 10 页写出前动笔,因为写什么取决于第 10 页写成了啥。只能一页接一页。

**硬件视角:** GPU 上千核擅长大矩阵并行。

|               | prefill          | decode                           |
| ------------- | ---------------- | -------------------------------- |
| 工作形态      | 一大批独立矩阵乘 | 一次只算 1 个 token              |
| 瓶颈          | 算力(喂满 GPU)   | 显存带宽(每步读一遍整个 KV 缓存) |
| 每 token 速度 | 快(并行摊薄)     | 慢(算力闲置,卡在搬数据)          |

> 一句话:prefill 能并行因输入每个 token 都已知,各位置 Q/K/V 可独立同时算,注意力用因果 mask 在一次大矩阵运算里实现"只看前缀",不需排队;decode 下一个 token 必须等上一个被采样出来,这种"输入=上一步输出"的依赖才是串行的唯一根源。

### 打地基 · prefill 阶段独立吗(跨 token / 跨层)

**两个轴分开看:跨层=严格串行不独立;跨 token=并行计算但不相互独立(注意力把它们耦合)。**

**纠误解:不是"每个 token 各自跑完所有层"。** 真实数据流是逐层推进、每层一次性处理所有 token:

```
Layer 1: 对【所有 token 一起】(Q/K/V→注意力混合→FFN)→ 所有 token 的 layer-1 输出
Layer 2: 拿 layer-1【全部】输出再对所有 token 算一遍 → layer-2 输出
... Layer L → 每 token 最终隐藏状态
```

为什么 token 不能自己单独跑完所有层:第 ℓ 层算 token i 要读 token 1..i 在第 ℓ 层的 K/V→必须等 1..i 都到第 ℓ 层→只能全员一层层推。

**一层内三步,哪步独立:**

| 子步                   | 跨 token 独立? | 说明                                             |
| ---------------------- | -------------- | ------------------------------------------------ |
| ① Q/K/V 投影           | 独立           | token i 的 Q/K/V=自己当前向量@W,不看别人         |
| ② 注意力 softmax(QKᵀ)V | 不独立(耦合)   | token i 融合 1..i 所有信息——token 间唯一"对话"处 |
| ③ FFN                  | 独立           | 只吃自己注意力后的向量                           |

"独立"只在①③成立;②注意力把所有 token 绑一起。"并行"=同层所有 token 一次大矩阵算完,≠互不依赖。

**独立性精确表述(接因果引理):**

- 对未来 token(j>i):独立。因果 mask → ∂(kᵢ,vᵢ)/∂xⱼ=0, j>i。
- 对过去 token(j<i):不独立。注意力混入前缀信息→token i 依赖整个前缀。
- 层数细节:第 1 层 kᵢ¹ 只由 token i 自己 embedding 算出(未经注意力混合)→只依赖自己;越深层 kᵢˡ(ℓ≥2)依赖 x_{1:i}。但无论第几层,对未来 token 永远独立(mask 管着),故缓存始终成立。

**类比:** 整本书已印好,组队逐句写批注——逐层=写好几轮:第一轮各看本句写粗批注(Q/K/V 独立),第二轮回看前文所有批注精修(注意力耦合),第三轮各自润色(FFN 独立);每轮全员做完才进下一轮(层串行);同轮内可同时动笔(token 并行)但精修必须参考前文(不独立);谁都不许偷看自己后面的句子(因果)。

> 一句话:prefill 不是"每 token 独立跑完各层";层严格串行(ℓ+1 吃 ℓ 的输出),同层所有 token 并行但不相互独立——Q/K/V 投影和 FFN 逐 token 独立,注意力把每个 token 与其前缀耦合;token 只对未来独立(因果 mask),对过去不独立,越深层越融合前缀。
