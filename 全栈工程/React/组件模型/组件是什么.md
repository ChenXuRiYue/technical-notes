# 📌 React 组件是什么

React 组件是一个**根据输入描述一块用户界面的可组合单元**。它把 UI 的结构、数据和交互边界放在一起，让开发者能像搭积木一样，用小组件组成更大的组件，最终形成整个页面。

例如，一个页面可以被拆成：

```text
App
├── Header
├── Sidebar
└── MainContent
    ├── SearchBox
    └── ProductList
        ├── ProductCard
        └── ProductCard
```

这棵树叫做**组件树**。React 应用不是一个巨大的 HTML 文件，而是一棵由组件组成的 UI 树。

## 📄 从一段重复的 UI 开始

假设页面中要显示多个用户：

```tsx
<div className="user-card">
  <h2>小明</h2>
  <p>后端工程师</p>
</div>

<div className="user-card">
  <h2>小红</h2>
  <p>前端工程师</p>
</div>
```

两块 UI 结构一样，只是数据不同。可以将稳定结构抽成组件：

```tsx
type UserCardProps = {
  name: string;
  role: string;
};

function UserCard({ name, role }: UserCardProps) {
  return (
    <div className="user-card">
      <h2>{name}</h2>
      <p>{role}</p>
    </div>
  );
}
```

然后使用它：

```tsx
<UserCard name="小明" role="后端工程师" />
<UserCard name="小红" role="前端工程师" />
```

这里发生了一次抽象：

```text
可变数据 name / role
           ↓
props 传入组件
           ↓
UserCard 决定稳定的 UI 结构
           ↓
得到具体用户卡片
```

因此，组件不只是「复用一段 HTML」。它定义了一块 UI 的**输入、输出和责任边界**。

## 📄 组件的输入和输出

可以先用一个类似函数的模型理解组件：

```text
UI = Component(props, state)
```

| 部分 | 含义 |
| --- | --- |
| `props` | 父组件传入的外部数据，组件不应直接修改 |
| `state` | 组件需要在多次渲染之间记住的内部数据 |
| JSX | 组件根据当前数据返回的 UI 描述 |

例如计数器组件：

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      点击次数：{count}
    </button>
  );
}
```

`count` 改变后，React 会再次执行 `Counter`，得到新的 JSX，然后将必要的变化更新到页面。

```text
用户点击
   ↓
state 改变
   ↓
组件重新渲染
   ↓
得到新的 UI 描述
   ↓
React 更新页面
```

## 📄 谁调用组件，返回的 JSX 去了哪里

初学时容易把函数组件理解成普通函数调用：

```tsx
const result = App();
```

但在常规 React 代码中，业务代码通常不直接调用组件函数，而是用 JSX 把组件放进组件树：

```tsx
root.render(<App />);
```

`<App />` 的含义不是“现在立刻按普通函数调用语法执行 App”，而是创建一份 React 元素描述：

```text
这个位置要渲染 App 组件。
```

接下来由 React 控制组件的调用时机：

```text
React 遇到 <App />
  -> React 调用 App 组件
  -> App 读取当前 props、state 和 Context
  -> App 返回 JSX
  -> React 继续遍历 JSX 描述的子组件和原生标签
  -> React DOM 将需要的变化同步到浏览器 DOM
```

因此，组件需要 `return JSX`，是因为 React 需要知道：

```text
根据当前输入和状态，这个组件此刻应该长什么样。
```

JSX 是 UI 描述，不是手写的 DOM 操作指令。组件可以返回 React 能渲染的内容，常见包括：

```tsx
return <button>保存</button>; // 一个 React 元素
return null;                    // 本次不渲染内容
return "加载中";               // 文本节点
return [<Item key="1" />];     // 一组可渲染节点
```

它不能随意返回业务普通对象：

```tsx
return { name: "Git Tributary" }; // 不是可渲染的 React 子节点
```

这也是组件函数与数据处理函数的关键区别：两者都是 JavaScript 函数，但组件的返回值要被 React 当作 UI 描述消费。

## 📄 组件定义、元素与实例

「组件」这个词在日常表达中容易混指三件事：

```tsx
function UserCard(props: UserCardProps) {
  return <h2>{props.name}</h2>;
}

function App() {
  return <UserCard name="小明" />;
}
```

| 层次 | 例子 | 理解 |
| --- | --- | --- |
| 组件定义 | `function UserCard(...)` | 一张可重复使用的 UI 蓝图 |
| React 元素 | `<UserCard name="小明" />` | 一份「请渲染 UserCard，并给它这些 props」的描述 |
| 组件在树中的存在 | React 渲染后维护的那个 `UserCard` | 这个位置对应自己的 state 和生命周期 |

类比来看：

```text
组件定义 ≈ Java 的 class
JSX 元素   ≈ 创建对象的意图描述
树中的组件 ≈ 运行期中的具体对象
```

这个类比只用来建立直觉，并不严格等价。特别是函数组件不会向业务代码暴露一个可以通过 `this` 操作的组件对象。

## 📄 函数组件与类组件

React 历史上有两种主要的组件写法。它们都是 React 组件，只是组织代码的方式不同。

### 🔖 函数组件

```tsx
function Welcome({ name }: { name: string }) {
  return <h1>你好，{name}</h1>;
}
```

- 用 JavaScript / TypeScript 函数定义。
- 直接返回 JSX。
- 通过 Hooks 使用 state、副作用等 React 能力。
- 是现代 React 代码的主流写法。

### 🔖 类组件

```tsx
class Welcome extends React.Component<{ name: string }> {
  render() {
    return <h1>你好，{this.props.name}</h1>;
  }
}
```

- 用 JavaScript / TypeScript 类定义。
- 通过 `render()` 返回 JSX。
- 通过 `this.state`、`this.setState()` 和生命周期方法使用 React 能力。
- 在旧项目、旧教程以及传统 Error Boundary 中仍然会遇到。

两者的共同点比写法差异更重要：它们都接收 `props`，都返回 UI 描述，都可以拥有状态，都是组件树中的节点。

### 🔖 为什么会并存两种写法

React 早期的函数组件只能根据 `props` 返回 UI，不能使用组件状态和生命周期，所以有交互的组件通常要写成类组件。React 16.8 引入 Hooks 后，函数组件也能使用状态、副作用和上下文，因此逐渐成为主流。

```text
新 React 代码   → 通常写函数组件 + Hooks
维护旧项目     → 仍然可能遇到类组件
传统错误边界   → 仍常使用类组件
```

对组件的使用者来说，两种实现的使用方式一样：

```tsx
<Welcome name="小明" />
```

调用方通常不需要知道 `Welcome` 内部是函数还是类。

## 📄 「有哪些组件」要看分类角度

「函数组件」、「页面组件」和「受控组件」不在同一个分类层级。同一个组件可以同时拥有多个标签。

| 分类角度 | 常见名称 | 回答的问题 |
| --- | --- | --- |
| 实现方式 | 函数组件、类组件 | 组件用什么语法实现？ |
| 业务职责 | 页面组件、布局组件、业务组件、通用 UI 组件 | 组件负责什么？ |
| 数据职责 | 容器组件、展示组件 | 谁获取和管理数据？ |
| 状态控制 | 受控组件、非受控组件 | 组件的值由谁保存？ |
| 执行位置 | 服务端组件、客户端组件 | 组件在哪里执行？ |

例如，一个组件可以同时是：

```text
函数组件 + 页面组件 + 客户端组件
```

### 🔖 宿主元素与自定义组件

JSX 中还要区分小写标签和大写标签：

```tsx
function SaveButton() {
  return <button>保存</button>;
}
```

- `<button>` 以小写开头，React DOM 将它当作浏览器原生元素。
- `<SaveButton />` 以大写开头，React 将它当作开发者定义的组件。

所以自定义组件名必须以大写字母开头。如果写成 `<saveButton />`，React 会尝试把它当作一个宿主标签，而不是调用 `SaveButton` 函数。

## 📄 组件怎么使用

一个组件从定义到进入组件树，通常经过五步。

### 🔖 定义组件

```tsx
type UserCardProps = {
  name: string;
  role: string;
};

function UserCard({ name, role }: UserCardProps) {
  return (
    <section>
      <h2>{name}</h2>
      <p>{role}</p>
    </section>
  );
}
```

### 🔖 导出与导入

默认导出：

```tsx
export default UserCard;
```

对应的导入：

```tsx
import UserCard from "./UserCard";
```

也可以使用具名导出：

```tsx
export function UserCard({ name, role }: UserCardProps) {
  return <section>{name}：{role}</section>;
}
```

对应的导入需要花括号：

```tsx
import { UserCard } from "./UserCard";
```

### 🔖 渲染组件并传入 props

```tsx
function App() {
  return (
    <main>
      <UserCard name="小明" role="后端工程师" />
      <UserCard name="小红" role="前端工程师" />
    </main>
  );
}
```

同一个组件定义可以渲染多次。每个位置都能获得自己的 `props` 和状态。

整个过程可以缩写为：

```text
定义组件
   ↓
导出与导入
   ↓
用 <组件名 /> 将它放入组件树
   ↓
通过 props 传入数据和回调
   ↓
React 调用组件并渲染它返回的 UI
```

### 🔖 通过 children 组合 UI

组件标签中间的内容会作为 `children` 传入：

```tsx
function Panel({ children }: { children: React.ReactNode }) {
  return <section className="panel">{children}</section>;
}
```

使用时：

```tsx
<Panel>
  <h2>系统设置</h2>
  <button>保存</button>
</Panel>
```

`Panel` 决定外层结构，调用方决定里面放什么。这种「将组件放进另一个组件」的方式叫做组合。

## 📄 组件之间如何协作

组件不是孤立的，它们主要通过下面几种方式连接：

| 方式 | 方向 | 作用 |
| --- | --- | --- |
| `props` | 父 → 子 | 将数据与配置传给子组件 |
| 回调函数 | 子 → 父 | 子组件通知父组件发生了事件 |
| `children` | 父 → 子 | 将一段 UI 内容交给子组件包裹 |
| Context | 上层 → 深层 | 共享跨越多层组件的数据 |

最常见的是单向数据流：

```text
父组件持有数据
      │ props
      ▼
子组件展示数据
      │ 回调函数通知事件
      ▲
父组件修改状态
```

子组件通常不直接修改父组件的数据，而是发出事件意图，由数据拥有者决定如何更新。

## 📄 什么时候应该拆组件

可以优先观察四个信号：

- **可复用**：同一种 UI 结构出现多次。
- **独立职责**：一块 UI 有清晰的业务含义，如用户卡片、搜索框、侧边栏。
- **独立交互**：它自己管理一组密切相关的状态和事件。
- **阅读边界**：当前组件过长，抽出后能用名称表达意图。

但不是每个 `<div>` 都要拆成组件。拆分的目标是形成清晰的责任边界，而不是追求组件数量。

## 📄 先建立的心智模型

学习 React 组件时，可以先记住五句话：

1. React 页面是一棵组件树。
2. 组件是对一块 UI 的责任封装，不只是 HTML 片段。
3. `props` 是外部输入，`state` 是内部记忆，JSX 是当前输出。
4. 数据变化后，React 重新渲染相关组件并更新页面。
5. 函数组件和类组件是实现组件的两种写法，函数组件是现代主流。

建立这个模型后，Hooks 就不再是一组孤立 API：它们是函数组件获取状态、处理副作用和复用有状态逻辑的入口。
