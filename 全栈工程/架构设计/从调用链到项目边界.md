# 📌 从调用链到项目边界

把控项目结构不是先背诵架构名词,而是能从真实调用链识别四件事:谁接收外部输入、谁决定用例顺序、谁保护业务规则、谁对接数据库、网络、文件系统等外部世界。目录是物理分组,职责和依赖方向才是逻辑分层。

## 📄 先从用户动作画调用链

阅读一个项目时,选择一个具体用户动作:

```text
点击保存
提交订单
发布文章
同步数据
切换分支
```

然后只跟这一条链:

```text
用户意图
  -> UI 事件
  -> API / IPC / Controller 入口
  -> 应用用例
  -> 领域或能力模块
  -> Repository / Gateway / 文件 / 网络
  -> 状态、事件和返回值
```

每走过一个函数,不要只记函数名,而要标注:

| 观察项 | 需要回答的问题 |
| --- | --- |
| 输入 | 谁构造数据,是否信任? |
| 输出 | 返回领域对象、DTO 还是底层记录? |
| 顺序 | 谁决定先做 A 再做 B? |
| 状态 | 数据属于谁,谁可以修改? |
| 副作用 | 写数据库、发网络、改文件还是发事件? |
| 失败 | 错误在哪层分类、重试或转译? |
| 验证 | 哪类测试保护这个行为? |

这张表能将“跟调用”转化为“读职责”。

## 📄 逻辑分层的四个核心角色

### 🔖 接口适配层

职责:

```text
接收 HTTP / IPC / CLI / UI 消息
解析和校验输入形式
提取身份、请求上下文或框架状态
调用应用用例
把结果转成协议边界的响应
```

它可以叫 controller、handler、command、resolver 或 endpoint。名称不决定职责。

边界信号:

- 依赖 Web 框架、Tauri、gRPC、CLI 或其他传输机制。
- 输入输出主要是 DTO 或协议可序列化类型。
- 不应让传输框架类型渗透到稳定的业务规则中。

### 🔖 应用用例层

职责:

```text
完成一次用户意图
决定多个能力的调用顺序
管理事务、幂等、重试、回滚和应用级权限
协调领域对象、仓储、网关和事件发布
```

典型名称:

```text
CreateOrder
PublishArticle
SyncWorkspace
ResetPassword
```

用例层可以有业务意义,但不应被某个具体传输框架绑死。同一用例可能被 HTTP、消息队列、CLI 和定时任务共用。

### 🔖 领域或核心能力层

职责:

```text
表达稳定业务概念
保护不变式和核心规则
提供可组合的业务能力
尽量不依赖传输框架
```

并非每个项目都需要复杂 DDD 模型。工具型项目的核心层可以是稳定的能力库,例如解析、计算、渲染、版本控制或存储语义。

### 🔖 基础设施与外部适配层

职责:

```text
具体数据库实现
网络客户端
文件系统
系统时钟、随机数、密钥和操作系统 API
第三方 SDK
```

核心问题不是“有没有 infrastructure 目录”,而是核心用例是否可在不启动真实外部资源的情况下被理解和验证。

## 📄 物理分层不等于逻辑分层

一个 `commands/` 文件可能同时包含:

```text
框架注解和参数提取       -> 接口适配
先校验、再保存、最后发事件 -> 应用用例
字段合法性与状态转换       -> 领域规则
调用数据库或文件 API          -> 基础设施
```

因此看到目录名时,不要立即认定边界已经干净。应根据代码实际决策内容标注职责。

反过来,一个小型用例即使逻辑上属于应用层,也未必需要立即拆成独立文件。逻辑分类先帮助思考,物理拆分需要有实际收益。

## 📄 判断代码该放哪里

### 🔖 接口适配层适合保留的内容

- 请求参数解析和协议级校验。
- 框架状态、身份和请求上下文提取。
- DTO 与应用输入输出的映射。
- 调用一个用例并转换协议响应。
- 与传输框架强绑定的轻量操作。

### 🔖 应用用例层适合承担的内容

- 多步操作的业务顺序。
- 多个领域能力或外部端口的协调。
- 事务边界、幂等键、重试和回滚策略。
- 用例级授权、审计与事件发布时机。
- 跨 HTTP、IPC、CLI、消息和定时任务的共享流程。

### 🔖 核心层适合承担的内容

- 不依赖具体 UI 和传输方式的业务概念。
- 多个用例共享的稳定规则。
- 状态转换、值对象、核心计算和不变式。
- 希望独立测试和复用的能力。

### 🔖 基础设施层适合承担的内容

- SQL、HTTP、文件路径、操作系统和 SDK 细节。
- 核心或应用层定义的端口实现。
- 连接池、客户端、序列化和存储格式。

## 📄 用依赖方向检查边界

理想上,变化频繁的外部细节依赖更稳定的内部规则:

```text
传输适配器 -----> 应用用例 -----> 核心规则
       |                  |
       |                  +-----> 端口 trait
       |                              ^
       +-----> 框架细节              |
                                  基础设施实现
```

不要将图机械地理解为“所有项目必须建四个目录”。依赖方向的真正目标是:

```text
更换 UI 或传输框架时,核心规则不被迫修改
更换数据库或第三方 SDK 时,用例意图仍清晰
核心行为可在较少环境依赖下测试
```

## 📄 从代码阅读上升到架构判断

### 🔖 看变化原因

同一文件中的代码是否因同一原因修改?

```text
传输协议变化
用例步骤变化
核心规则变化
存储技术变化
```

如果四种变化频繁修改同一文件,分离职责可能已经有价值。

### 🔖 看重复的业务意图

代码文本不同,不代表没有重复。两个入口可能都在实现:

```text
加载对象 -> 验证 -> 执行 -> 保存 -> 发布事件
```

如果这是同一用例,应优先共享用例,而不是共享一堆无业务名称的 helper。

### 🔖 看测试成本

如果测试一条核心规则必须启动框架、数据库、网络和大量全局状态,通常表示规则与环境绑定过深。

但不要为了 mock 一切而创建大量一次性 trait。先优先抽离纯规则、明确用例和真实变化的外部端口。

### 🔖 看状态与并发边界

当架构代码涉及多把锁、事务或异步任务时,需要补一张状态表:

| 状态 | 拥有者 | 修改入口 | 同步机制 | 失败后保证 |
| --- | --- | --- | --- | --- |
| 业务对象 | 领域/仓储 | 用例 | 事务或乐观锁 | 不变式 |
| 内存状态 | 运行时容器 | handler/服务 | Mutex/channel | 重建或恢复 |
| 外部状态 | 远端系统 | gateway | 幂等键/重试 | 补偿或对账 |

边界不清时,并发问题往往不只是锁用错,而是没有明确谁有权修改状态。

## 📄 何时需要抽出应用层

不应仅因为架构图好看就创建 `application/`。出现以下信号时,抽离开始有实际价值:

- 一个 handler 需要调用多个领域或基础能力。
- 处理顺序本身是业务规则。
- 同一流程被多个入口重复实现。
- 开始出现事务、幂等、重试、回滚和补偿。
- handler 测试需要构造大量框架状态。
- 长时任务、锁和状态更新已难以一眼看清。
- 传输层改动经常迫使业务流程一起修改。

抽离后的入口应趋向:

```rust
fn handler(request: RequestDto, framework_state: FrameworkState) -> ResponseDto {
    let input = map_request(request);
    let output = use_case.execute(input, framework_state.user())?;
    map_response(output)
}
```

用例中则保留真正顺序:

```rust
fn execute(&self, input: Input) -> Result<Output, UseCaseError> {
    let entity = self.repository.load(input.id)?;
    let changed = self.policy.apply(entity, input.command)?;
    self.repository.save(&changed)?;
    self.events.publish(Event::Changed(changed.id()))?;
    Ok(Output::from(changed))
}
```

## 📄 避免为了分层而分层

常见过度设计:

```text
Controller -> Service -> Manager -> Processor -> Repository
```

如果每层只是原样转发,没有独立的变化原因、协议转换、规则或边界,那么层数只增加导航成本。

抽象前问:

```text
它隔离了哪个变化轴?
它保护了什么不变式?
它会被哪些入口复用?
它是否让测试更聚焦?
删除这层会造成哪种具体耦合?
```

无法回答时,先保持局部和直接。

## 📄 从会看代码到把控项目的发展路线

### 🔖 阶段一: 语法导航

能看懂:

```text
函数签名
Option / Result / ?
所有权与借用
struct / enum / trait / impl
模块路径与可见性
闭包与迭代器
```

产出:用一句话解释一个函数的输入、输出、数据使用方式和失败。

### 🔖 阶段二: 调用链还原

能追踪:

```text
用户动作 -> 入口 -> 用例 -> 能力 -> 副作用 -> 响应
```

产出:一张包含数据、错误、状态和测试的调用链图。

### 🔖 阶段三: 边界评估

能判断:

```text
物理目录与逻辑职责是否一致
依赖方向是否稳定
用例编排是否被框架绑定
核心规则是否可独立测试
状态所有者和并发边界是否明确
```

产出:问题列表和证据,而不是先给项目贴“架构不好”的标签。

### 🔖 阶段四: 演进决策

能给出:

```text
当前不改的理由
需要抽离的职责
最小可验证改动
需要的测试安全网
迁移顺序和回滚方式
抽象收益与新增成本
```

产出:从真实痛点出发的渐进方案,而不是一次性重写。

## 📄 一页式项目结构检查表

| 维度 | 健康信号 | 风险信号 |
| --- | --- | --- |
| 入口 | 入口类型与业务类型有明确转换 | 框架对象渗透到所有模块 |
| 用例 | 一次用户意图有可定位的编排点 | 同一流程分散在多个 handler |
| 核心 | 稳定规则不依赖 UI 和传输层 | 规则只能启动完整应用后测试 |
| 依赖 | 外部细节依赖稳定接口 | 核心模块直接知道多个具体 SDK |
| 状态 | 所有者、修改入口和同步机制清晰 | 任意模块都可直接修改全局状态 |
| 错误 | 保留结构,在边界转换 | 底层错误过早变成无类型字符串 |
| 并发 | 锁顺序和持有时间可见 | 持锁跨长 I/O 或 `await` |
| 测试 | 单元、集成和端到端各保护合适边界 | 只有大型端到端测试能发现规则回归 |
| 抽象 | 每个抽象对应真实变化轴 | 多层无语义转发 |
