# 📌 依赖注入方式（Spring DI）

基础谈资：IoC 容器的核心职责 —— 谁来创建对象、谁来装配依赖。三种注入方式的本质区别不在语法，在于**依赖关系的可见时机和可变性**。


## 📄 字段注入 @Autowired

把依赖直接标注在类的字段上，Spring 容器看到 @Autowired 就自动注入。

```java
@Service
public class OrderService {

    @Autowired
    private UserService userService;      // Spring 自动注入

    @Autowired
    private OrderRepository orderRepo;    // Spring 自动注入

    public void createOrder(Long userId) {
        User user = userService.findById(userId);  // 直接用，userService 肯定有值
        // ...
    }
}
```

看起来很干净，但有几个隐含问题：

**问题 1：外部看不出 OrderService 依赖什么。** 只看构造函数或 public 方法，完全不知道它需要 UserService 和 OrderRepository。拿到 new OrderService() 的人不知道缺了什么。

**问题 2：单元测试困难。** 想单独测 OrderService，必须启动 Spring 容器，或者用反射把私有字段塞进去：

```java
// 测试时的痛苦写法
OrderService orderService = new OrderService();
Field field = OrderService.class.getDeclaredField("userService");
field.setAccessible(true);  // 反射强制注入
field.set(orderService, mockUserService);
```

**问题 3：依赖可以被随意修改。** 任何地方拿到 orderService 实例后，都能通过反射改掉 userService，没有任何保护。


## 📄 Setter 注入

把依赖通过 setter 方法注入，方法上标注 @Autowired。

```java
@Service
public class OrderService {

    private UserService userService;
    private OrderRepository orderRepo;
    private EmailService emailService;  // 可选依赖，不是必须的

    @Autowired
    public void setUserService(UserService userService) {
        this.userService = userService;
    }

    @Autowired
    public void setOrderRepo(OrderRepository orderRepo) {
        this.orderRepo = orderRepo;
    }

    // emailService 没有 @Autowired，不注入也行
    public void setEmailService(EmailService emailService) {
        this.emailService = emailService;
    }

    public void createOrder(Long userId) {
        User user = userService.findById(userId);
        // ...
        if (emailService != null) {
            emailService.send(user.getEmail(), "下单成功");
        }
    }
}
```

**优势：可选依赖。** emailService 不是必须的，不注入也能跑。这在字段注入里做不到 —— 字段注入要么全注入，要么全不注入（按类型找不到就报错）。

**优势：测试方便一些。** 不需要反射，直接调 setter：

```java
OrderService orderService = new OrderService();
orderService.setUserService(mockUserService);
orderService.setOrderRepo(mockOrderRepo);
// emailService 不 set 也行，测试不影响
```

**代价：对象创建后状态不完整。** new OrderService() 出来时，userService 是 null，如果有人忘记调 setter 就直接用，运行时才 NPE。


## 📄 构造器注入

把依赖通过构造函数注入，字段声明为 final。

```java
@Service
public class OrderService {

    private final UserService userService;
    private final OrderRepository orderRepo;

    // Spring 4.3+ 单构造器时可省略 @Autowired
    public OrderService(UserService userService, OrderRepository orderRepo) {
        this.userService = userService;
        this.orderRepo = orderRepo;
    }

    public void createOrder(Long userId) {
        User user = userService.findById(userId);
        // userService 永远不可能是 null —— 编译期保证
    }
}
```

**核心优势 1：创建即完整。** new OrderService(userService, orderRepo) 出来那一刻，所有依赖就绑定了，不存在"半成品对象"。

**核心优势 2：编译期保护。** 如果有人写了 new OrderService() 但没传参数，编译直接报错。字段注入和 Setter 注入都做不到这一点 —— 它们编译通过，运行时才 NPE。

**核心优势 3：测试最简单。** 不需要 Spring 容器，不需要反射，不需要 setter：

```java
// 直接 new，传 mock，完事
UserService mockUser = Mockito.mock(UserService.class);
OrderRepository mockRepo = Mockito.mock(OrderRepository.class);
OrderService orderService = new OrderService(mockUser, mockRepo);
```

**核心优势 4：final 保护。** 字段是 final 的，一旦构造完成就不能被修改。任何人拿到 orderService 实例，都没法换掉它的依赖。

**代价：循环依赖编译失败。** 如果 A 依赖 B、B 又依赖 A，构造器注入会在编译期直接报错。这其实是好事 —— 循环依赖是设计问题，早发现比运行时 StackOverflow 强。


## 📄 三种方式对比决策

| 维度 | 字段注入 | Setter 注入 | 构造器注入 |
|---|---|---|---|
| 强制依赖 | ❌ 无法表达 | ❌ 无法表达 | ✅ 构造器参数即契约 |
| 不可变性 | ❌ 可被修改 | ❌ 可被修改 | ✅ final 字段 |
| 单元测试 | 需反射 / Spring 容器 | 可手动 set | 直接 new |
| 循环依赖 | 容器自动处理 | 容器自动处理 | ❌ 编译期报错 |
| 空安全 | 运行时 NPE | 运行时 NPE | 编译期保证 |
| 可选依赖 | ❌ 不支持 | ✅ 不调 setter 就行 | ⚠️ 需 @Nullable 标注 |
| 代码量 | 最少 | 中等 | 最多 |


## 📄 注解选型：@Autowired / @Resource / @Inject

注入方式（字段/Setter/构造器）解决"依赖放在哪"，注入注解解决"容器怎么识别要注入"。核心区别在于**匹配策略**。

### @Autowired —— 按类型匹配

Spring 看到 @Autowired，就去容器里找**类型匹配**的 Bean。

```java
// 容器里有且只有一个 UserService 类型的 Bean → 注入成功
@Autowired
private UserService userService;
```

**什么时候出问题？同一个接口有多个实现时。**

```java
public interface PayService {
    void pay(BigDecimal amount);
}

@Service
public class AlipayService implements PayService { ... }

@Service
public class WechatPayService implements PayService { ... }
```

```java
@Autowired
private PayService payService;
// 报错！找到 2 个 Bean：AlipayService 和 WechatPayService，不知道注入哪个
```

解决办法 —— 加 @Qualifier 指定名称：

```java
@Autowired
@Qualifier("alipayService")   // 指定 Bean 名称
private PayService payService;
```

### @Resource —— 按名称匹配（默认）

@Resource 默认先按**字段名**去容器里找同名的 Bean，找不到再按类型。

```java
// 字段名 alipayService 和容器里 Bean 名称一致 → 直接注入
@Resource
private PayService alipayService;

// 字段名 payService 找不到同名 Bean → 退回按类型找 → 找到 2 个 → 报错
@Resource
private PayService payService;

// 用 name 属性显式指定 Bean 名称
@Resource(name = "wechatPayService")
private PayService payService;
```

**直观理解：** @Autowired 是"我不管名字，只要类型对就行"，@Resource 是"我要找名字叫 xxx 的那个"。

### @Inject —— 和 @Autowired 几乎一样

来自 JSR-330 标准，按类型匹配，需要配合 @Named 指定名称（等价于 @Qualifier）。

```java
@Inject                      // 按类型找
@Named("alipayService")     // 指定名称（等价于 @Qualifier）
private PayService payService;
```

纯 Spring 项目几乎没有用 @Inject 的理由，它存在的意义是框架无关 —— 如果项目可能从 Spring 迁移到 Guice 或 CDI，用 @Inject 可以减少改动。

### 注解对比表

| 维度 | @Autowired | @Resource | @Inject |
|---|---|---|---|
| 来源 | Spring 自带 | JSR-250 (Java 标准) | JSR-330 (Java 标准) |
| 默认匹配策略 | 按**类型** | 按**名称**（字段名） | 按**类型** |
| 多实现时 | 报错，需 @Qualifier 辅助 | 字段名对上就直接注入 | 报错，需 @Named 辅助 |
| 指定名称 | @Autowired + @Qualifier("name") | @Resource(name="name") | @Inject + @Named("name") |
| 支持位置 | 字段、Setter、构造器 | 字段、Setter | 字段、Setter、构造器 |
| required=false | ✅ 可选注入 | ❌ 不支持 | ❌ 不支持 |
| 额外依赖 | 无（Spring 项目自带） | 无（JDK 自带 javax.annotation） | 需引入 javax.inject |

### 速查表

| 场景 | 推荐写法 |
|---|---|
| 只有一个实现，直接注入 | @Autowired private PayService payService |
| 多个实现，指定一个 | @Autowired @Qualifier("alipayService") 或 @Resource(name="alipayService") |
| 按字段名匹配更直观 | @Resource private PayService alipayService |
| 需要框架无关 | @Inject @Named("alipayService") |


## 📄 为什么规范和 AI 偏爱构造器注入

训练语料偏斜：Spring 官方文档、Spring Boot 源码、主流开源项目（Spring Security、Spring Cloud）全面采用构造器注入。AI 生成代码时自然继承这一偏好。

工程理由：构造器注入是唯一能在**编译期**暴露问题的方式 —— 忘记注入？编译失败。字段注入则在运行时才 NPE。


## 📄 字段注入为什么还在流传

历史包袱：早期 Spring 教程和脚手架全是 @Autowired 字段注入，大量存量代码延续这一写法。

体感差异：写 3 个依赖时，字段注入 3 行 vs 构造器注入 3 行参数 + 3 行赋值。但依赖超过 5 个时，构造器注入反而更清晰 —— 而且 5+ 个依赖本身就是设计问题的信号。


## 📄 Spring 官方态度演变

- Spring 1.x ~ 2.x：XML 配置为主，setter 注入是主流
- Spring 3.x：@Autowired 字段注入流行，"少写代码"成为卖点
- Spring 4.3：单构造器自动注入，无需 @Autowired，明确推荐构造器注入
- Spring Boot 2.x+：官方示例和文档全面转向构造器注入
- Spring 6 / Spring Boot 3：构造器注入已是事实标准

**Q1：Bean 注入容器时，名称和类型是在哪里确认的？**

分两个阶段：**注册时确定**，**注入时匹配**。

注册阶段（容器启动扫描）：Spring 扫描 @Component/@Service 等注解，为每个 Bean 生成 BeanDefinition，记录名称和类型。

| 信息 | 来源 | 默认值 |
|---|---|---|
| 名称 | @Component("xxx") 显式指定 | 类名首字母小写，如 `UserServiceImpl` → `userServiceImpl` |
| 类型 | 类本身 | 注解标注的类 |
| 接口类型 | 类实现了哪些接口 | 自动继承 |

注入阶段（@Autowired / @Resource 匹配）：@Autowired 拿 PayService 类型遍历容器找匹配；@Resource 拿字段名 alipayService 按名称查容器。

---

**Q2：@Autowired 和 @Resource 同时标注在一个字段上会怎样？**

都能工作，但逻辑不同。@Resource 先按名称找，@Autowired 按类型找。同时存在时 Spring 会优先用 @Resource 的匹配结果（因为 JSR-250 注解优先级更高）。实际开发中不应该同时标注两个，选一个统一风格即可。
