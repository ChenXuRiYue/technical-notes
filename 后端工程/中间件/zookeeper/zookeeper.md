# 📌 Zookeeper

**一句话定义**：分布式协调服务，不存业务数据，解决分布式系统里的"协调问题"——谁是 leader、配置怎么同步、资源怎么互斥访问。

类比：分布式系统的"大管家"，不干具体活，但所有组件都得找它注册、汇报、领任务。

### 发展历史

- **2006 年**：Yahoo! 内部开发，解决 Hadoop 生态协调问题
- **2010 年**：Apache 顶级项目
- **2011 年**：Kafka 诞生，选 ZK 做元数据管理
- **2014 年左右**：大数据黄金年代，Hadoop/HBase/Kafka/Solr 都用 ZK
- **2019 年**：Kafka 宣布去 ZK 化（KIP-500）
- **2023 年**：Kafka 3.4+ 支持 KRaft 模式，不再依赖 ZK

分两段：前半段"万物皆可 ZK"，后半段被批评太重，逐渐被 etcd/Consul/Nacos 替代。

### 核心能力（为什么能流行）

1. **强一致性**：写入后读取保证最新（CP 系统），适合做"权威数据源"
2. **Watch 机制**：监听节点变化，实现实时通知
3. **临时节点 + 会话**：客户端断开，临时节点自动删除——天然适合服务注册、leader 选举

### 应用生态

| 场景 | 代表系统 | 说明 |
|------|---------|------|
| 元数据管理 | Kafka、HBase | 存 broker 列表、region 信息 |
| 服务注册发现 | Dubbo | 注册 provider 地址 |
| 分布式锁 | Curator 框架 | 临时顺序节点实现公平锁 |
| 配置中心 | 滴滴、美团内部 | 节点存配置，watch 通知 |
| Leader 选举 | Hadoop NameNode | 主备切换 |
| 分布式队列 | Kafka 消费者组 | 消费进度协调 |

现在：Kafka 正在摆脱 ZK（KRaft），Dubbo 支持 Nacos/Consul 替代，新项目基本不选 ZK，首选 etcd（K8s 生态）或 Nacos（阿里生态）。

## 📄 面向开发者的核心能力

### 数据模型：树形节点（ZNode）

ZK 的数据结构像文件系统，但不是文件系统：

```
/
├── /config        ← 持久节点，存配置
├── /services
│   ├── /order     ← 持久节点
│   │   ├── /provider-0000000001  ← 临时顺序节点
│   │   └── /provider-0000000002  ← 临时顺序节点
│   └── /user
└── /locks
    └── /order-lock-0000000001   ← 临时顺序节点，分布式锁
```

每个节点叫 **ZNode**，有两类属性：
- **数据**：节点可以存一小段数据（最大 1MB），不是让你存业务数据，是存元数据、配置、状态
- **子节点**：节点可以有子节点，形成树

#### 四种节点类型

| 类型 | 说明 | 典型场景 |
|------|------|---------|
| **持久节点** | 创建后一直存在，除非主动删除 | 配置、服务分组 |
| **持久顺序节点** | 持久 + 自动编号（如 0000000001） | 分布式队列 |
| **临时节点** | 绑定 session，session 断开自动删除 | 服务实例注册 |
| **临时顺序节点** | 临时 + 自动编号 | 分布式锁（公平锁） |

**关键点**：临时节点不能有子节点。

### Watcher 机制：一次性通知

客户端可以对某个节点"打标签"（注册 watcher），当这个节点发生变化时，服务端会通知你。

但有个坑：**watcher 是一次性的**。触发一次后就没了，想继续监听得重新注册。

```
客户端：getData("/config", watcher=true)  →  注册监听
服务端：/config 数据变了  →  触发通知
客户端：收到事件  →  重新注册 watcher
```

#### 事件类型

| 事件 | 触发条件 |
|------|---------|
| NodeCreated | 节点被创建 |
| NodeDeleted | 节点被删除 |
| NodeDataChanged | 节点数据变更 |
| NodeChildrenChanged | 子节点列表变更 |

**典型场景**：配置中心——监听 `/config`，配置变了立即拉取最新。

### Session 会话

#### "连接 ZK"意味着什么？

ZK 是集群架构（通常 3 或 5 个节点），客户端**连接的是集群中的某一个节点**，不是连接整个集群。

```
客户端
  ↓
  ┌─────────────────────────────┐
  │  ZK 集群                    │
  │  ┌─────┐ ┌─────┐ ┌─────┐  │
  │  │ ZK1 │ │ ZK2 │ │ ZK3 │  │
  │  └─────┘ └─────┘ └─────┘  │
  │       Leader   Follower    │
  └─────────────────────────────┘
```

连接建立时发生的事：
1. **TCP 连接**：底层是 TCP 长连接
2. **创建 Session**：服务端分配一个 `sessionId`，客户端和服务端都维护这个 session 状态
3. **协商超时时间**：客户端可以指定想要的超时时间（比如 30 秒），服务端会返回实际分配的超时时间
4. **开始心跳**：客户端定期发送 ping 保持 session 存活

写请求会自动转发给 Leader，读请求可以在任意节点处理。

#### Session 的核心作用

Session 的核心价值是**绑定临时节点的生命周期**：

- 客户端定期发送心跳保持 session
- 如果超时没心跳，session 过期，**临时节点自动删除**
- session 过期后，客户端得重新连接、重新注册 watcher

**这就是为什么临时节点能做服务注册**——服务挂了，session 过期，节点自动消失，其他服务就知道它下线了。

#### 连接断了会怎样？

| 情况 | 结果 |
|------|------|
| **短暂断开**（在超时时间内重连） | Session 保持，临时节点还在 |
| **长时间断开**（超过超时时间） | Session 过期，临时节点删除，watcher 失效 |

客户端重连时，会带着 `sessionId` 去连接任意节点，如果 session 还没过期，可以恢复。

### 基本 API 操作

ZK 的 API 很简单，就这几个核心操作：

| 操作 | 说明 | 等价于 |
|------|------|--------|
| `create(path, data)` | 创建节点 | mkdir + echo > file |
| `delete(path)` | 删除节点 | rm |
| `exists(path)` | 判断节点是否存在 | test -e |
| `getData(path)` | 读取节点数据 | cat |
| `setData(path, data)` | 写入节点数据 | echo > file |
| `getChildren(path)` | 获取子节点列表 | ls |

#### 版本控制（CAS 乐观锁）

每个节点有个 `version` 号，每次 `setData` 会校验 version：

```java
// 先读
Stat stat = new Stat();
byte[] data = zk.getData("/config", false, stat);  // stat.version = 3

// 再写（带 version）
zk.setData("/config", newData, stat.version);  // 只有 version=3 时才成功
```

这叫 **CAS（Compare-And-Set）**，防止并发写覆盖。

### Curator：高级封装库

原生 ZK API 太底层，Apache Curator 帮你封装了常用模式：

```java
// 原生 API：你得自己处理重连、异常、watcher
zk.create("/lock", data, EPHEMERAL_SEQUENTIAL, ...);

// Curator：直接用
InterProcessMutex lock = new InterProcessMutex(client, "/lock");
lock.acquire();
try {
    // 业务逻辑
} finally {
    lock.release();
}
```

Curator 提供的能力：
- **分布式锁**：`InterProcessMutex`、`InterProcessSemaphoreMutex`
- **Leader 选举**：`LeaderSelector`
- **分布式缓存**：`PathChildrenCache`、`TreeCache`（自动 watch + 缓存）
- **重试策略**：连接断了自动重连
- **各种 Recipe**：分布式队列、屏障、计数器等



## 📄 一致性协议 — ZAB

ZAB（Zookeeper Atomic Broadcast）是 ZK 自己的一致性协议，核心目标：**所有节点的数据保持一致**。

### 两个阶段

#### 崩溃恢复（选举阶段）

当集群启动或 Leader 挂了，需要选出新 Leader。

**选举规则**：ZXID 最大的节点优先（ZXID = 事务 ID，越大说明数据越新）。如果 ZXID 相同，则 serverId 大的优先（serverId 是配置时指定的）。

ZXID 是 64 位：高 32 位是 epoch（Leader 任期编号），低 32 位是事务计数器。每次选出新 Leader，epoch 加 1，计数器归零，保证全局唯一且递增。

#### 消息广播（正常工作阶段）

Leader 把写操作广播给所有 Follower：

```
Client → 写请求 → Leader
Leader → 提议(Proposal) → 所有 Follower
Follower → ACK(确认) → Leader
Leader → 收到多数派 ACK → 提交(Commit)
Leader → 提交结果 → 所有 Follower
```

**关键点**：多数派（quorum）确认才算成功。3 节点集群需要 2 个确认，5 节点需要 3 个。

### 为什么叫"原子广播"？

- **原子**：要么所有节点都执行，要么都不执行（多数派机制保证）
- **广播**：Leader 把操作广播给所有节点

### 和 Paxos/Raft 的区别

| 协议 | 特点 | 使用者 |
|------|------|--------|
| ZAB | 专为 ZK 设计，强调 Leader 选举 + 消息广播 | Zookeeper |
| Paxos | 理论基础，难以实现 | Chubby |
| Raft | 更易理解，分 term 选举 | etcd, Consul |

ZAB 和 Raft 很像，都是 Leader + 多数派确认，但细节不同（ZXID vs Term + Index）。


## 📄 集群架构与选举机制

### 🔖 角色：Leader / Follower / Observer

#### Leader（领导者）

**唯一写入点**。所有写请求都必须经过 Leader。

- 接收写请求，生成事务 ID（ZXID）
- 把写操作广播给所有 Follower
- 收到多数派（quorum）确认后，才算写入成功
- 也处理读请求（但读不需要经过 Leader）

**关键点**：Leader 是选举产生的，集群中只能有一个 Leader。

#### Follower（跟随者）

**参与投票 + 处理读请求**。

- 接收 Leader 的写广播，写入本地数据
- 对写操作进行投票（同意/拒绝）
- 直接处理读请求（不需要转发给 Leader）
- 参与 Leader 选举投票

**典型配置**：3 节点集群 = 1 Leader + 2 Follower

#### Observer（观察者）

**只处理读，不参与投票**。

- 接收 Leader 的写广播，写入本地数据
- 直接处理读请求
- **不参与 Leader 选举投票**
- **不对写操作投票**

**用途**：扩展读能力，不影响写性能。比如 100 个客户端读，但只有 3 个节点，读压力大。加 Observer 可以水平扩展读能力，而不需要增加投票节点（投票节点越多，写延迟越高）。

#### 三种角色对比

| 角色 | 读 | 写 | 投票 | 用途 |
|------|----|----|------|------|
| Leader | ✓ | ✓ | ✓ | 唯一写入点，保证一致性 |
| Follower | ✓ | 转发 | ✓ | 参与一致性保证 |
| Observer | ✓ | 转发 | ✗ | 扩展读能力 |

**核心思想**：写必须经过 Leader 保证顺序，读可以分散到所有节点。

## 📄 典型应用场景

### 🔖 分布式锁

#### 简单锁（临时节点）

```
客户端 A：create("/lock", EPHEMERAL)  →  成功，获得锁
客户端 B：create("/lock", EPHEMERAL)  →  失败，节点已存在
客户端 B：exists("/lock", watcher=true)  →  等待
客户端 A：释放锁（删除节点）
客户端 B：收到通知，重新尝试创建
```

**问题**：惊群效应——所有等待者同时收到通知，同时竞争。

#### 公平锁（临时顺序节点）

```
客户端 A：create("/lock/lock-", EPHEMERAL_SEQUENTIAL)  →  /lock/lock-0000000001
客户端 B：create("/lock/lock-", EPHEMERAL_SEQUENTIAL)  →  /lock/lock-0000000002
客户端 C：create("/lock/lock-", EPHEMERAL_SEQUENTIAL)  →  /lock/lock-0000000003

客户端 B：getChildren("/lock")  →  发现自己不是最小的
客户端 B：exists("/lock/lock-0000000001", watcher=true)  →  监听前一个节点

客户端 A：释放锁（删除 0000000001）
客户端 B：收到通知，检查自己是不是最小的  →  是，获得锁
```

**关键**：每个客户端只监听前一个节点，避免惊群。

### 🔖 配置管理

```
/config/db-url = "jdbc:mysql://..."  ← 持久节点，存配置

客户端：getData("/config/db-url", watcher=true)  →  获取配置 + 监听
配置变更：setData("/config/db-url", newUrl)
客户端：收到通知 → 重新 getData → 更新本地配置
```

**优势**：配置变更实时推送，不需要轮询。

### 🔖 命名服务

```
/services/order/provider-0000000001
/services/order/provider-0000000002
/services/order/provider-0000000003

客户端：getChildren("/services/order")  →  获取所有 provider 列表
```

**典型场景**：Dubbo 的服务注册发现。


## 📄 横向对比：ZK vs Nacos vs Consul vs etcd

### 定位对比

| 组件 | 原始定位 | 核心能力 |
|------|---------|---------|
| **Zookeeper** | 分布式协调服务 | 强一致性 + Watch + 临时节点 |
| **Nacos** | 服务注册 + 配置中心 | 服务发现 + 配置管理 + 动态 DNS |
| **Consul** | 服务网格 + 注册中心 | 服务发现 + 健康检查 + KV + Service Mesh |
| **etcd** | 分布式 KV 存储 | 强一致性 + 简单 KV + Watch |

### 关键差异

#### 一致性模型

| 组件 | 一致性 | 协议 | 说明 |
|------|-------|------|------|
| ZK | CP（强一致） | ZAB | 写入后读取保证最新，但节点挂多了不可用 |
| Nacos | AP/CP 可切换 | Raft + Distro | 临时实例用 AP（最终一致），持久实例用 CP |
| Consul | CP（强一致） | Raft | 和 ZK 类似，强一致 |
| etcd | CP（强一致） | Raft | 和 ZK 类似，强一致 |

**Nacos 的亮点**：支持 AP 模式，临时实例心跳检测，节点挂了不影响其他节点可用性。这对服务注册场景更友好。

#### 服务健康检查

| 组件 | 方式 | 说明 |
|------|------|------|
| ZK | Session 心跳 | 临时节点绑定 session，session 过期自动删除 |
| Nacos | 客户端心跳 + 服务端探测 | 临时实例客户端发心跳，持久实例服务端主动探测 |
| Consul | 多种检查 | TCP/HTTP/gRPC/Script/Sidecar，最丰富 |
| etcd | Lease 心跳 | 类似 ZK 的 session |

**Consul 的亮点**：健康检查最丰富，支持 HTTP 端点探测、脚本执行等。

#### 配置管理

| 组件 | 配置能力 | 说明 |
|------|---------|------|
| ZK | 需自己实现 | 节点存配置 + watcher，但没有管理界面 |
| Nacos | 原生支持 | 专门的配置中心，有管理界面、灰度发布、回滚 |
| Consul | KV 存储 | 可以存配置，但功能简单 |
| etcd | KV 存储 | 可以存配置，但功能简单 |

**Nacos 的亮点**：配置管理是核心功能，有专门的管理界面、版本管理、灰度发布。

#### 服务网格支持

| 组件 | Service Mesh | 说明 |
|------|-------------|------|
| ZK | ✗ | 不支持 |
| Nacos | 有限 | 可以对接 Istio |
| Consul | ✓ | 原生支持 Service Mesh，有 Sidecar 代理 |
| etcd | ✗ | 不支持 |

**Consul 的亮点**：原生支持 Service Mesh，可以做流量管理、安全通信。

#### 易用性

| 组件 | 学习曲线 | 运维复杂度 | 管理界面 |
|------|---------|-----------|---------|
| ZK | 高 | 高 | ✗（需要第三方） |
| Nacos | 低 | 低 | ✓（自带） |
| Consul | 中 | 中 | ✓（自带） |
| etcd | 中 | 中 | ✗（需要第三方） |

### 选型建议

| 场景 | 推荐 | 理由 |
|------|------|------|
| **新项目服务注册** | Nacos | 功能全、易用、阿里生态 |
| **K8s 生态** | etcd | K8s 原生使用，强一致 |
| **需要 Service Mesh** | Consul | 原生支持，功能最全 |
| **已有 ZK 生态** | ZK | 不迁移成本 |
| **强一致性 + 简单 KV** | etcd | 最轻量、最简单 |
| **配置中心为主** | Nacos | 配置管理功能最全 |

### 为什么 ZK 在衰落？

1. **运维重**：集群部署复杂，没有管理界面
2. **性能瓶颈**：写入必须经过 Leader，高并发下成为瓶颈
3. **功能单一**：只有协调能力，配置管理需要自己实现
4. **Watch 一次性**：每次触发都得重新注册，开发体验差
5. **Java 生态重**：客户端依赖重，不适合云原生轻量场景

### 一句话总结

- **ZK**：老管家，强一致但重
- **Nacos**：全能选手，服务注册 + 配置中心
- **Consul**：Service Mesh 专家
- **etcd**：K8s 的好搭档，轻量 KV

### ZAB vs Raft 对比

| 维度 | ZAB | Raft |
|------|-----|------|
| 选举 | ZXID 最大的优先 | Term + 投票 |
| 日志复制 | Proposal + ACK + Commit | AppendEntries + ACK |
| 任期 | Epoch | Term |
| 日志编号 | ZXID（全局递增） | Index + Term |

两者本质都是 Leader + 多数派，只是术语和细节不同。
