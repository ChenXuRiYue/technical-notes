# 📌 认证与加密

## 📄 核心概念速查

| 概念 | 本质 | 一句话 |
|------|------|--------|
| Cookie | 浏览器存储机制 | 服务端告诉浏览器"存这个"，浏览器每次请求自动带上 |
| Session | 服务端状态 | 用户的数据存在服务端，Cookie 里只存一个 Session ID |
| Token | 认证凭证 | 服务端签发的"通行证"，客户端自己保管，服务端不存状态 |
| JWT | Token 的一种实现 | 把用户信息编码进 Token 本身，自带签名防篡改 |
| 对称加密（AES） | 一把钥匙 | 加密和解密用同一把密钥 |
| 非对称加密（RSA） | 两把钥匙 | 公钥加密，私钥解密；或私钥签名，公钥验证 |
| HTTPS = 非对称 + 对称 | 混合加密 | 先用 RSA 交换密钥，再用 AES 传输数据 |

## 📄 Cookie vs Token

| 维度 | Cookie + Session | Token（如 JWT） |
|------|------------------|-----------------|
| 状态 | 有状态（服务端存 Session） | 无状态（服务端不存） |
| 存储位置 | 浏览器自动管理 | 客户端手动存（localStorage / Cookie） |
| 跨域 | 受同源策略限制 | 天然支持跨域 |
| 扩展性 | 多机部署需同步 Session | 随便扩展，Token 自带信息 |
| 安全性 | CSRF 风险（浏览器自动带 Cookie） | XSS 风险（Token 存在前端） |
| 适用场景 | 传统 Web 应用 | 前后端分离 / 移动端 / 微服务 |

## 📄 对称加密 vs 非对称加密

| 维度 | 对称加密（AES） | 非对称加密（RSA） |
|------|-----------------|-------------------|
| 密钥 | 一把（加解密相同） | 两把（公钥 + 私钥） |
| 速度 | 快 | 慢（比 AES 慢 100-1000 倍） |
| 安全性 | 密钥传输是瓶颈 | 公钥可公开分发 |
| 典型用途 | 数据加密（大量数据） | 密钥交换、数字签名 |
| 常见算法 | AES-128/256 | RSA-2048、ECC |

## 📄 Token 业务设计

### Token 的层次

| 层次 | 说明 | 例子 |
|------|------|------|
| Access Token | 短期凭证，用于访问资源 | JWT，有效期 15-30 分钟 |
| Refresh Token | 长期凭证，用于刷新 Access Token | 有效期 7-30 天 |
| API Key | 应用级凭证，标识调用方 | 第三方接口鉴权 |

### Token 存储方案

| 方案 | 安全性 | 适用场景 |
|------|--------|----------|
| HttpOnly Cookie | 最高（防 XSS） | 传统 Web |
| localStorage | 中（XSS 可读取） | 前后端分离 |
| 内存变量 | 高（刷新丢失） | 高安全要求 |

## 📄 Cookie 协议的便利

浏览器自动完成三件事，零前端代码：

| 便利 | 说明 |
|------|------|
| 自动存储 | 服务端 `Set-Cookie`，浏览器自动存 |
| 自动发送 | 每次请求同源地址，浏览器自动带上 |
| 自动清理 | 过期了浏览器自动删 |

**限制：** 每个域名最多 50 个 Cookie，单条最大 4KB。

## 📄 项目实践指南

### 选型：该用哪种方案？

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 同域单体应用 | Cookie + Session | 浏览器全包，前端零代码 |
| 前后端分离 / 跨域 | Token（JWT） | Cookie 跨域麻烦，Token 天然支持 |
| 移动端 | Token | 移动端没有浏览器自动机制 |
| 微服务 | Token | 无状态，随便扩展 |

### 后端职责

**Cookie + Session 方案：**

1. 登录成功时签发 Cookie：`Set-Cookie: session_id=xxx; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=86400`
2. 每个请求从 Cookie 读 session_id，查 Session 获取用户信息
3. 必须设置的安全属性：

| 属性 | 必须？ | 作用 |
|------|--------|------|
| HttpOnly | 必须 | JS 读不到，防 XSS |
| Secure | 必须 | 只在 HTTPS 下传输 |
| SameSite=Lax | 必须 | 防 CSRF |
| Path | 建议 | 控制 Cookie 适用路径 |
| Domain | 按需 | 子域共享时才设 |
| Max-Age/Expires | 建议 | 不设就是会话级，关浏览器就丢 |

**Token 方案：**

1. 登录成功时签发 Access Token + Refresh Token
2. Access Token 有效期短（15-30 min），Refresh Token 有效期长（7-30 天）
3. Access Token 过期时，用 Refresh Token 换新的，用户无感
4. 服务端不存 Token 状态，只验签

**Session 方案：**

1. 服务端存 Session 数据（内存 / Redis / 数据库）
2. Cookie 里只存 Session ID，不存用户信息
3. 多机部署时 Session 必须共享（Redis 等），否则用户登录状态丢失

### 前端职责

| 方案 | 前端要做的 |
|------|-----------|
| Cookie + Session | 几乎不用管，浏览器自动带。跨域时 `fetch` 加 `credentials: 'include'` |
| Token | 手动存（localStorage / Cookie），每次请求手动加 `Authorization: Bearer xxx` |

### 通用规范

| 规范 | 原因 |
|------|------|
| 认证信息用 HttpOnly Cookie，不用 localStorage | XSS 拿不到 |
| 前后端分离、跨域场景用 Token | Cookie 跨域麻烦 |
| 敏感操作（改密码、支付）要求重新验证 | 防会话劫持 |
| 不要在 Cookie / Token 里存用户信息，只存 ID | Cookie 会被日志记录，JWT 的 payload 只是 Base64 |
| Session 必须设过期时间 | 防止无限占用服务端资源 |
| Refresh Token 必须支持主动失效 | 用户改密码 / 退出时要能吊销 |

## 🌳 生长思考

- **Cookie 是机制，Token 是凭证，加密是手段** — 三者解决不同层面的问题，不要混为一谈
- **没有绝对安全，只有成本和收益的平衡** — AES 够快但密钥难分发，RSA 安全但慢，所以 HTTPS 混合用
- **Session 和 Token 之争本质是有状态 vs 无状态** — 微服务时代 Token 胜出，但传统 Web Session 依然好用

## 💭 反复绊脚

- Token 和 Cookie 不是对立关系，Token 可以存在 Cookie 里
- JWT 的 payload 是 Base64 编码，不是加密，任何人都能解码看内容
- HTTPS 的 S 是 Security，不是 Safe；它加密的是传输过程，不是数据本身
